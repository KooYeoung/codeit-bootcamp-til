## 1. 읽기 좋은 코드란

읽기 좋은 코드는 단순히 짧거나 문법적으로 깔끔한 코드만을 의미하지 않는다.

읽기 좋은 코드는 다른 사람이 코드를 봤을 때 빠르게 이해할 수 있고, 변경이 필요할 때 안정적으로 수정할 수 있는 코드이다.

즉, 코드를 잘 작성한다는 것은 결국 읽는 사람의 시간을 줄이고 유지보수를 쉽게 만드는 것이라고 이해했다.

정리하면 다음과 같다.

- 코드가 잘 읽힌다.
- 이해가 잘 된다.
- 유지보수하기 쉽다.
- 팀의 시간과 자원을 줄일 수 있다.

코드는 작성하는 순간부터 언젠가는 다시 읽혀야 하는 대상이 되기 때문에, 처음 작성할 때부터 읽는 사람을 고려해야 한다.

## 2. 추상화

추상화는 중요한 정보만 남기고 덜 중요한 세부사항은 숨기는 과정이다.

모든 정보를 그대로 드러내면 오히려 이해하기 어려워진다.

예를 들어 “친구랑 노래방 갔다”라는 표현은 많은 세부 정보를 생략하지만, 상황을 이해하는 데 필요한 핵심 정보는 충분히 전달한다.

코드에서도 마찬가지로 너무 구체적인 구현이 한 곳에 모두 드러나 있으면 읽는 사람이 많은 정보를 기억해야 한다.

그래서 적절한 이름의 메서드나 객체로 의미를 묶어 추상화하는 것이 중요하다.

예를 들어 다음과 같은 코드는 구체적인 조건과 처리가 한눈에 들어오지 않을 수 있다.

    if (user == null || channel == null) {
        return;
    }

이런 경우 의도를 드러내는 메서드로 추출할 수 있다.

    if (isUserOrChannelMissing(user, channel)) {
        return;
    }

이렇게 하면 조건의 세부 구현보다 “사용자나 채널 정보가 없는 경우”라는 의미가 먼저 보인다.

---

## 3. 추상화 레벨 맞추기

추상화 레벨이 맞는다는 것은 하나의 코드 블록 안에서 비슷한 수준의 표현이 사용된다는 뜻이다.

높은 수준의 도메인 로직과 낮은 수준의 세부 구현이 한 메서드 안에 섞이면 읽기 어려워진다.

예를 들어 한 메서드 안에서 다음과 같은 내용이 모두 섞여 있다면 이해하기 어렵다.

- 입력 검증
- 객체 조회
- 비즈니스 로직
- 저장소 저장 방식
- 출력 메시지 작성

이럴 때는 메서드를 분리해서 각각의 역할을 드러내는 것이 좋다.

    validateRequest();
    User user = findUser();
    Channel channel = findChannel();
    createMessage(user, channel);

이런 흐름은 내부 구현을 전부 보지 않아도 전체 로직의 의도를 파악할 수 있다.

## 4. 논리적 사고 흐름과 Early return

코드를 읽는 사람은 한 번에 많은 조건을 기억하기 어렵다.

중첩된 `if`, `else if`, `else`가 많아지면 현재 어떤 조건 안에 있는지 계속 기억해야 한다.

이때 Early return을 사용하면 흐름을 단순하게 만들 수 있다.

예를 들어 다음과 같은 구조는 중첩이 깊어질 수 있다.

    if (user != null) {
        if (channel != null) {
            createMessage(user, channel);
        }
    }

Early return을 사용하면 예외 상황을 먼저 처리하고, 정상 흐름을 아래에 둘 수 있다.

    if (user == null) {
        return;
    }

    if (channel == null) {
        return;
    }

    createMessage(user, channel);

이 방식은 정상 흐름을 더 쉽게 읽을 수 있게 해준다.

정리하면 Early return의 장점은 다음과 같다.

- 중첩을 줄일 수 있다.
- 예외 상황을 먼저 제거할 수 있다.
- 정상 흐름이 더 잘 보인다.
- 읽는 사람이 기억해야 할 조건이 줄어든다.

## 5. 객체지향 패러다임

객체지향에서 객체는 데이터와 기능을 함께 가진 책임 단위이다.

객체는 단순히 필드만 모아둔 구조가 아니라, 자신의 데이터를 바탕으로 필요한 행동을 수행해야 한다.

객체는 외부 세계와 공개 메서드를 통해 소통하고, 내부 데이터와 세부 구현은 숨기는 것이 좋다.

객체를 설계할 때 중요한 질문은 다음과 같다.

- 이 객체의 책임은 무엇인가?
- 이 객체가 외부에 제공해야 하는 기능은 무엇인가?
- 이 객체 내부에서 처리해야 할 로직은 무엇인가?
- 다른 객체와 어떻게 협력해야 하는가?

예를 들어 `User` 객체가 자신의 정보를 수정할 책임을 가진다면, 외부에서 필드를 직접 변경하기보다 `update()` 같은 메서드를 통해 변경 의도를 표현할 수 있다.

    user.update(nickname, realName, password, email, phoneNumber);

이 방식은 단순 setter보다 “사용자 정보를 수정한다”는 의도를 더 잘 드러낸다.

## 6. getter와 setter 사용에 대한 생각

getter와 setter는 편리하지만 무분별하게 사용하면 객체가 스스로 책임을 가지지 못하고 단순 데이터 묶음처럼 사용될 수 있다.

특히 setter를 많이 열어두면 객체의 상태가 외부에서 쉽게 변경될 수 있다.

그래서 단순한 `setName()`보다 변경 의도가 드러나는 메서드명을 고민하는 것이 좋다.

예를 들어 다음 이름보다:

    setNickname()

상황에 따라 아래 이름이 더 의미 있을 수 있다.

    changeNickname()
    updateProfile()

중요한 것은 단순히 값을 바꾸는 것이 아니라, 어떤 의도로 상태가 변경되는지 코드에 드러내는 것이다.

## 7. 상속보다 조합

상속은 부모 클래스의 기능을 자식 클래스가 물려받는 구조이다.

하지만 상속은 부모와 자식의 결합도가 높아질 수 있다.

부모 클래스의 변경이 자식 클래스에 영향을 줄 수 있고, 계층 구조가 깊어질수록 수정이 어려워질 수 있다.

그래서 단순히 코드 중복을 제거하기 위해 상속을 사용하는 것은 주의해야 한다.

조합은 필요한 객체를 필드로 가지고 협력하는 방식이다.

상속보다 조합을 사용하면 더 유연하게 구조를 변경할 수 있다.

정리하면 다음과 같다.

- 상속은 강한 관계를 만든다.
- 부모와 자식의 결합도가 높아진다.
- 조합은 필요한 기능을 가진 객체와 협력하는 방식이다.
- 유연한 구조를 만들 때는 상속보다 조합을 먼저 고려할 수 있다.

## 8. Value Object

Value Object는 도메인의 어떤 값을 표현하는 객체이다.

값 객체는 식별자보다 내부 값이 중요하다.

예를 들어 돈을 표현하는 `Money` 객체가 있다면, 서로 다른 인스턴스라도 내부 금액이 같으면 같은 값으로 볼 수 있다.

Value Object에서 중요한 특징은 다음과 같다.

- 불변성
- 동등성
- 유효성 검증

불변성을 위해 필드를 `final`로 두고 setter를 제공하지 않을 수 있다.

동등성을 위해 `equals()`와 `hashCode()`를 재정의해야 한다.

유효성 검증은 생성 시점에 처리하는 것이 좋다.

    public class Money {
        private final long value;

        public Money(long value) {
            if (value < 0) {
                throw new IllegalArgumentException("돈은 0원 이상이어야 합니다.");
            }

            this.value = value;
        }
    }

이렇게 하면 잘못된 값으로 객체가 생성되는 것을 막을 수 있다.

## 9. 좋은 주석

주석은 무조건 많다고 좋은 것이 아니다.

코드가 스스로 의미를 드러내지 못해서 주석으로 설명해야 한다면, 먼저 코드 자체를 개선할 수 있는지 고민해야 한다.

좋은 주석은 코드로 표현하기 어려운 의사결정의 이유나 히스토리를 남길 때 사용한다.

예를 들어 다음과 같은 내용은 주석으로 남길 수 있다.

- 특정 정책을 선택한 이유
- 외부 시스템 제약 때문에 우회한 이유
- 비즈니스 의사결정의 배경
- 참고해야 할 문서 링크

반대로 코드만 읽어도 알 수 있는 내용을 반복하는 주석은 줄이는 것이 좋다.

예를 들어:

    // user를 저장한다.
    userService.save(user);

이런 주석은 코드와 같은 내용을 반복할 뿐이라 큰 도움이 되지 않는다.

더 좋은 방향은 메서드명과 변수명 자체가 의도를 드러내도록 작성하는 것이다.

## 10. 리팩토링 관점

리팩토링은 단순히 코드를 예쁘게 바꾸는 작업이 아니다.

기능은 유지하면서 구조를 개선하는 작업이다.

리팩토링할 때는 한 번에 크게 바꾸기보다 작은 단위로 진행하는 것이 좋다.

중요한 점은 각 리팩토링 단계마다 왜 그렇게 바꾸는지 설명할 수 있어야 한다는 것이다.

리팩토링할 때 확인할 수 있는 기준은 다음과 같다.

- 추상화 레벨이 맞는가?
- 메서드가 너무 많은 책임을 가지고 있지 않은가?
- 객체가 자신의 책임을 잘 가지고 있는가?
- 중복을 제거할 수 있는가?
- 객체 간 결합도가 너무 높지 않은가?
- 상속보다 조합이 더 적절하지 않은가?
- 도메인 개념을 객체로 표현할 수 있지 않은가?

sprint-mission 코드를 다시 볼 때도 이 기준으로 점검할 수 있다.

## 11. sprint-mission 적용 정리

이번 sprint-mission에서는 `misson2.md` 요구사항을 기준으로 채팅 서비스의 기본 도메인과 서비스 구조를 구현했다.

주요 목표는 다음과 같았다.

- Git과 GitHub를 통한 프로젝트 관리
- 채팅 서비스 도메인 모델 설계
- Java 클래스 구현
- 서비스 인터페이스 설계
- JCF 기반 구현체 작성
- CRUD 기능 구현
- Stream API를 통한 데이터 조회
- 싱글톤 패턴 구현
- 팩토리 패턴을 통한 의존성 관리 이해

구현 과정에서는 User, Channel, Message를 주요 도메인으로 나누고 각각의 CRUD 흐름을 확인했다.

서비스는 인터페이스와 구현체로 나누어 작성하고, JCF 구현체에서는 Map이나 List 같은 컬렉션을 활용해 데이터를 메모리에 저장했다.

예를 들어 User 조회 후 수정하는 흐름에서는 Map에서 꺼낸 객체가 같은 참조라면, 객체의 필드를 변경했을 때 Map 안의 객체에도 변경 내용이 반영된다.

    byId.update(nickname, realName, password, email, phoneNumber);

이런 참조 관계를 이해하는 것도 중요했다.

## 12. 오늘 정리

오늘 학습의 핵심은 “읽기 좋은 코드”를 코드 스타일 문제가 아니라 유지보수성과 사고 흐름의 문제로 바라보는 것이다.

좋은 코드는 동작만 하는 코드가 아니라, 나중에 다시 읽었을 때 의도를 빠르게 이해할 수 있는 코드이다.

앞으로 코드를 작성할 때는 다음 기준을 계속 확인해야겠다.

- 이름이 의도를 드러내는가?
- 메서드의 추상화 레벨이 맞는가?
- 조건문 중첩을 줄일 수 있는가?
- 객체가 자신의 책임을 가지고 있는가?
- setter/getter를 무분별하게 사용하고 있지 않은가?
- 주석이 코드의 부족함을 가리는 용도로 쓰이고 있지 않은가?
- 리팩토링 이유를 설명할 수 있는가?

sprint-mission 기능 구현은 완료했지만, 이후에는 단순 구현 완료에서 끝내지 않고 Readable Code 관점으로 다시 코드를 점검해볼 필요가 있다.

# SQL에서 DDL과 DML의 차이점

## 1. DDL과 DML의 차이

DDL과 DML은 SQL 명령어를 목적에 따라 나눈 분류입니다.

`DDL(Data Definition Language)`은 데이터베이스의 구조를 정의하거나 변경하는 명령어입니다.
즉, 테이블, 컬럼, 제약조건, 인덱스와 같은 데이터 저장 구조를 만들거나 수정하거나 삭제할 때 사용합니다.

반면 `DML(Data Manipulation Language)`은 이미 만들어진 구조 안에 저장된 실제 데이터를 조작하는 명령어입니다.
즉, 테이블에 데이터를 추가, 조회, 수정, 삭제할 때 사용합니다.

간단히 말하면, `DDL은 데이터베이스의 구조를 다루고, DML은 그 구조 안에 들어 있는 데이터를 다룹니다.`

---

## 2. DDL의 대표 명령어와 용도

DDL은 데이터베이스 객체를 생성, 수정, 삭제할 때 사용하는 명령어입니다.

### CREATE

새로운 데이터베이스 객체를 생성할 때 사용합니다.

예를 들어 `CREATE TABLE`은 새로운 테이블을 생성할 때 사용합니다.

```sql
CREATE TABLE employee (
    id BIGINT PRIMARY KEY,
    name VARCHAR(50),
    email VARCHAR(100)
);
```

### ALTER

기존 데이터베이스 객체의 구조를 변경할 때 사용합니다.

예를 들어 테이블에 컬럼을 추가하거나, 컬럼 타입을 변경하거나, 제약조건을 추가할 수 있습니다.

```sql
ALTER TABLE employee ADD COLUMN phone_number VARCHAR(20);
```

### DROP

기존 데이터베이스 객체를 삭제할 때 사용합니다.

예를 들어 `DROP TABLE`은 테이블 자체를 삭제합니다.

```sql
DROP TABLE employee;
```

### TRUNCATE

테이블의 구조는 유지하면서 테이블 안의 모든 데이터를 빠르게 삭제할 때 사용합니다.

```sql
TRUNCATE TABLE employee;
```

`DELETE`와 비슷해 보이지만, `TRUNCATE`는 테이블 전체 데이터를 한 번에 삭제하는 데 사용되며 일반적으로 DDL로 분류됩니다.

---

## 3. DML의 대표 명령어와 용도

DML은 테이블 안에 저장된 데이터를 조작할 때 사용하는 명령어입니다.

### SELECT

테이블에 저장된 데이터를 조회할 때 사용합니다.

```sql
SELECT * FROM employee;
```

일부 분류에서는 `SELECT`를 DQL(Data Query Language)로 따로 구분하기도 하지만, 넓은 의미에서는 데이터를 다루는 명령어로 함께 설명하기도 합니다.

### INSERT

테이블에 새로운 데이터를 추가할 때 사용합니다.

```sql
INSERT INTO employee (id, name, email)
VALUES (1, '홍길동', 'hong@example.com');
```

### UPDATE

기존 데이터를 수정할 때 사용합니다.

```sql
UPDATE employee
SET email = 'new@example.com'
WHERE id = 1;
```

### DELETE

기존 데이터를 삭제할 때 사용합니다.

```sql
DELETE FROM employee
WHERE id = 1;
```

---

## 4. 정리

DDL은 데이터베이스의 구조를 정의하거나 변경하는 명령어이고, DML은 그 구조 안에 저장된 실제 데이터를 다루는 명령어입니다.

예를 들어 테이블을 생성하거나 컬럼을 추가하는 것은 DDL이고, 그 테이블에 직원을 등록하거나 수정하거나 삭제하는 것은 DML입니다.

# 역정규화가 필요한 상황과 적용 시 고려사항

## 1. 역정규화란?

역정규화는 정규화를 통해 나누어진 테이블을 성능이나 조회 편의성을 위해 일부러 다시 합치거나, 중복 컬럼을 추가하는 설계 방식입니다.

정규화는 데이터 중복을 줄이고 데이터의 일관성을 높이는 것을 목표로 합니다.
예를 들어 상품 테이블과 주문 테이블을 분리하면 상품명, 가격, 재고 같은 상품 정보는 상품 테이블에서 관리하고, 주문 테이블은 어떤 상품을 몇 개 주문했는지만 저장할 수 있습니다.

이렇게 하면 상품 정보가 중복 저장되지 않기 때문에 데이터 관리가 깔끔해지고, 변경이 발생했을 때 한 곳만 수정하면 된다는 장점이 있습니다.

하지만 정규화를 많이 적용하면 조회할 때 여러 테이블을 조인해야 하는 경우가 많아집니다.
예를 들어 주문 목록 화면에서 주문 정보, 상품명, 주문 수량, 처리 수량, 주문 상태 등을 함께 보여줘야 한다면 여러 테이블을 조인하거나 서브쿼리를 사용해야 할 수 있습니다.

데이터가 많아질수록 이러한 조인 비용은 커질 수 있고, 조회 성능이 느려질 수 있습니다.
이런 경우 조회 성능을 개선하기 위해 역정규화를 고려할 수 있습니다.

---

## 2. 역정규화가 필요한 상황

역정규화는 다음과 같은 상황에서 고려할 수 있습니다.

### 1. 조회 성능이 중요한 화면에서 조인 비용이 큰 경우

사용자가 자주 보는 목록 화면이나 대시보드에서 여러 테이블을 매번 조인해야 한다면 응답 속도가 느려질 수 있습니다.

예를 들어 주문 목록 화면에서 주문 정보와 상품명, 주문 상태, 처리 수량을 매번 조인해서 가져와야 한다면 조회 비용이 커질 수 있습니다.
이 경우 주문 테이블이나 별도 조회용 테이블에 상품명, 처리 수량 같은 값을 일부 중복 저장하여 조회 성능을 개선할 수 있습니다.

### 2. 통계나 집계 데이터를 매번 계산하기 어려운 경우

부서별 직원 수, 상품별 주문 수량, 월별 매출 합계처럼 매번 계산하면 비용이 큰 데이터는 별도의 컬럼이나 집계 테이블로 관리할 수 있습니다.

예를 들어 게시글의 댓글 수를 매번 `COUNT` 쿼리로 계산하는 대신, 게시글 테이블에 `comment_count` 컬럼을 두고 댓글이 추가되거나 삭제될 때 값을 갱신할 수 있습니다.

### 3. 데이터가 자주 조회되지만 변경은 비교적 적은 경우

변경이 자주 발생하는 데이터는 중복 관리 비용이 큽니다.
반대로 자주 조회되지만 변경이 적은 데이터라면 중복 데이터 관리 부담이 상대적으로 작기 때문에 역정규화 효과를 얻기 쉽습니다.

예를 들어 주문 당시의 상품명이나 상품 가격은 이후 상품 정보가 변경되더라도 주문 시점의 값을 보존해야 하는 경우가 많습니다.
이런 경우 주문 상세 테이블에 주문 당시의 상품명과 가격을 저장하는 것은 데이터 이력 보존 측면에서도 의미가 있습니다.

---

## 3. 역정규화 적용 시 고려해야 할 사항

역정규화는 조회 성능을 개선할 수 있지만, 데이터 중복을 허용하는 방식이기 때문에 신중하게 적용해야 합니다.

먼저 실제로 성능 문제가 있는지 확인해야 합니다.
단순히 조인이 많아 보인다는 이유만으로 역정규화를 적용하기보다는, 실행 계획이나 응답 시간을 확인해서 병목이 실제로 발생하는지 검증해야 합니다.

또한 중복된 데이터를 어떻게 동기화할 것인지 정해야 합니다.
예를 들어 상품명이 변경되었는데 주문 테이블에 저장된 상품명은 그대로 남아 있다면 데이터 불일치가 발생할 수 있습니다.
따라서 데이터 변경 시 함께 갱신할지, 주문 당시의 값으로 보존할지, 배치나 이벤트를 통해 동기화할지 기준을 정해야 합니다.

마지막으로 역정규화로 인해 쓰기 로직이 복잡해질 수 있다는 점도 고려해야 합니다.
조회는 빨라질 수 있지만, 데이터를 추가하거나 수정할 때 여러 테이블이나 컬럼을 함께 관리해야 하기 때문입니다.

---

## 4. 역정규화의 장점

역정규화의 가장 큰 장점은 조회 성능을 개선할 수 있다는 점입니다.

여러 테이블을 조인하지 않고 필요한 데이터를 바로 조회할 수 있기 때문에 쿼리가 단순해지고 응답 속도가 빨라질 수 있습니다.

또한 화면에서 필요한 데이터를 미리 저장해두면 API 응답을 구성하기 쉬워지고, 복잡한 조회 로직을 줄일 수 있습니다.

---

## 5. 역정규화의 단점

역정규화의 단점은 데이터 중복이 발생한다는 점입니다.

같은 데이터가 여러 곳에 저장되면 수정, 삭제, 갱신 로직이 복잡해질 수 있습니다.
또한 동기화가 제대로 이루어지지 않으면 데이터 불일치가 발생할 수 있습니다.

저장 공간을 더 많이 사용할 수 있다는 점도 단점입니다.
특히 대량의 데이터를 중복 저장하면 스토리지 비용이 증가할 수 있습니다.

---

## 6. 정리

역정규화는 조회 성능 향상을 위해 데이터 중복을 일부 허용하는 설계 방식입니다.

조인이나 집계 비용이 커서 조회 성능 문제가 발생하는 경우, 또는 자주 조회되지만 변경이 적은 데이터가 있는 경우 역정규화를 고려할 수 있습니다.

하지만 역정규화는 데이터 일관성을 해칠 수 있기 때문에 무조건 적용하면 안 됩니다.
실제 성능 문제가 있는지 확인하고, 중복 데이터를 어떻게 동기화할지 기준을 정한 뒤 신중하게 적용해야 합니다.

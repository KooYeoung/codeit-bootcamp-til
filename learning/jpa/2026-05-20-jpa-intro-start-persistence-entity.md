## 1. SQL 중심 개발의 문제점

Java는 객체지향 언어이고, 대부분의 서비스 데이터는 관계형 데이터베이스에 저장된다.

객체는 참조를 통해 관계를 맺고, 관계형 데이터베이스는 외래 키와 조인을 통해 관계를 맺는다.

이 차이 때문에 JDBC나 MyBatis 중심 개발에서는 다음 작업이 반복된다.

- 객체 필드를 SQL 파라미터로 변환
- INSERT, UPDATE, SELECT, DELETE SQL 작성
- SQL 결과를 다시 객체로 변환
- 필드가 추가될 때마다 SQL과 매핑 코드 수정
- 객체 그래프 탐색 대신 식별자 기반 조회 반복

JPA는 이런 반복을 줄이고 객체와 관계형 데이터베이스 사이의 차이를 줄이기 위해 등장한 ORM 기술이다.

---

## 2. JPA와 ORM

ORM은 Object Relational Mapping의 약자이다.

객체와 관계형 데이터베이스를 매핑하는 기술이다.

JPA는 Java 진영의 ORM 표준 인터페이스이고, Hibernate는 대표적인 JPA 구현체이다.

정리하면 다음과 같다.

- JPA: Java ORM 표준
- Hibernate: JPA 구현체
- ORM: 객체와 관계형 DB를 매핑하는 기술

JPA를 사용하면 개발자는 엔티티 객체 중심으로 개발하고, JPA가 필요한 SQL을 생성한다.

다만 JPA는 SQL을 몰라도 되는 기술이 아니다. 성능 문제와 복잡한 조회를 해결하려면 JPA가 어떤 SQL을 실행하는지 이해해야 한다.

---

## 3. Hello JPA 기본 구성

엔티티 클래스는 다음처럼 작성한다.

```java
@Entity
public class Member {

    @Id
    private Long id;

    private String name;
}
```

`@Entity`는 JPA가 관리할 클래스라는 뜻이다.

`@Id`는 데이터베이스의 기본 키와 매핑되는 필드이다.

JPA 기본 실행 흐름은 다음과 같다.

1. `EntityManagerFactory` 생성
2. `EntityManager` 생성
3. 트랜잭션 시작
4. 엔티티 저장, 조회, 수정, 삭제
5. 커밋 또는 롤백
6. 엔티티 매니저 종료
7. 애플리케이션 종료 시 엔티티 매니저 팩토리 종료

---

## 4. EntityManagerFactory와 EntityManager

`EntityManagerFactory`는 엔티티 매니저를 만드는 공장이다.

생성 비용이 크기 때문에 애플리케이션 전체에서 하나만 생성해서 공유한다.

`EntityManager`는 실제로 엔티티를 저장, 조회, 수정, 삭제하는 작업을 수행한다.

엔티티 매니저는 쓰레드 간 공유하면 안 된다.

요청이나 트랜잭션 단위로 만들고 사용이 끝나면 닫아야 한다.

JPA의 모든 데이터 변경은 트랜잭션 안에서 실행해야 한다.

---

## 5. 영속성 컨텍스트

영속성 컨텍스트는 “엔티티를 영구 저장하는 환경”이라는 뜻이다.

```java
em.persist(member);
```

`persist()`를 호출하면 단순히 DB에 바로 INSERT 하는 것이 아니라, 엔티티가 영속성 컨텍스트에 저장되고 관리된다.

영속성 컨텍스트는 눈에 보이는 객체가 아니라 논리적인 개념이다.

개발자는 `EntityManager`를 통해 영속성 컨텍스트에 접근한다.

---

## 6. 엔티티 생명주기

엔티티는 다음 상태를 가질 수 있다.

### 비영속

영속성 컨텍스트와 전혀 관계가 없는 상태이다.

```java
Member member = new Member();
member.setId(1L);
member.setName("member1");
```

### 영속

영속성 컨텍스트에 관리되는 상태이다.

```java
em.persist(member);
```

### 준영속

영속성 컨텍스트에 저장되었다가 분리된 상태이다.

```java
em.detach(member);
em.clear();
em.close();
```

준영속 상태에서는 변경 감지 같은 영속성 컨텍스트 기능을 사용할 수 없다.

### 삭제

삭제 대상으로 관리되는 상태이다.

```java
em.remove(member);
```

---

## 7. 영속성 컨텍스트의 장점

영속성 컨텍스트는 다음 기능을 제공한다.

- 1차 캐시
- 동일성 보장
- 트랜잭션을 지원하는 쓰기 지연
- 변경 감지
- 지연 로딩

이 기능들은 JPA가 단순 SQL 생성기가 아니라 엔티티를 관리하는 환경이라는 점을 보여준다.

---

## 8. 1차 캐시와 동일성 보장

영속성 컨텍스트는 내부에 1차 캐시를 가진다.

엔티티를 저장하거나 조회하면 영속성 컨텍스트의 1차 캐시에 보관된다.

같은 트랜잭션 안에서 같은 ID로 조회하면 같은 객체 인스턴스를 반환한다.

```java
Member a = em.find(Member.class, 1L);
Member b = em.find(Member.class, 1L);

System.out.println(a == b); // true
```

이를 동일성 보장이라고 한다.

---

## 9. 쓰기 지연과 변경 감지

JPA는 `persist()`를 호출했다고 해서 즉시 INSERT SQL을 실행하지 않을 수 있다.

영속성 컨텍스트는 INSERT SQL을 쓰기 지연 SQL 저장소에 모아두고, 트랜잭션 커밋 시점에 SQL을 데이터베이스에 보낸다.

영속 상태의 엔티티는 값을 변경하면 별도의 update 호출 없이도 변경 감지가 동작한다.

```java
Member member = em.find(Member.class, 1L);
member.setName("changedName");
```

트랜잭션 커밋 시점에 JPA가 스냅샷과 현재 상태를 비교해 UPDATE SQL을 실행한다.

---

## 10. 플러시

플러시는 영속성 컨텍스트의 변경 내용을 데이터베이스에 반영하는 과정이다.

플러시가 발생하면 다음 작업이 수행된다.

1. 변경 감지가 동작한다.
2. 수정된 엔티티의 UPDATE SQL이 쓰기 지연 SQL 저장소에 등록된다.
3. 쓰기 지연 SQL 저장소의 INSERT, UPDATE, DELETE SQL이 데이터베이스로 전송된다.

플러시가 발생하는 시점은 다음과 같다.

- `em.flush()` 직접 호출
- 트랜잭션 커밋
- JPQL 쿼리 실행

플러시는 영속성 컨텍스트를 비우는 것이 아니라, 변경 내용을 데이터베이스와 동기화하는 것이다.

---

## 11. 엔티티 매핑

엔티티 매핑은 객체와 테이블을 연결하는 작업이다.

대표적인 애노테이션은 다음과 같다.

- `@Entity`
- `@Table`
- `@Column`
- `@Id`
- `@GeneratedValue`
- `@Enumerated`
- `@Lob`
- `@Transient`

`@Entity`가 붙은 클래스는 JPA가 관리하는 엔티티이다.

주의할 점은 다음과 같다.

- 기본 생성자가 필요하다.
- 기본 생성자는 public 또는 protected가 적절하다.
- final 클래스는 사용하면 안 된다.
- enum, interface, inner class는 엔티티로 사용하면 안 된다.
- 저장할 필드에 final을 사용하면 안 된다.

---

## 12. 스키마 자동 생성

JPA는 애플리케이션 실행 시점에 DDL을 자동 생성할 수 있다.

설정은 `hibernate.hbm2ddl.auto`를 사용한다.

대표 옵션은 다음과 같다.

- `create`: 기존 테이블 삭제 후 다시 생성
- `create-drop`: create와 같지만 종료 시점에 DROP
- `update`: 변경분만 반영
- `validate`: 엔티티와 테이블 매핑이 맞는지만 검증
- `none`: 사용하지 않음

운영 장비에서 `create`, `create-drop`, `update`를 사용하면 안 된다.

운영 환경에서는 보통 `validate` 또는 `none`을 사용하고, DDL은 별도 마이그레이션 도구나 DBA 관리 절차를 따르는 것이 안전하다.

---

## 13. 보충 정리

JPA를 처음 학습할 때 놓치기 쉬운 부분은 다음과 같다.

- JPA는 SQL을 모르게 해주는 기술이 아니라 SQL 작성 반복을 줄여주는 기술이다.
- 엔티티 매니저는 쓰레드 안전하지 않으므로 공유하면 안 된다.
- 영속성 컨텍스트는 트랜잭션과 함께 이해해야 한다.
- 엔티티 수정은 변경 감지로 처리하는 것이 기본이다.
- 플러시는 커밋이 아니라 DB와 SQL을 동기화하는 과정이다.
- 운영 DB 스키마 관리는 JPA 자동 생성에 의존하지 않는 것이 좋다.
- 엔티티에는 가급적 비즈니스 의미가 있는 메서드를 두고, setter를 무분별하게 열어두지 않는 것이 좋다.

---

## 14. 오늘 정리

JPA의 핵심은 객체와 관계형 데이터베이스 사이의 차이를 줄이는 것이다.

JPA를 제대로 사용하려면 단순히 `@Entity`와 `@Id`만 아는 것으로는 부족하다.

영속성 컨텍스트의 동작 원리를 이해해야 변경 감지, 쓰기 지연, 플러시, 지연 로딩 같은 기능을 제대로 사용할 수 있다.

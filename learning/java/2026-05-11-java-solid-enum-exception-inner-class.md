## SOLID 원칙

### SRP

- SRP는 단일 책임 원칙이다.
- 하나의 클래스는 하나의 책임만 가져야 하며, 변경되는 이유도 하나에 가까워야 한다는 원칙이라고 이해했다.
- 한 클래스가 여러 역할을 동시에 맡으면 요구사항이 바뀔 때 수정 범위가 커지고, 어떤 기능 때문에 코드가 바뀌는지 파악하기 어려워진다.
- 책임을 나눌 때는 단순히 메서드 수가 많다는 이유보다, 변경 이유가 서로 다른지를 기준으로 보는 것이 좋다고 느꼈다.

### OCP

- OCP는 개방-폐쇄 원칙이다.
- 기능 확장에는 열려 있어야 하고, 기존 코드 수정에는 닫혀 있어야 한다는 원칙이다.
- 새로운 기능을 추가할 때 기존 코드를 계속 수정해야 한다면 버그가 생길 가능성이 커진다.
- 인터페이스와 다형성을 활용하면 기존 코드를 크게 바꾸지 않고 새로운 구현체를 추가하는 방식으로 확장할 수 있다.
- 다만 처음부터 모든 변경 가능성을 예상해 추상화를 많이 만들면 오히려 구조가 복잡해질 수 있으므로 실제로 변화가 예상되는 지점을 기준으로 적용해야 한다.

### LSP

- LSP는 리스코프 치환 원칙이다.
- 부모 타입을 사용하는 코드에서 자식 타입 객체를 넣어도 기존 동작의 의미가 깨지지 않아야 한다는 원칙이라고 이해했다.
- 단순히 문법적으로 상속이 가능하다고 해서 항상 좋은 상속 구조가 되는 것은 아니다.
- 자식 클래스가 부모 클래스의 기본 약속을 지키지 못하면 사용하는 쪽 코드는 자식 타입을 별도로 의심해야 하므로 다형성의 장점이 줄어든다.

### ISP

- ISP는 인터페이스 분리 원칙이다.
- 하나의 인터페이스에 너무 많은 기능을 넣으면 구현 클래스가 필요 없는 메서드까지 강제로 구현해야 한다.
- 역할별로 인터페이스를 나누면 각 클래스는 자신에게 필요한 기능만 의존할 수 있다.
- 인터페이스를 분리할 때는 기능 단위보다 사용하는 클라이언트의 목적을 기준으로 나누는 것이 더 자연스럽다고 느꼈다.

### DIP

- DIP는 의존성 역전 원칙이다.
- 구체 클래스에 직접 의존하기보다 역할을 나타내는 인터페이스에 의존하는 방향으로 설계하는 원칙이다.
- 구현체를 직접 알고 있으면 구현이 바뀔 때 사용하는 코드도 함께 흔들릴 수 있다.
- 인터페이스를 사이에 두면 사용하는 쪽은 “무엇을 할 수 있는가”에 집중하고, 실제 구현은 바꿔 끼우기 쉬워진다.

## enum

### 핵심 개념

- `enum`은 미리 정해진 상수들의 집합을 타입으로 표현하는 문법이다.
- 상태, 종류, 옵션처럼 가질 수 있는 값이 제한되어 있을 때 사용하기 좋다.
- 문자열 상수나 숫자 상수보다 값의 의미가 명확하고, 허용되지 않은 값이 들어오는 실수를 줄일 수 있다.

### 내가 이해한 내용

- `enum`은 단순한 상수 묶음이 아니라 하나의 타입이다.
- 예를 들어 주문 상태가 `READY`, `SHIPPED`, `CANCELED` 중 하나만 가능하다면 문자열보다 `enum`으로 표현하는 것이 더 안전하다.
- 문자열을 사용하면 오타가 나도 컴파일 단계에서 잡기 어렵지만, `enum`은 정의된 값만 사용할 수 있다.

### 예시 코드 흐름

```java
enum OrderStatus {
    READY,
    SHIPPED,
    CANCELED
}

class Order {
    private OrderStatus status;

    Order(OrderStatus status) {
        this.status = status;
    }
}
```

### 주의할 점

- 값의 종류가 자주 바뀌거나 외부 입력 문자열을 그대로 받아야 하는 경우에는 변환 로직이 필요하다.
- `enum`에 너무 많은 책임을 넣으면 상수 타입 이상의 복잡한 객체가 될 수 있으므로 역할을 조심해서 정해야 한다.

## 예외 처리

### 핵심 개념

- 예외는 프로그램 실행 중 예상 가능한 문제가 발생했을 때 처리할 수 있는 오류 상황이다.
- 예외 처리는 오류를 없애는 것이 아니라, 문제가 생겼을 때 프로그램이 어떻게 대응할지 정하는 작업이다.
- Java에서는 `try-catch-finally`, `throws`, 사용자 정의 예외 등을 사용해 예외 흐름을 다룬다.

### try-catch-finally

- `try` 블록에는 예외가 발생할 수 있는 코드를 작성한다.
- 예외가 발생하면 해당 예외를 처리할 수 있는 `catch` 블록으로 이동한다.
- `finally` 블록은 예외 발생 여부와 상관없이 실행되어야 하는 정리 작업에 사용한다.

```java
try {
    int result = 10 / number;
    System.out.println(result);
} catch (ArithmeticException e) {
    System.out.println("0으로 나눌 수 없습니다.");
} finally {
    System.out.println("계산을 종료합니다.");
}
```

### throws

- `throws`는 메서드 내부에서 예외를 직접 처리하지 않고 호출한 쪽으로 넘기는 방식이다.
- 메서드 내부에서 복구 방법을 알기 어렵다면 호출한 쪽에서 상황에 맞게 처리하도록 넘길 수 있다.
- 예외 처리를 한곳으로 모아 흐름을 정리하는 데 도움이 될 수 있다.

### 사용자 정의 예외

- Java 표준 예외만으로 도메인의 문제 상황을 충분히 표현하기 어려울 때 사용자 정의 예외를 만들 수 있다.
- 보통 `Exception` 또는 `RuntimeException`을 상속하고, 이름은 `Exception`으로 끝나게 짓는다.
- 예외 이름만 봐도 어떤 문제가 발생했는지 드러나야 한다.

### 헷갈린 부분

- 모든 예외를 바로 `try-catch`로 감싸는 것이 좋은 것은 아니라고 느꼈다.
- 예외를 잡은 뒤 의미 있는 처리를 하지 못한다면 오히려 문제를 숨길 수 있다.
- 처리할 수 있는 위치에서는 잡고, 처리 기준이 없는 곳에서는 호출한 쪽으로 넘기는 판단이 필요하다.

## 내부 클래스와 익명 클래스

### 내부 클래스

- 내부 클래스는 클래스 안에 정의된 클래스이다.
- 외부 클래스와 밀접하게 관련된 기능을 내부에 묶어 표현할 수 있다.
- 외부 클래스의 멤버에 접근하기 쉬워 관련성이 강한 구조를 만들 수 있다.

### 종류

- 인스턴스 내부 클래스는 외부 클래스의 인스턴스가 있어야 생성할 수 있다.
- 정적 내부 클래스는 외부 클래스의 인스턴스 없이 생성할 수 있고, 외부 클래스의 정적 멤버에 접근하기 쉽다.
- 지역 내부 클래스는 메서드 내부에서 정의되며, 해당 메서드 안에서만 의미가 있을 때 사용한다.
- 익명 클래스는 이름 없이 즉석에서 정의하는 일회성 클래스이다.

### 익명 클래스

- 익명 클래스는 인터페이스나 추상 클래스의 구현이 한 번만 필요할 때 사용할 수 있다.
- 이름 있는 클래스를 따로 만들지 않아도 되어 코드가 간결해질 수 있다.
- 다만 로직이 길어지면 오히려 읽기 어려워지므로 간단한 구현에만 사용하는 것이 좋다.

```java
Runnable task = new Runnable() {
    @Override
    public void run() {
        System.out.println("작업 실행");
    }
};
```

## 오늘 정리

- SOLID 원칙은 문법이 아니라 코드의 의존 방향과 역할 분리를 판단하는 기준이라고 이해했다.
- `enum`은 제한된 값의 집합을 안전하게 표현할 때 유용하다.
- 예외 처리는 문제가 생겼을 때의 대응 흐름을 설계하는 과정이다.
- 내부 클래스와 익명 클래스는 외부 클래스와의 관계나 일회성 구현 필요성에 따라 선택해야 한다.

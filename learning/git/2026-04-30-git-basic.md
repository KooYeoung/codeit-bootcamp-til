## 1. Git이란?

Git은 파일의 변경 이력을 관리하는 분산 버전 관리 시스템이다.

개발을 하다 보면 파일을 계속 수정하게 된다.  
이때 Git을 사용하면 언제 어떤 내용이 변경되었는지 기록할 수 있고, 필요하면 이전 상태로 되돌아갈 수도 있다.

Git은 특히 소스 코드 관리에 많이 사용되며, 여러 사람이 함께 개발할 때 변경 이력을 관리하고 협업하는 데 도움이 된다.

---

## 2. 버전 관리 시스템이란?

버전 관리 시스템은 파일이나 소스 코드의 변경 내역을 기록하고 관리하는 시스템이다.

버전 관리를 사용하면 다음과 같은 장점이 있다.

- 이전 버전으로 되돌아갈 수 있다.
- 누가 어떤 내용을 수정했는지 확인할 수 있다.
- 여러 사람이 같은 프로젝트를 함께 작업할 수 있다.
- 실수로 변경한 내용을 복구할 수 있다.

버전 관리 시스템은 크게 두 가지로 나눌 수 있다.

### 중앙집중식 버전 관리 시스템

중앙 서버가 모든 버전 정보를 관리하는 방식이다.

예전에는 SVN 같은 도구가 많이 사용되었다.  
개발자는 중앙 서버에서 소스 코드를 받아 작업하고, 변경사항을 다시 중앙 서버에 반영한다.

### 분산 버전 관리 시스템

각 사용자가 전체 저장소를 복제해서 가지고 있는 방식이다.

대표적인 예시는 Git이다.

분산 버전 관리 시스템에서는 각 사용자의 컴퓨터에도 저장소 이력이 있기 때문에,
인터넷이 연결되어 있지 않아도 로컬에서 커밋을 만들고 이력을 확인할 수 있다.

## 3. Git과 GitHub의 차이

Git과 GitHub는 이름이 비슷하지만 역할이 다르다.

### Git

Git은 로컬에서 파일의 변경 이력을 관리하는 도구이다.

즉, 내 컴퓨터에서 다음과 같은 작업을 할 수 있다.

- 변경사항 확인
- 변경사항 저장
- 커밋 생성
- 브랜치 생성
- 이전 버전으로 되돌리기

### GitHub

GitHub는 Git 저장소를 온라인에서 관리하고 공유할 수 있게 해주는 서비스이다.

GitHub를 사용하면 다음과 같은 작업을 할 수 있다.

- 원격 저장소 관리
- 코드 공유
- 협업
- Pull Request
- 코드 리뷰
- Issue 관리
- GitHub Actions를 이용한 자동화

정리하면, Git은 버전 관리 도구이고 GitHub는 Git 저장소를 온라인에서 관리하는 서비스이다.

## 4. Git Repository

Git Repository는 Git이 변경 이력을 관리하는 저장소이다.

프로젝트 폴더를 Git 저장소로 만들면 Git은 해당 폴더의 변경사항을 추적할 수 있다.

```bash
git init
```

위 명령어를 실행하면 현재 폴더에 `.git` 폴더가 생성되고, Git 저장소가 된다.

---

## 5. Git Repository의 주요 영역

Git은 크게 세 가지 영역을 기준으로 동작한다.

### Working Directory

실제로 파일을 만들고 수정하는 작업 공간이다.

예를 들어, 코드를 작성하거나 파일을 삭제하는 작업은 Working Directory에서 이루어진다.

### Staging Area

커밋에 포함할 변경사항을 임시로 올려두는 영역이다.

`git add` 명령어를 사용하면 변경사항이 Staging Area에 올라간다.

### Repository

커밋이 저장되는 공간이다.

`git commit` 명령어를 사용하면 Staging Area에 있는 변경사항이 Repository에 저장된다.

---

## 6. Git 기본 작업 흐름

Git의 기본 흐름은 아래와 같다.

```text
파일 수정
→ git status로 상태 확인
→ git add로 스테이징
→ git commit으로 커밋 생성
```

자주 사용하는 명령어는 다음과 같다.

```bash
git status
git add 파일명
git add .
git commit -m "커밋 메시지"
git log
git log --oneline
```

---

## 7. git status

`git status`는 현재 Git 저장소의 상태를 확인하는 명령어이다.

```bash
git status
```

이 명령어를 사용하면 다음 내용을 확인할 수 있다.

- 현재 브랜치
- 수정된 파일
- 스테이징된 파일
- 아직 Git이 추적하지 않는 파일
- 커밋할 내용이 있는지 여부

`git status`는 저장소를 변경하지 않고 상태만 확인하는 명령어이므로 자주 사용해도 안전하다.

---

## 8. git add

`git add`는 변경사항을 Staging Area에 올리는 명령어이다.

```bash
git add 파일명
```

모든 변경사항을 한 번에 올리고 싶을 때는 아래 명령어를 사용할 수 있다.

```bash
git add .
```

다만 모든 파일을 한 번에 올리기 전에 `git status`나 `git diff`로 변경사항을 먼저 확인하는 습관이 필요하다.

---

## 9. git commit

`git commit`은 Staging Area에 올라간 변경사항을 하나의 버전으로 저장하는 명령어이다.

```bash
git commit -m "커밋 메시지"
```

커밋은 Git에서 변경사항을 저장하는 하나의 단위이다.

좋은 커밋을 만들기 위해서는 하나의 커밋에 하나의 작업 단위만 포함하는 것이 좋다.

---

## 10. Atomic Commit

Atomic Commit은 하나의 커밋에 하나의 의미 있는 변경사항만 담는 것을 말한다.

예를 들어, 기능 추가와 버그 수정을 한 번에 커밋하기보다는 각각 나누는 것이 좋다.

```bash
git add feature-file
git commit -m "Add login feature"

git add bug-fix-file
git commit -m "Fix login validation bug"
```

Atomic Commit을 지키면 다음과 같은 장점이 있다.

- 변경 이력을 이해하기 쉽다.
- 코드 리뷰가 쉬워진다.
- 문제가 발생했을 때 원인을 찾기 쉽다.
- 특정 커밋만 되돌리기 쉽다.

---

## 11. 좋은 커밋 메시지

좋은 커밋 메시지는 나중에 변경 이력을 이해하는 데 도움이 된다.

좋은 커밋 메시지를 작성할 때는 다음을 고려한다.

- 무엇을 변경했는지 명확하게 작성한다.
- 가능하면 하나의 작업 단위로 작성한다.
- 너무 길지 않게 작성한다.
- 팀이나 프로젝트의 커밋 규칙이 있다면 따른다.

예시:

```bash
git commit -m "Add Git basic workflow notes"
git commit -m "Fix typo in README"
git commit -m "Update TIL folder structure"
```

---

## 12. git commit --amend

`git commit --amend`는 직전 커밋을 수정할 때 사용하는 명령어이다.

사용할 수 있는 경우는 다음과 같다.

- 직전 커밋 메시지에 오타가 있는 경우
- 직전 커밋에 포함해야 할 파일을 빠뜨린 경우
- 직전 커밋 내용을 조금 수정해야 하는 경우

```bash
git add 빠뜨린파일
git commit --amend
```

주의할 점은 `amend`를 사용하면 기존 커밋이 수정되면서 커밋 해시가 변경된다는 것이다.

따라서 이미 원격 저장소에 push한 커밋이나 다른 사람과 공유한 커밋에는 신중하게 사용해야 한다.

## 13. .gitignore

`.gitignore`는 Git이 추적하지 않아야 하는 파일이나 폴더를 지정하는 파일이다.

주로 빌드 결과물, 로그 파일, 캐시 파일, 개인 설정 파일, 보안 정보가 포함된 파일 등을 제외할 때 사용한다.

예시:

```gitignore
*.log
.env
node_modules/
build/
dist/
```

주의할 점은 이미 커밋된 파일은 `.gitignore`에 추가해도 자동으로 추적 대상에서 제외되지 않는다는 것이다.

이미 Git이 추적 중인 파일을 제외하려면 다음 명령어를 사용할 수 있다.

```bash
git rm --cached 파일명
```

이 명령어는 내 로컬 파일은 삭제하지 않고, Git의 추적 대상에서만 제거한다.  
다만 이 변경사항을 커밋하고 push하면 저장소에서는 해당 파일이 삭제된 것으로 기록된다.  
따라서 협업자가 이후에 pull을 받을 경우, 협업자의 작업 폴더에서도 해당 파일이 삭제될 수 있으므로 주의해야 한다.

개인 설정 파일은 Git에서 제외하고, 대신 예시 파일을 함께 제공하는 것이 좋다.

```text
.env          # 실제 개인 설정 파일, Git 추적 제외
.env.example  # 공유 가능한 예시 파일, Git에 포함
```

특정 파일만 예외로 추적하고 싶을 때는 `!`를 사용한다.

```gitignore
*.log
!important.log
```

위 설정은 모든 `.log` 파일은 제외하되, `important.log` 파일은 Git에 포함한다는 의미이다.

폴더 안의 특정 파일만 예외로 포함하려면 다음과 같이 작성할 수 있다.

```gitignore
logs/*
!logs/important.log
```

위 설정은 `logs` 폴더 안의 파일들은 제외하되, `logs/important.log` 파일은 Git에 포함한다는 의미이다.

## 14. Branch

Branch는 독립적인 작업 공간이다.

새로운 기능을 개발하거나 실험적인 작업을 할 때 main 브랜치에서 바로 작업하지 않고 별도의 브랜치를 만들어 작업할 수 있다.

브랜치를 사용하면 main 브랜치를 안정적으로 유지하면서 새로운 작업을 분리해서 진행할 수 있다.

### 브랜치 목록 확인

현재 로컬 브랜치 목록은 다음 명령어로 확인할 수 있다.

```bash
git branch
```

현재 내가 위치한 브랜치에는 `*` 표시가 붙는다.

원격 브랜치까지 함께 확인하려면 다음 명령어를 사용할 수 있다.

```bash
git branch -a
```

### 브랜치 생성 및 이동

브랜치를 생성하려면 다음 명령어를 사용한다.

```bash
git branch 브랜치명
```

생성한 브랜치로 이동하려면 다음 명령어를 사용한다.

```bash
git switch 브랜치명
```

브랜치를 만들면서 바로 이동하려면 다음 명령어를 사용한다.

```bash
git switch -c 브랜치명
```

### 브랜치 이름 변경

현재 위치한 브랜치의 이름을 변경하려면 다음 명령어를 사용한다.

```bash
git branch -m 새브랜치명
```

다른 브랜치의 이름을 변경하려면 다음 명령어를 사용한다.

```bash
git branch -m 기존브랜치명 새브랜치명
```

### 브랜치 삭제

작업이 끝난 로컬 브랜치는 다음 명령어로 삭제할 수 있다.

```bash
git branch -d 브랜치명
```

`-d` 옵션은 병합이 완료된 브랜치만 삭제한다.

아직 병합되지 않은 브랜치를 강제로 삭제하려면 다음 명령어를 사용한다.

```bash
git branch -D 브랜치명
```

단, `-D`는 병합되지 않은 작업 내용이 사라질 수 있으므로 주의해서 사용해야 한다.

## 15. HEAD

HEAD는 현재 Git이 바라보고 있는 위치를 의미한다.

일반적으로 HEAD는 현재 브랜치의 최신 커밋을 가리킨다.

예를 들어 main 브랜치에서 작업 중이라면 HEAD는 main 브랜치의 최신 커밋을 바라본다.

```bash
HEAD
HEAD~1
HEAD~2
```

- `HEAD`: 현재 커밋
- `HEAD~1`: 현재 커밋의 부모 커밋
- `HEAD~2`: 현재 커밋의 부모의 부모 커밋

---

## 16. checkout과 switch

예전에는 브랜치 이동에 `checkout`을 많이 사용했다.

```bash
git checkout 브랜치명
```

하지만 `checkout`은 브랜치 이동 외에도 특정 커밋으로 이동하거나 파일을 되돌리는 등 여러 기능을 가지고 있어 초보자에게 헷갈릴 수 있다.

그래서 Git 2.23 이후에는 브랜치 이동 용도로 `switch` 명령어를 사용할 수 있다.

```bash
git switch 브랜치명
```

정리하면 다음과 같다.

- `git switch`: 브랜치 이동에 사용
- `git checkout`: 브랜치 이동, 특정 커밋 이동, 파일 복원 등 여러 기능을 가짐

## 17. Merge

Merge는 다른 브랜치의 작업 내용을 현재 브랜치에 합치는 작업이다.

예를 들어 feature 브랜치에서 작업한 내용을 main 브랜치에 합치려면 main 브랜치로 이동한 뒤 merge를 실행한다.

```bash
git switch main
git merge feature
```

중요한 점은 병합을 받을 브랜치로 먼저 이동해야 한다는 것이다.

---

## 18. Fast-Forward Merge

Fast-Forward Merge는 현재 브랜치에 추가 커밋이 없고, 병합하려는 브랜치만 앞서 있는 경우 발생한다.

이 경우 Git은 별도의 병합 커밋을 만들지 않고 브랜치 포인터만 앞으로 이동시킨다.

```bash
git switch main
git merge feature
```

위 상황에서 main 브랜치에 추가 커밋이 없고 feature 브랜치만 앞서 있다면 Fast-Forward Merge가 발생한다.

장점은 히스토리가 한 줄로 깔끔하게 유지된다는 것이다.

단점은 별도의 병합 커밋이 남지 않기 때문에, 어떤 브랜치에서 작업한 내용이 병합되었는지 이력이 명확하게 남지 않을 수 있다는 것이다.

Fast-Forward가 가능한 상황에서도 병합 커밋을 남기고 싶다면 `--no-ff` 옵션을 사용할 수 있다.

```bash
git merge --no-ff feature
```

`--no-ff` 옵션을 사용하면 Git은 브랜치 포인터만 이동하지 않고, 별도의 merge commit을 생성한다.

이렇게 하면 커밋 히스토리에 feature 브랜치가 병합되었다는 기록이 남기 때문에, 나중에 어떤 작업 단위가 병합되었는지 확인하기 쉽다.

정리하면 다음과 같다.

```bash
git merge feature
```

- Fast-Forward가 가능하면 병합 커밋 없이 포인터만 이동한다.
- 히스토리가 깔끔하다.
- 브랜치 병합 이력이 명확하게 남지 않을 수 있다.

```bash
git merge --no-ff feature
```

- Fast-Forward가 가능해도 병합 커밋을 생성한다.
- 브랜치 병합 이력이 명확하게 남는다.
- 커밋 히스토리가 조금 더 복잡해질 수 있다.

---

## 19. 3-Way Merge

3-Way Merge는 두 브랜치가 공통 조상 이후 서로 다른 커밋을 가진 상태에서 병합할 때 발생한다.

이 경우 Git은 두 브랜치의 변경사항과 공통 조상을 비교해서 병합 커밋을 생성한다.

만약 같은 파일의 같은 부분을 서로 다르게 수정했다면 conflict가 발생할 수 있다.

---

## 20. Conflict

Conflict는 Git이 자동으로 병합할 수 없는 충돌 상황을 말한다.

주로 두 브랜치에서 같은 파일의 같은 부분을 다르게 수정했을 때 발생한다.

충돌이 발생하면 파일 안에 아래와 같은 표시가 생긴다.

```text
<<<<<<< HEAD
현재 브랜치 내용
=======
병합하려는 브랜치 내용
>>>>>>> branch-name
```

충돌 해결 순서는 다음과 같다.

```text
충돌 파일 확인
→ 원하는 내용으로 직접 수정
→ git add
→ git commit
```

---

## 21. Git Diff

`git diff`는 변경 전후 차이를 확인할 때 사용하는 명령어이다.

```bash
git diff
```

주요 사용법은 다음과 같다.

```bash
git diff
git diff HEAD
git diff --staged
git diff 브랜치1..브랜치2
git diff 커밋1..커밋2
```

### git diff

아직 스테이징하지 않은 변경사항을 확인한다.

### git diff HEAD

최근 커밋과 현재 작업 상태의 차이를 확인한다.

스테이징된 변경사항과 스테이징되지 않은 변경사항을 모두 확인할 수 있다.

### git diff --staged

스테이징된 변경사항을 확인한다.

커밋하기 전에 어떤 내용이 커밋될지 확인할 때 유용하다.

---

## 22. Git Stash

`git stash`는 아직 커밋하지 않은 변경사항을 임시로 저장할 때 사용한다.

작업 중이던 내용을 커밋하지 않고 잠시 보관한 뒤, 다른 브랜치로 이동해야 할 때 유용하다.

```bash
git stash
```

stash한 내용을 다시 적용하려면 다음 명령어를 사용할 수 있다.

```bash
git stash apply
```

stash를 적용하면서 목록에서 제거하려면 다음 명령어를 사용한다.

```bash
git stash pop
```

stash 목록을 확인하려면 다음 명령어를 사용한다.

```bash
git stash list
```

stash 목록은 보통 다음과 같은 형태로 출력된다.

```text
stash@{0}: WIP on main: ...
stash@{1}: WIP on feature: ...
stash@{2}: WIP on test: ...
```

여기서 `stash@{0}`은 가장 최근에 저장한 stash를 의미한다.

특정 stash를 적용하려면 인덱스를 지정해서 사용할 수 있다.

```bash
git stash apply stash@{번호}
```

예시:

```bash
git stash apply stash@{1}
```

위 명령어는 `stash@{1}`에 저장된 변경사항을 다시 적용한다.  
단, `apply`는 적용 후에도 stash 목록에서 해당 항목을 제거하지 않는다.

특정 stash를 적용하면서 목록에서 제거하려면 다음과 같이 사용한다.

```bash
git stash pop stash@{번호}
```

예시:

```bash
git stash pop stash@{1}
```

위 명령어는 `stash@{1}`에 저장된 변경사항을 적용한 뒤, stash 목록에서 제거한다.

특정 stash를 삭제하려면 다음 명령어를 사용한다.

```bash
git stash drop stash@{번호}
```

예시:

```bash
git stash drop stash@{1}
```

정리하면 다음과 같다.

- `git stash`: 현재 변경사항 임시 저장
- `git stash list`: stash 목록 확인
- `git stash apply`: 가장 최근 stash 다시 적용, stash 목록에는 유지
- `git stash apply stash@{번호}`: 특정 stash 다시 적용, stash 목록에는 유지
- `git stash pop`: 가장 최근 stash 다시 적용, stash 목록에서 제거
- `git stash pop stash@{번호}`: 특정 stash 다시 적용, stash 목록에서 제거
- `git stash drop stash@{번호}`: 특정 stash 삭제

## 23. Detached HEAD

Detached HEAD는 특정 브랜치가 아니라 특정 커밋을 직접 바라보는 상태이다.

예를 들어 특정 커밋 해시로 checkout하면 Detached HEAD 상태가 된다.

```bash
git checkout 커밋해시
```

이 상태에서는 예전 커밋의 파일을 살펴볼 수 있다.

하지만 Detached HEAD 상태에서 새 커밋을 만들면 그 커밋은 특정 브랜치에 연결되지 않기 때문에 나중에 찾기 어려울 수 있다.

작업을 이어가고 싶다면 새 브랜치를 만들어야 한다.

```bash
git switch -c 새브랜치명
```

---

## 24. Git Restore

`git restore`는 작업 디렉토리의 변경사항을 되돌릴 때 사용한다.

```bash
git restore 파일명
```

위 명령어를 사용하면 해당 파일이 마지막 커밋 상태로 되돌아간다.

스테이징된 파일을 다시 unstaged 상태로 돌릴 때는 다음 명령어를 사용한다.

```bash
git restore --staged 파일명
```

특정 커밋의 파일 상태로 복원하려면 다음 명령어를 사용할 수 있다.

```bash
git restore --source 커밋해시 파일명
```

주의할 점은 `git restore`로 되돌린 작업 디렉토리의 변경사항은 복구하기 어렵다는 것이다.

---

## 25. Git Reset

`git reset`은 HEAD를 특정 커밋으로 이동시키는 명령어이다.

```bash
git reset 커밋해시
```

옵션을 지정하지 않으면 기본적으로 `--mixed` 방식으로 동작한다.

### reset 옵션

```bash
git reset --soft 커밋해시
git reset --mixed 커밋해시
git reset --hard 커밋해시
```

### --soft

커밋 이력만 되돌리고, 변경사항은 Staging Area에 남겨둔다.

### --mixed

커밋 이력과 Staging Area를 되돌리고, 변경사항은 Working Directory에 남겨둔다.

### --hard

커밋 이력, Staging Area, Working Directory를 모두 되돌린다.

`--hard`는 작업 내용이 사라질 수 있으므로 매우 주의해서 사용해야 한다.

---

## 26. Git Revert

`git revert`는 기존 커밋 이력은 유지한 채, 특정 커밋의 변경사항을 반대로 적용한 새로운 커밋을 생성하는 명령어이다.

```bash
git revert 취소할커밋해시
```

`reset`과 달리 기존 커밋 이력을 삭제하지 않는다.  
대신 지정한 커밋의 변경사항을 반대로 적용한 새 커밋을 만든다.

예를 들어 어떤 커밋이 파일에 내용을 추가했다면, `revert`는 그 내용을 제거하는 새 커밋을 만든다.

반대로 어떤 커밋이 내용을 삭제했다면, `revert`는 그 내용을 다시 추가하는 새 커밋을 만든다.

따라서 이미 원격 저장소에 올라갔거나 다른 사람과 공유된 커밋을 되돌릴 때는 `reset`보다 `revert`를 사용하는 것이 안전하다.

---

## 27. Reset과 Revert의 차이

`reset`과 `revert`는 모두 이전 상태로 되돌릴 때 사용할 수 있지만, 동작 방식이 다르다.

| 구분 | reset | revert |
| --- | --- | --- |
| 동작 방식 | HEAD를 특정 커밋으로 이동 | 특정 커밋의 변경사항을 반대로 적용한 새 커밋 생성 |
| 기존 이력 | 이후 커밋이 브랜치 이력에서 사라진 것처럼 보임 | 기존 커밋 이력을 그대로 유지 |
| 주 사용 상황 | 개인 로컬 작업 정리 | 공유된 커밋의 변경사항 되돌리기 |
| 협업 안정성 | 주의 필요 | 비교적 안전 |
| 기준 커밋 | 돌아가고 싶은 커밋 | 변경사항을 되돌리고 싶은 커밋 |

정리하면, 개인 작업에서 아직 원격 저장소에 공유하지 않은 커밋은 `reset`으로 정리할 수 있다.

하지만 이미 원격 저장소에 올라갔거나 다른 사람과 공유한 커밋은 기존 이력을 변경하지 않는 `revert`를 사용하는 것이 안전하다.

---

## 28. 오늘 정리

오늘은 Git의 기본 개념부터 변경사항을 관리하고 되돌리는 방법까지 학습했다.

가장 중요한 흐름은 다음과 같다.

```text
수정
→ git status
→ git add
→ git commit
```

그리고 변경사항을 다룰 때는 다음 명령어의 차이를 구분해야 한다.

- `git diff`: 변경사항 비교
- `git stash`: 변경사항 임시 저장
- `git restore`: 커밋하지 않은 변경사항 되돌리기
- `git reset`: 커밋 이력 자체를 특정 시점으로 이동
- `git revert`: 기존 커밋 이력은 유지한 채, 특정 커밋의 변경사항을 되돌리는 새 커밋 생성

Git을 사용할 때는 무작정 명령어를 실행하기보다, 먼저 현재 상태를 확인하는 습관이 중요하다.

```bash
git status
git log --oneline
git diff
```

특히 `reset --hard`, `restore`처럼 작업 내용이 사라질 수 있는 명령어는 실행 전에 반드시 상태를 확인해야 한다.

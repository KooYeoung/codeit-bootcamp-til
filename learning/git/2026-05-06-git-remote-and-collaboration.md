## 1. GitHub의 역할

GitHub는 Git 저장소를 온라인에서 관리할 수 있게 해주는 코드 호스팅 서비스이다.

Git 자체는 내 컴퓨터에서 버전 관리를 수행하는 도구이고, GitHub는 Git 저장소를 원격에서 공유하고 협업할 수 있게 해주는 서비스이다.

GitHub에서 할 수 있는 주요 기능은 다음과 같다.

- 코드 호스팅
- 버전 관리
- 협업
- 이슈 트래킹
- 프로젝트 관리
- 외부 도구와의 통합 및 확장

즉, GitHub는 단순히 코드를 올려두는 공간이 아니라 여러 사람이 함께 개발하기 위한 협업 플랫폼으로 볼 수 있다.

---

## 2. 로컬 저장소와 원격 저장소

### 로컬 저장소

로컬 저장소는 개인 컴퓨터에 위치한 Git 저장소이다.

내 컴퓨터 안에서 파일을 수정하고, staging하고, commit을 만드는 작업은 로컬 저장소에서 이루어진다.

로컬 저장소 안에는 `.git` 디렉토리가 존재하며, 이 안에 버전 히스토리, 브랜치 정보, 설정 정보 등이 저장된다.

### 원격 저장소

원격 저장소는 GitHub 같은 외부 서버에 위치한 저장소이다.

여러 사람이 같은 원격 저장소를 기준으로 작업 내용을 공유할 수 있다.

로컬 저장소에서 만든 커밋을 원격 저장소에 올릴 때는 `git push`를 사용한다.

원격 저장소는 협업을 위한 공간이면서 동시에 코드 백업 역할도 한다.

정리하면 다음과 같다.

- 로컬 저장소: 개인 작업 공간
- 원격 저장소: 공유 및 협업 공간

---

## 3. git clone

`git clone`은 원격 저장소의 내용을 내 컴퓨터로 복사하는 명령어이다.

원격 저장소를 처음 가져올 때 사용한다.

`clone`을 실행하면 원격 저장소의 파일뿐만 아니라 Git 이력과 원격 저장소 연결 정보도 함께 가져온다.

```bash
git clone 원격저장소주소
```

`clone`을 하면 새로운 디렉토리가 생성되고, 그 안에서 Git 저장소로 작업할 수 있다.

기본적으로 clone 후에는 원격 저장소 이름이 `origin`으로 등록된다.

---

## 4. git remote

`git remote`는 로컬 저장소와 연결된 원격 저장소 정보를 확인하거나 관리하는 명령어이다.

자주 사용하는 명령어는 다음과 같다.

- `git remote`
- `git remote -v`
- `git remote add origin 원격저장소주소`
- `git remote show origin`
- `git remote remove 이름`
- `git remote rename 기존이름 새이름`

`git remote -v`를 사용하면 현재 연결된 원격 저장소 주소를 확인할 수 있다.

결과가 나오지 않으면 현재 로컬 저장소에 연결된 원격 저장소가 없는 상태이다.

원격 저장소를 추가할 때는 보통 다음 형식을 사용한다.

```bash
git remote add origin 원격저장소주소
```

여기서 `origin`은 원격 저장소 이름이다.

---

## 5. origin

`origin`은 원격 저장소 이름으로 자주 사용되는 관례적인 이름이다.

반드시 `origin`이라는 이름을 사용해야 하는 것은 아니지만, 대부분의 프로젝트에서는 기본 원격 저장소 이름으로 `origin`을 사용한다.

예를 들어 다음 명령어는 `origin`이라는 원격 저장소로 현재 브랜치의 커밋을 올리는 의미이다.

```bash
git push origin main
```

정리하면 다음과 같다.

- `origin`: 원격 저장소의 이름
- `main`: 원격 저장소에 올릴 브랜치 이름

---

## 6. git push

`git push`는 로컬 저장소의 커밋을 원격 저장소로 업로드하는 명령어이다.

즉, 내 컴퓨터에서 작업한 내용을 GitHub 같은 원격 저장소에 반영할 때 사용한다.

```bash
git push origin main
```

현재 브랜치 기준으로 push할 수도 있지만, 원격 저장소 이름과 브랜치명을 명시하면 실수를 줄일 수 있다.

push를 하기 전에 보통 다음 상태를 확인하는 것이 좋다.

- 현재 브랜치가 맞는지
- 커밋할 변경사항이 남아 있지 않은지
- 원격 저장소 주소가 맞는지
- 다른 사람이 먼저 push한 내용이 있는지

작업 내용이 커밋되지 않은 상태에서는 push할 수 없다.

push는 커밋 단위로 원격 저장소에 반영된다.

---

## 7. 원격 추적 브랜치

원격 추적 브랜치는 원격 저장소의 브랜치 상태를 로컬에서 참조하기 위한 브랜치이다.

예를 들어 `origin/main`은 원격 저장소 `origin`의 `main` 브랜치 상태를 나타낸다.

원격 브랜치 목록은 다음 명령어로 확인할 수 있다.

```bash
git branch -r
```

로컬 브랜치와 원격 브랜치를 모두 확인하려면 다음 명령어를 사용할 수 있다.

```bash
git branch -a
```

원격 추적 브랜치는 원격 저장소의 상태를 로컬에서 확인하기 위한 참조이며, 직접 작업하는 브랜치는 아니다.

실제 작업은 로컬 브랜치에서 진행한다.

예를 들어 원격에 `feature/login` 브랜치가 있다면, 로컬에서 같은 이름의 브랜치를 만들어 추적할 수 있다.

```bash
git switch feature/login
```

이 명령은 Git이 원격 브랜치를 추론할 수 있는 경우 편리하게 사용할 수 있다.

원격 브랜치를 기준으로 로컬 브랜치를 만들 때는 다음처럼 명시할 수 있다.

```bash
git switch -c feature/login origin/feature/login
```

---

## 8. git fetch

`git fetch`는 원격 저장소의 최신 정보를 가져오는 명령어이다.

중요한 점은 `fetch`는 원격 저장소의 변경사항을 가져오기만 하고, 내 현재 작업 브랜치에 자동으로 병합하지 않는다는 것이다.

```bash
git fetch origin
```

`git fetch`를 실행하면 원격 브랜치의 최신 상태가 로컬의 원격 추적 브랜치에 반영된다.

예를 들어 원격의 `main` 브랜치가 바뀌었다면, `origin/main`이 최신 상태로 갱신된다.

하지만 내 로컬 `main` 브랜치가 자동으로 바뀌지는 않는다.

정리하면 다음과 같다.

- `fetch`: 원격 정보를 가져오기만 함
- 현재 브랜치에 자동 병합하지 않음
- 변경 내용을 확인한 뒤 직접 merge 또는 rebase할 수 있음

---

## 9. git pull

`git pull`은 원격 저장소의 변경사항을 가져오고, 현재 브랜치에 반영하는 명령어이다.

일반적으로 `git pull`은 다음 흐름으로 이해할 수 있다.

```text
git fetch
+
git merge
```

즉, `pull`은 원격 저장소의 최신 내용을 가져온 뒤 현재 브랜치에 병합한다.

설정에 따라 `merge` 대신 `rebase` 방식으로 동작할 수도 있다.

`pull`은 편리하지만, 자동으로 병합을 시도하기 때문에 충돌이 발생할 수 있다.

따라서 협업 상황에서는 먼저 `git fetch`로 원격 변경사항을 확인한 뒤, 필요한 방식으로 병합하는 흐름이 더 안전할 수 있다.

정리하면 다음과 같다.

- `git fetch`: 가져오기만 함
- `git pull`: 가져오고 병합까지 함

## 10. collaborators

GitHub에서 collaborators는 저장소에 직접 협업자로 초대된 사람을 의미한다.

작은 규모의 협업에서는 collaborators 방식이 유용할 수 있다.

collaborators는 권한에 따라 다음 작업을 할 수 있다.

- 코드 push
- issue 관리
- Pull Request 관리
- Wiki 및 Pages 관리

다만 collaborators 방식은 저장소에 직접 접근 권한을 주는 방식이므로, 권한 관리가 중요하다.

프로젝트 규모가 커지거나 외부 기여자가 많아지면 Fork와 Pull Request 방식을 사용하는 경우도 많다.

## 11. Git Workflow

Git Workflow는 여러 사람이 Git을 사용해 협업하는 흐름 또는 전략을 의미한다.

대표적인 방식에는 다음이 있다.

- 중앙 집중형 워크플로우
- Feature Branch Workflow
- Git Flow
- Forking Workflow

각 방식은 프로젝트 규모, 배포 방식, 팀의 협업 스타일에 따라 선택할 수 있다.

---

## 12. 중앙 집중형 워크플로우

중앙 집중형 워크플로우는 모든 개발자가 하나의 중심 브랜치를 기준으로 작업하는 방식이다.

보통 `main` 또는 `master` 브랜치에서 작업이 이루어진다.

장점은 구조가 단순하다는 것이다.

하지만 여러 사람이 같은 브랜치에 직접 작업하면 충돌이 발생할 가능성이 높다.

또한 안정적인 main 브랜치를 유지하기 어렵다.

정리하면 다음과 같다.

- 구조가 단순함
- 초보자가 이해하기 쉬움
- 충돌 가능성이 높음
- main 브랜치 안정성을 유지하기 어려움

---

## 13. Feature Branch Workflow

Feature Branch Workflow는 기능별로 브랜치를 만들어 작업하는 방식이다.

새로운 기능을 개발할 때 main 브랜치에서 직접 작업하지 않고 별도 feature 브랜치를 만든다.

예시:

```text
feature/login
feature/post-create
feature/comment
```

작업이 끝나면 Pull Request를 통해 main 브랜치에 병합한다.

이 방식의 장점은 main 브랜치를 안정적으로 유지하면서 기능 개발을 분리할 수 있다는 것이다.

또한 Pull Request를 통해 코드 리뷰를 진행할 수 있다.

중앙 집중형 워크플로우와 비교하면 다음과 같다.

- 중앙 집중형 워크플로우
  - main 브랜치에서 직접 작업
  - 충돌 가능성 높음
  - main 안정성 유지 어려움
- Feature Branch Workflow
  - 기능별 브랜치에서 작업
  - main 브랜치 안정성 유지에 유리
  - Pull Request 기반 코드 리뷰 가능

## 14. Git Flow

Git Flow는 브랜치를 역할별로 나누어 관리하는 전략이다.

대표적인 브랜치는 다음과 같다.

- `main`
- `develop`
- `feature`
- `release`
- `hotfix`

### main

`main` 브랜치는 배포 가능한 안정적인 코드를 관리한다.

운영 환경에 배포되는 코드는 보통 main 브랜치를 기준으로 한다.

### develop

`develop` 브랜치는 다음 배포를 위한 개발 내용을 모으는 브랜치이다.

여러 feature 브랜치가 develop 브랜치로 병합된다.

### feature

`feature` 브랜치는 새로운 기능을 개발할 때 사용한다.

기능 개발이 끝나면 develop 브랜치에 병합한다.

### release

`release` 브랜치는 배포 준비 단계에서 사용한다.

배포 전 테스트, 버그 수정, 버전 정리 등을 진행한다.

### hotfix

`hotfix` 브랜치는 운영 중 긴급 수정이 필요할 때 사용한다.

main 브랜치에서 분기하여 수정한 뒤 main과 develop에 반영한다.

Git Flow는 구조가 명확하지만, 브랜치가 많아지기 때문에 작은 프로젝트에서는 다소 복잡할 수 있다.

---

## 15. Pull Request

Pull Request는 내가 작업한 브랜치의 변경사항을 다른 브랜치에 병합해달라고 요청하는 기능이다.

예를 들어 `feature/login` 브랜치에서 로그인 기능을 개발한 뒤, 이를 `main` 브랜치에 병합해달라고 요청할 수 있다.

Pull Request를 사용하면 다음을 할 수 있다.

- 변경사항 확인
- 코드 리뷰
- 피드백 반영
- 테스트 결과 확인
- 병합 기록 관리

Pull Request는 단순히 병합 요청만 하는 기능이 아니라 협업과 코드 리뷰의 중심 역할을 한다.

---

## 16. Fork

Fork는 다른 사람의 원격 저장소를 내 GitHub 계정으로 복사하는 기능이다.

원본 저장소에 직접 push 권한이 없을 때 Fork를 사용한다.

Fork 후 내 저장소에서 작업하고, 원본 저장소에 Pull Request를 보낼 수 있다.

흐름은 다음과 같다.

```text
원본 저장소 Fork
→ 내 계정의 저장소로 복사
→ 내 로컬로 clone
→ 브랜치 생성 후 작업
→ 내 원격 저장소로 push
→ 원본 저장소에 Pull Request 생성
```

Fork는 오픈소스 프로젝트나 외부 기여 방식에서 자주 사용된다.

---

## 17. 오늘 정리

오늘은 GitHub와 원격 저장소를 이용한 협업 흐름을 학습했다.

가장 중요한 흐름은 다음과 같다.

```text
로컬 저장소에서 작업
→ commit 생성
→ 원격 저장소로 push
→ Pull Request 생성
→ 코드 리뷰
→ main 브랜치에 병합
```

특히 `fetch`와 `pull`의 차이를 구분하는 것이 중요하다.

`fetch`는 원격 정보를 가져오기만 하고, `pull`은 가져온 뒤 병합까지 진행한다.

협업에서는 main 브랜치에 직접 작업하기보다 기능별 브랜치를 만들고 Pull Request를 통해 병합하는 방식이 더 안전하다고 이해했다.

또한 원격 저장소는 단순한 백업 공간이 아니라 여러 사람이 함께 작업하기 위한 협업 공간이라는 점을 정리했다.

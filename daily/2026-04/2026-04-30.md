## 오늘 배운 것

### Git과 버전 관리

- Git은 파일의 변경 이력을 관리하는 분산 버전 관리 시스템이다.
- 버전 관리를 사용하면 특정 시점의 코드로 되돌아가거나, 변경 이력을 추적할 수 있다.
- Git은 로컬에서 버전 관리를 수행하는 도구이고, GitHub는 Git 저장소를 온라인에서 관리하고 협업할 수 있게 해주는 서비스이다.

### Git 기본 흐름

- Git의 기본 작업 흐름은 `수정 → 상태 확인 → 스테이징 → 커밋` 순서로 진행된다.
- `git status`를 사용하면 현재 파일의 변경 상태를 확인할 수 있다.
- `git add`는 변경사항을 스테이징 영역에 올릴 때 사용한다.
- `git commit`은 스테이징된 변경사항을 하나의 버전으로 기록할 때 사용한다.

### Git Repository

- Git Repository는 Git이 프로젝트의 변경 이력을 관리하는 저장소이다.
- `git init`을 사용하면 현재 폴더를 Git 저장소로 초기화할 수 있다.
- `.git` 폴더에는 Git의 메타데이터와 변경 이력이 저장되므로 직접 수정하거나 삭제하지 않아야 한다.

### .gitignore

- `.gitignore`는 Git이 추적하지 않아야 하는 파일이나 폴더를 지정할 때 사용한다.
- 로그 파일, 빌드 결과물, 개인 설정 파일 등은 보통 Git에 포함하지 않는다.
- 이미 커밋된 파일은 `.gitignore`에 추가해도 자동으로 제외되지 않기 때문에 별도 처리가 필요하다.

### Commit

- 커밋은 변경사항을 저장하는 하나의 단위이다.
- 하나의 커밋에는 하나의 작업 단위만 포함하는 것이 좋다.
- 좋은 커밋 메시지는 나중에 변경 이력을 이해하는 데 도움이 된다.
- `git commit --amend`를 사용하면 직전 커밋 메시지나 내용을 수정할 수 있다.

### Branch

- 브랜치는 독립적으로 작업할 수 있는 작업 공간이다.
- 새로운 기능 개발이나 실험적인 작업은 별도의 브랜치에서 진행하는 것이 좋다.
- `git branch`로 브랜치 목록을 확인하거나 새 브랜치를 만들 수 있다.
- `git switch`로 브랜치를 이동할 수 있다.
- `git switch -c 브랜치명`을 사용하면 브랜치를 생성하면서 바로 이동할 수 있다.

### Merge

- Merge는 다른 브랜치의 작업 내용을 현재 브랜치에 합치는 작업이다.
- Fast-Forward Merge는 현재 브랜치에 추가 커밋이 없을 때 포인터만 앞으로 이동하는 방식이다.
- 3-Way Merge는 두 브랜치가 서로 다른 커밋을 가진 상태에서 병합 커밋을 생성하는 방식이다.
- 충돌이 발생하면 충돌 파일을 직접 수정한 뒤 `git add`, `git commit`으로 병합을 완료해야 한다.

### Git Diff

- `git diff`는 변경 전후의 차이를 확인할 때 사용한다.
- 커밋하기 전에 어떤 부분이 변경되었는지 확인하는 데 유용하다.
- `git diff HEAD`는 최근 커밋과 현재 작업 상태의 차이를 확인할 때 사용한다.
- `git diff --staged`는 스테이징된 변경사항을 확인할 때 사용한다.

### Git Stash

- `git stash`는 아직 커밋하지 않은 변경사항을 임시로 저장할 때 사용한다.
- 작업 중인 내용을 잠시 보관하고 다른 브랜치로 이동해야 할 때 유용하다.
- `git stash apply`는 stash 내용을 다시 적용하되 stash 목록에는 남겨둔다.
- `git stash pop`은 stash 내용을 다시 적용하고 목록에서 제거한다.

### Git Restore / Reset

- `git restore`는 작업 디렉토리의 변경사항을 되돌릴 때 사용한다.
- `git restore --staged`는 스테이징된 파일을 다시 unstaged 상태로 되돌릴 때 사용한다.
- `git reset`은 HEAD를 특정 커밋으로 이동시켜 커밋 이력을 되돌릴 때 사용한다.
- `reset`은 옵션에 따라 작업 디렉토리와 스테이징 영역에 미치는 영향이 달라지므로 주의해서 사용해야 한다.

### Git Revert

- `git revert`는 기존 커밋 이력은 유지한 채, 특정 커밋의 변경사항을 반대로 적용한 새로운 커밋을 생성하는 명령어이다.
- `reset`처럼 기존 커밋을 삭제하거나 이력을 바꾸는 것이 아니라, 지정한 커밋의 변경사항을 반대로 적용한 새 커밋을 만든다.
- 이미 원격 저장소에 올라갔거나 다른 사람과 공유된 커밋을 되돌릴 때는 `reset`보다 `revert`를 사용하는 것이 안전하다.
- `git revert <취소할 커밋 해시>` 형태로 사용한다.
- `reset`은 돌아가고 싶은 커밋 해시를 지정하지만, `revert`는 취소하고 싶은 커밋 해시를 지정한다.

### Reset과 Revert의 차이

- `git reset`은 특정 커밋으로 HEAD를 이동시켜 이후 커밋을 브랜치 이력에서 제거하는 방식이다.
- `git revert`는 기존 커밋 이력을 유지하면서 특정 커밋의 변경사항을 반대로 적용한 새 커밋을 만든다.
- 개인 로컬 작업에서 아직 공유하지 않은 커밋을 되돌릴 때는 `reset`을 사용할 수 있다.
- 협업 중이거나 이미 공유된 커밋을 되돌릴 때는 `revert`를 사용하는 것이 안전하다.

## 어려웠던 점

- `git add`와 `git commit`의 차이는 이해했지만, 어떤 기준으로 커밋을 나누어야 하는지는 더 연습이 필요하다고 느꼈다.
- Branch와 Merge는 개념은 이해했지만, Fast-Forward Merge와 3-Way Merge를 실제 상황에서 구분하는 부분이 어려웠다.
- `reset`과 `revert`가 모두 커밋을 되돌리는 명령어처럼 보였지만, 실제로는 이력을 변경하는 방식이 달라 헷갈렸다.
- `reset`은 돌아갈 커밋을 기준으로 사용하고, `revert`는 취소할 커밋을 기준으로 사용한다는 점을 구분하는 데 시간이 필요했다.

## 해결 방법

- 커밋은 가능한 한 하나의 작업 단위로 나누어 작성해야 한다는 점을 기억하기로 했다.
- 브랜치는 독립적인 작업 공간, merge는 브랜치의 작업 내용을 합치는 과정으로 구분해서 이해했다.
- `reset`은 커밋 이력을 되돌리는 방식이고, `revert`는 취소 커밋을 새로 만드는 방식으로 정리했다.
- 이미 공유된 커밋은 이력을 바꾸면 문제가 생길 수 있으므로, 협업 상황에서는 `revert`를 우선 고려하기로 했다.

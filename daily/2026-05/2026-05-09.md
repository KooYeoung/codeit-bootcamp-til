## 오늘 배운 것

### Readable Code

- 읽기 좋은 코드는 단순히 동작하는 코드가 아니라, 다른 사람이 이해하고 유지보수하기 쉬운 코드라고 이해했다.
- 코드를 잘 작성한다는 것은 결국 코드를 읽는 사람의 인지 부담을 줄이는 것과 연결된다고 느꼈다.
- 추상화는 중요한 정보는 남기고 덜 중요한 정보는 숨기는 과정이다.
- 추상화 레벨이 맞지 않으면 코드를 읽는 사람이 불필요한 세부사항까지 기억해야 하므로 이해하기 어려워진다.
- Early return을 사용하면 중첩된 조건문을 줄이고 코드의 흐름을 더 단순하게 만들 수 있다.
- 중첩 분기문과 중첩 반복문은 사고의 depth를 깊게 만들기 때문에 필요 이상으로 복잡해지지 않도록 주의해야 한다.
- 주석은 코드를 설명하기 위해 남발하기보다, 코드만으로 표현하기 어려운 의사결정의 이유나 히스토리를 남길 때 사용하는 것이 좋다.

### 객체지향 관점

- 객체는 단순히 데이터만 가진 구조가 아니라, 데이터와 기능을 함께 가진 책임 단위라고 이해했다.
- 객체는 외부 세계와 공개 메서드를 통해 소통하고, 내부 구현은 숨기는 것이 좋다.
- 객체의 책임이 명확해야 협력이 자연스러워지고 유지보수하기 쉬운 구조가 된다.
- setter를 무분별하게 사용하기보다, `update~`처럼 의도가 드러나는 메서드를 사용하는 것이 좋다.
- 객체에 값을 꺼내서 처리하기보다 객체에게 메시지를 보내는 방향으로 생각해야 한다.
- 상속은 부모와 자식의 결합도가 높아질 수 있으므로, 가능한 경우 조합과 인터페이스를 활용하는 것이 더 유연할 수 있다.
- Value Object는 값 자체를 표현하는 객체이므로 불변성, 동등성, 유효성 검증이 중요하다.

## 어려웠던 점

- 읽기 좋은 코드가 단순히 짧은 코드나 중복이 없는 코드만을 의미하는 것은 아니라는 점을 정리하는 데 시간이 필요했다.
- 추상화 레벨을 맞춘다는 개념이 처음에는 추상적으로 느껴졌다.
- 객체에게 책임을 부여하는 방식과 단순히 필드를 꺼내서 처리하는 방식의 차이를 코드에 적용하는 것이 쉽지 않았다.

## 해결 방법

- 코드를 작성할 때 “나중에 읽는 사람이 이 흐름을 쉽게 이해할 수 있는가”를 기준으로 생각했다.
- 조건문이 길어지는 경우 Early return을 사용해 흐름을 단순하게 만들 수 있는지 확인했다.
- 메서드명은 구현 방식보다 의도를 드러내는 이름을 사용하려고 했다.
- 객체 내부에서 처리할 수 있는 로직은 외부에서 값을 꺼내 처리하기보다 객체의 메서드로 표현하려고 했다.

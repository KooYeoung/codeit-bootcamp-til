## 오늘 배운 것

### Querydsl이 필요한 이유

- JPQL은 문자열로 작성하기 때문에 오타나 필드명 변경 문제를 실행 시점에 발견할 수 있다.
- Querydsl은 자바 코드로 쿼리를 작성하므로 컴파일 시점에 오류를 더 빨리 발견할 수 있다.
- Querydsl은 Q-Type을 기반으로 엔티티 필드에 타입 안전하게 접근한다.
- 파라미터 바인딩도 Querydsl이 자동으로 처리해주므로 JPQL 문자열을 직접 조립하는 위험을 줄일 수 있다.
- 특히 동적 쿼리에서 조건이 있을 때만 where 절에 추가하는 구조를 깔끔하게 만들 수 있다.

### 프로젝트 환경설정과 도메인 모델

- Spring Boot 3.x에서는 Java 17 이상을 사용해야 한다.
- JPA 관련 패키지는 기존 `javax.persistence`가 아니라 `jakarta.persistence`를 사용해야 한다.
- Querydsl 5.0에서는 `querydsl-jpa:5.0.0:jakarta`와 `querydsl-apt`의 jakarta 설정을 맞춰야 한다.
- 생성된 Q-Type은 빌드 결과물이므로 일반적으로 Git에 포함하지 않고, `clean` 시 삭제되도록 관리하는 것이 좋다.
- 예제 도메인은 `Member`와 `Team`으로 구성되며, `Member.team`이 외래 키를 관리하는 연관관계의 주인이다.
- `@ToString`에는 연관관계 필드를 제외해야 무한 순환이나 불필요한 지연 로딩 문제를 줄일 수 있다.

### Querydsl 기본 문법

- `JPAQueryFactory`는 `EntityManager`를 기반으로 Querydsl 쿼리를 생성한다.
- Q-Type은 `new QMember("m")`처럼 직접 별칭을 지정하거나, `QMember.member` 기본 인스턴스를 static import해서 사용할 수 있다.
- 같은 테이블을 여러 번 조인하는 경우가 아니라면 기본 인스턴스를 사용하는 방식이 간단하다.
- `selectFrom(member)`는 `select(member).from(member)`를 합친 형태로 사용할 수 있다.
- `where()`에 조건을 여러 개 파라미터로 넘기면 AND 조건으로 연결된다.
- `where()`에 전달된 조건이 `null`이면 무시되므로 동적 쿼리 작성에 유리하다.
- `fetch()`는 리스트, `fetchOne()`은 단건, `fetchFirst()`는 첫 번째 한 건 조회에 사용한다.
- Querydsl 5.0에서는 `fetchResults()`, `fetchCount()` 사용을 피하고 count 쿼리를 별도로 작성하는 것이 좋다.

### 중급 문법과 동적 쿼리

- 프로젝션은 select 대상을 지정하는 것이다.
- 조회 대상이 하나면 해당 타입으로 바로 받을 수 있고, 둘 이상이면 `Tuple` 또는 DTO로 조회한다.
- DTO 조회는 `Projections.bean`, `Projections.fields`, `Projections.constructor`, `@QueryProjection` 방식이 있다.
- `@QueryProjection`은 컴파일 시점 타입 체크가 가능하지만 DTO가 Querydsl에 의존하고 DTO용 Q-Type이 생성된다는 단점이 있다.
- 동적 쿼리는 `BooleanBuilder`로 만들 수 있지만, 조건 메서드를 분리한 where 다중 파라미터 방식이 더 읽기 좋고 재사용하기 쉽다.
- 벌크 연산은 영속성 컨텍스트를 거치지 않고 DB에 직접 반영되므로 실행 후 영속성 컨텍스트 초기화가 필요할 수 있다.

### 실무 활용

- 순수 JPA 리포지토리에서도 `JPAQueryFactory`를 주입받아 Querydsl을 사용할 수 있다.
- 스프링이 주입하는 `EntityManager`는 프록시이므로, 트랜잭션 단위로 실제 엔티티 매니저를 찾아 사용한다.
- Spring Data JPA와 Querydsl을 함께 사용할 때는 사용자 정의 리포지토리를 활용한다.
- 구조는 사용자 정의 인터페이스 작성, 구현체 작성, Spring Data JPA 리포지토리에 사용자 정의 인터페이스 상속 순서로 구성한다.
- Querydsl 페이징은 `offset`, `limit`를 사용하고, `PageableExecutionUtils.getPage()`로 count 쿼리 최적화를 할 수 있다.
- count 쿼리가 필요한 경우 select 쿼리와 별도로 명시적으로 작성하는 것이 Querydsl 5.0 흐름에 맞다.
- 단순 조회는 Spring Data JPA 메서드 쿼리로 처리하고, 복잡한 동적 검색은 Querydsl로 분리하는 것이 실무적으로 좋다.

### Spring Data JPA가 제공하는 Querydsl 기능

- `QuerydslPredicateExecutor`는 간단한 Predicate 기반 조회를 제공하지만 left join이 어렵고 서비스나 컨트롤러가 Querydsl에 의존할 수 있다.
- Querydsl Web 지원도 단순 조건에는 편리하지만 복잡한 실무 요구사항에는 한계가 있다.
- `QuerydslRepositorySupport`는 페이징 변환을 도와주지만 Querydsl 3.x 중심으로 설계되어 최신 `JPAQueryFactory` 사용 흐름에는 아쉬움이 있다.
- 복잡한 실무 환경에서는 직접 Querydsl 지원 클래스를 만들어 `select`, `selectFrom`, 페이징, count 쿼리 분리, Sort 변환 등을 명확하게 관리하는 것이 좋다.

## 어려웠던 점

- JPQL과 Querydsl 모두 결국 JPQL/SQL로 실행된다는 점은 같지만, 작성 방식과 오류 발견 시점이 다르다는 점을 구분해야 했다.
- Q-Type이 자동 생성되는 구조라서 빌드 설정과 생성 경로 관리가 처음에는 헷갈렸다.
- `fetch()`, `fetchOne()`, `fetchFirst()`의 차이와 Querydsl 5.0에서 `fetchResults()`, `fetchCount()`를 피해야 하는 이유를 구분해야 했다.
- DTO 조회 방식이 여러 가지라서 각각의 장점과 단점을 정리할 필요가 있었다.
- `BooleanBuilder` 방식과 where 다중 파라미터 방식은 둘 다 동적 쿼리를 만들 수 있지만, 실무에서는 재사용성과 가독성 차이가 중요했다.
- Spring Data JPA와 Querydsl을 함께 사용할 때 사용자 정의 리포지토리 구조를 정확히 이해해야 했다.
- count 쿼리 최적화는 단순히 페이징을 적용하는 것과 별개로 성능을 고려해야 하는 부분이라 더 복습이 필요했다.

## 해결 방법

- Querydsl은 “JPQL을 자바 코드로 안전하게 작성하는 빌더”라고 정리했다.
- Q-Type은 엔티티 구조를 기반으로 생성된 타입 안전한 쿼리 전용 클래스라고 이해했다.
- 동적 쿼리는 조건 메서드를 `BooleanExpression`으로 분리하고, `where()`에 여러 파라미터로 전달하는 방식을 우선적으로 사용한다고 정리했다.
- DTO 조회는 단순한 경우 `Projections.constructor`, 타입 안정성이 중요하면 `@QueryProjection`, 계층 의존성을 줄이고 싶으면 `Projections` 계열을 선택한다고 구분했다.
- 페이징은 content 조회 쿼리와 count 조회 쿼리를 분리해서 생각했다.
- Spring Data JPA의 기본 조회 기능으로 부족한 복잡한 검색은 사용자 정의 리포지토리와 Querydsl로 확장한다고 이해했다.
- Spring Data가 제공하는 Querydsl 편의 기능은 제약이 있으므로 실무에서는 직접 구현한 Querydsl 지원 구조나 QueryRepository를 우선 고려한다고 정리했다.

## 오늘 배운 것

### 자바 웹 기술의 변천

- 초기 Servlet 방식은 Java 클래스에서 HTTP 요청/응답을 직접 처리하고 HTML 응답까지 직접 작성하는 구조였다.
- Servlet 방식은 요청과 응답을 세밀하게 제어할 수 있지만, 비즈니스 로직과 화면 코드가 한 클래스 안에 섞이기 쉬웠다.
- JSP 기반 Model 1 방식은 HTML 안에 Java 코드를 넣어 화면 작성은 편해졌지만, 여전히 화면과 비즈니스 로직이 한 파일에 섞이는 문제가 있었다.
- Model 2 방식은 Servlet이 Controller 역할을 맡고 JSP가 View 역할을 맡아 화면과 요청 처리 책임을 분리했다.
- Spring MVC는 Model 2 구조를 발전시켜 Controller, Service, Repository, View의 역할을 더 명확하게 분리한다.

### 프론트 컨트롤러 패턴과 Spring MVC

- Model 2 구조에서는 여러 Servlet Controller가 각자 공통 기능을 처리하면서 중복 코드가 발생할 수 있다.
- 프론트 컨트롤러 패턴은 모든 요청을 하나의 진입점으로 모아 공통 기능을 중앙에서 처리하는 구조이다.
- 인증, 권한, 로깅, 예외 처리 같은 공통 기능을 프론트 컨트롤러에서 처리하면 중복을 줄일 수 있다.
- Spring MVC에서 프론트 컨트롤러 역할을 하는 핵심 객체가 DispatcherServlet이다.
- DispatcherServlet은 요청을 받아 적절한 Handler를 찾고, HandlerAdapter를 통해 호출한 뒤, ViewResolver를 통해 View를 찾아 응답을 만든다.

### Servlet, Request, Response

- Servlet은 클라이언트 요청을 처리하는 서버 측 Java 프로그램이며 Servlet Container 안에서 실행된다.
- Servlet은 보통 최초 요청 또는 애플리케이션 시작 시점에 생성되고, 요청마다 `service()`가 호출된다.
- `HttpServletRequest`는 HTTP 메서드, URI, 헤더, 파라미터, 세션, 쿠키, 요청 본문 같은 요청 정보를 제공한다.
- `HttpServletResponse`는 상태 코드, 응답 헤더, Content-Type, 응답 본문, redirect, cookie 같은 응답 정보를 설정한다.
- Form 데이터와 URL 쿼리 파라미터는 `getParameter()` 계열 API로 읽을 수 있다.
- JSON이나 Plain Text처럼 요청 본문에 직접 담긴 데이터는 `getReader()` 또는 `getInputStream()`으로 읽어야 한다.

### Spring MVC 초기화와 기본 구조

- 전통적인 Spring MVC에서는 `WebApplicationInitializer`, `ContextLoaderListener`, DispatcherServlet 등록을 직접 이해해야 한다.
- Spring Boot는 내장 WAS와 자동 설정을 통해 DispatcherServlet, WebMvc 관련 구성 요소를 자동으로 등록한다.
- `DispatcherServletAutoConfiguration`은 DispatcherServlet 빈과 서블릿 등록을 자동 구성한다.
- `WebMvcAutoConfiguration`은 `@EnableWebMvc`를 직접 사용하지 않을 때 Spring Boot의 기본 MVC 설정을 제공한다.
- `WebMvcConfigurer`를 사용하면 Spring Boot의 기본 자동 설정을 유지하면서 필요한 MVC 설정만 추가할 수 있다.
- Spring MVC의 기본 요청 흐름은 `DispatcherServlet → HandlerMapping → HandlerAdapter → Controller → ViewResolver → View` 순서로 이해할 수 있다.

## 어려웠던 점

- Servlet, JSP, Model 2, Front Controller, Spring MVC가 모두 웹 요청을 처리하는 구조라서 처음에는 차이가 비슷하게 느껴졌다.
- `HttpServletRequest`와 `HttpServletResponse`가 실제로는 WAS 내부 구현 객체와 Facade 객체로 감싸져 전달된다는 구조가 낯설었다.
- Spring Boot가 많은 MVC 초기화 작업을 자동으로 해주기 때문에, 내부적으로 어떤 클래스들이 동작하는지 보이지 않아 이해하기 어려웠다.
- DispatcherServlet이 단순히 Controller를 호출하는 것이 아니라 HandlerMapping, HandlerAdapter, ViewResolver 등 여러 전략 객체와 협력한다는 점이 중요했다.

## 해결 방법

- Servlet은 “Java에서 직접 HTTP와 HTML을 다루는 초기 방식”, JSP는 “HTML 안에 Java를 넣은 방식”, Model 2는 “Controller와 View를 분리한 방식”으로 구분했다.
- Front Controller는 여러 Controller 앞에 있는 공통 진입점으로 이해했고, Spring MVC에서는 DispatcherServlet이 그 역할을 담당한다고 정리했다.
- Request는 클라이언트가 보낸 정보를 읽는 객체, Response는 서버가 클라이언트에게 보낼 정보를 설정하는 객체로 나누어 이해했다.
- Spring Boot 자동 설정은 편리하지만, 내부적으로는 Servlet Container와 Spring Container가 연결되고 DispatcherServlet이 등록된다는 흐름을 기준으로 정리했다.
- Spring MVC 기본 구조는 “요청을 누가 찾고, 누가 호출하고, 누가 화면을 찾는가”라는 관점으로 HandlerMapping, HandlerAdapter, ViewResolver를 구분했다.

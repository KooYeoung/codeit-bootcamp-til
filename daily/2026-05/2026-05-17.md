## 오늘 배운 것

### JDBC

- JDBC는 Java 애플리케이션에서 데이터베이스에 접근하기 위한 표준 인터페이스이다.
- 데이터베이스마다 연결 방법, SQL 전달 방법, 결과 응답 방식이 다르면 애플리케이션 코드가 DB 기술에 강하게 묶인다.
- JDBC는 이런 차이를 표준화해서 개발자가 같은 방식으로 DB에 접근할 수 있게 해준다.
- JDBC를 직접 사용할 때는 `Connection`, `PreparedStatement`, `ResultSet`을 다룬다.
- `PreparedStatement`는 SQL과 파라미터를 분리해서 전달할 수 있어 SQL 실행을 더 안전하게 처리할 수 있다.
- JDBC 코드는 동작 원리를 이해하는 데 좋지만, 커넥션 획득, SQL 실행, 결과 매핑, 리소스 정리 코드가 반복된다.

### 커넥션 풀과 DataSource

- 데이터베이스 커넥션을 매번 새로 생성하면 TCP/IP 연결, 인증, DB 세션 생성 등의 과정이 필요해서 비용이 크다.
- 커넥션 풀은 애플리케이션 시작 시점에 커넥션을 미리 만들어두고, 필요할 때 빌려주고 다시 반환받는 방식이다.
- 커넥션 풀을 사용하면 요청마다 새 커넥션을 만드는 시간을 줄일 수 있다.
- `DataSource`는 커넥션을 얻는 방법을 추상화한 인터페이스이다.
- `DriverManagerDataSource`를 사용하다가 `HikariDataSource` 같은 커넥션 풀로 바꿔도, 애플리케이션 코드는 `DataSource`에만 의존하면 변경을 줄일 수 있다.
- 설정은 `DataSource` 생성 시점에 하고, 사용하는 쪽은 `dataSource.getConnection()`만 호출하면 된다.

### 트랜잭션

- 트랜잭션은 하나의 거래를 안전하게 처리하도록 보장하는 기능이다.
- 계좌이체처럼 여러 SQL이 하나의 작업으로 묶여야 하는 경우, 모두 성공하면 커밋하고 중간에 실패하면 롤백해야 한다.
- 트랜잭션의 ACID는 원자성, 일관성, 격리성, 지속성을 의미한다.
- 자동 커밋 상태에서는 SQL 하나마다 바로 반영되므로, 여러 SQL을 하나의 작업으로 묶기 어렵다.
- 수동 커밋 모드로 변경하면 개발자가 직접 커밋과 롤백 시점을 제어할 수 있다.
- 커밋 전 변경 데이터는 해당 트랜잭션을 시작한 세션에서만 보이고, 다른 세션에는 보이지 않는다.
- 동시에 같은 데이터를 수정할 때 데이터 정합성을 지키기 위해 DB 락이 필요하다.

### 스프링 트랜잭션 문제 해결

- 서비스 계층은 핵심 비즈니스 로직을 담당하므로 특정 기술에 직접 의존하지 않는 것이 좋다.
- 순수 JDBC 트랜잭션을 서비스 계층에서 직접 처리하면 `Connection`, `DataSource`, `SQLException` 같은 JDBC 기술이 서비스 계층에 섞인다.
- 스프링은 `PlatformTransactionManager`를 통해 트랜잭션 기술을 추상화한다.
- JDBC를 사용할 때는 `DataSourceTransactionManager`, JPA를 사용할 때는 `JpaTransactionManager`를 사용할 수 있다.
- 트랜잭션 동기화 매니저는 트랜잭션에서 사용하는 커넥션을 쓰레드 로컬에 보관해 같은 트랜잭션 안에서 같은 커넥션을 사용하게 도와준다.
- `@Transactional`을 사용하면 트랜잭션 시작, 커밋, 롤백 같은 반복 코드를 AOP로 분리할 수 있다.
- 트랜잭션을 직접 코드로 작성하는 것보다 선언적으로 적용하는 방식이 서비스 계층을 더 깔끔하게 유지할 수 있다.

### 예외와 JdbcTemplate

- `Exception`과 그 하위 예외는 기본적으로 체크 예외이며, `RuntimeException`과 그 하위 예외는 언체크 예외이다.
- 체크 예외는 잡거나 던지도록 강제되기 때문에 실수 방지에는 도움이 되지만, 처리할 수 없는 예외까지 서비스 계층에 의존시키는 문제가 생길 수 있다.
- 기본적으로는 언체크 예외를 사용하고, 비즈니스 로직상 반드시 처리해야 하는 예외만 체크 예외로 고려하는 것이 좋다.
- `SQLException` 같은 JDBC 예외가 서비스 계층까지 전파되면 서비스 계층이 JDBC 기술에 의존하게 된다.
- 스프링은 데이터 접근 예외를 추상화해서 JDBC, JPA 등 기술별 예외를 공통된 예외 체계로 다룰 수 있게 해준다.
- `JdbcTemplate`은 JDBC의 반복 코드를 줄여주고, 트랜잭션 동기화와 예외 변환도 함께 지원한다.

## 어려웠던 점

- JDBC를 직접 사용할 때 커넥션 획득, SQL 준비, 실행, 결과 매핑, 리소스 정리 흐름이 길어서 전체 흐름을 한 번에 보기 어려웠다.
- 커넥션 풀과 DataSource가 비슷하게 느껴졌지만, 커넥션 풀은 커넥션을 미리 관리하는 기술이고 DataSource는 커넥션 획득 방법을 추상화한 인터페이스라는 점을 구분해야 했다.
- 트랜잭션에서 같은 커넥션을 유지해야 하는 이유가 처음에는 어렵게 느껴졌다.
- 서비스 계층을 순수하게 유지해야 한다는 말은 이해했지만, 트랜잭션을 적용하면서 JDBC 기술이 서비스 계층에 섞이는 문제를 코드 관점에서 연결하는 데 시간이 필요했다.
- 체크 예외가 안전해 보이지만, 특정 기술에 대한 의존을 전파할 수 있다는 점이 중요했다.
- `@Transactional`이 내부적으로 AOP를 통해 트랜잭션을 적용한 다는 흐름을 더 복습해야 한다고 느꼈다.

## 해결 방법

- JDBC 흐름은 `Connection 획득 → PreparedStatement 생성 → 파라미터 바인딩 → SQL 실행 → ResultSet 처리 → 리소스 정리` 순서로 정리했다.
- 커넥션 풀은 “미리 만들어둔 커넥션을 빌려 쓰는 방식”, DataSource는 “커넥션을 얻는 방법을 감춘 표준 인터페이스”라고 구분했다.
- 트랜잭션은 계좌이체 예제로 연결해서 “여러 SQL을 하나의 작업처럼 성공 또는 실패시켜야 하는 경우”라고 이해했다.
- 스프링 트랜잭션 추상화는 “서비스 계층이 JDBC/JPA 같은 구현 기술을 직접 알지 않게 만드는 장치”로 정리했다.
- 예외는 “복구 가능하고 반드시 처리해야 하는 예외인지”를 기준으로 체크 예외와 언체크 예외 사용을 구분해야 한다고 이해했다.
- JdbcTemplate은 JDBC 원리를 감추는 것이 아니라, 반복되는 JDBC 사용 코드를 대신 처리해주는 도구로 정리했다.

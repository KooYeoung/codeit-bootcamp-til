## 오늘 배운 것

### SOLID 원칙

- SRP는 하나의 클래스가 하나의 책임을 가지도록 설계해야 한다는 원칙이라고 이해했다.
- 책임이 여러 개로 섞이면 수정 이유도 여러 개가 되어 코드 변경 범위가 커질 수 있다.
- OCP는 기능 확장에는 열려 있고, 기존 코드 수정에는 닫혀 있어야 한다는 원칙이다.
- 인터페이스와 다형성을 활용하면 기존 코드를 크게 바꾸지 않고 새로운 구현을 추가할 수 있다.
- LSP는 부모 타입으로 하던 일을 자식 타입으로 바꿔도 동작 흐름이 깨지지 않아야 한다는 원칙이라고 이해했다.
- ISP는 하나의 큰 인터페이스에 필요 없는 메서드까지 강제로 구현하게 만들기보다, 역할별로 인터페이스를 분리하는 것이 좋다는 원칙이다.
- DIP는 구체 구현체에 직접 의존하지 않고 역할을 나타내는 인터페이스에 의존하도록 설계하는 방향이라고 정리했다.

### enum과 예외 처리

- `enum`은 정해진 값의 집합을 타입으로 표현할 때 사용하며, 문자열이나 숫자 상수보다 타입 안정성과 가독성이 좋다.
- 값의 범위가 제한되어 있을 때 `enum`을 사용하면 잘못된 값이 들어오는 문제를 컴파일 단계에서 줄일 수 있다.
- 예외 처리는 프로그램 오류를 없애는 것이 아니라, 문제가 발생할 가능성이 있는 지점에서 어떻게 대응할지 미리 정해두는 방식이다.
- `try-catch`는 예외를 직접 처리하는 구조이고, `throws`는 예외 처리 책임을 호출한 쪽으로 넘기는 구조라고 이해했다.
- `finally`는 예외 발생 여부와 상관없이 정리 작업이 필요할 때 사용한다.

### Java 스레드

- 스레드는 하나의 프로세스 안에서 실행되는 작업 흐름이며, 여러 작업을 동시에 처리하는 것처럼 보이게 만들 수 있다.
- 스레드를 직접 계속 생성하면 생성 비용과 관리 비용이 커질 수 있으므로, 반복적인 작업 처리에는 스레드 풀과 Executor 프레임워크를 고려해야 한다.
- 여러 스레드가 같은 자원에 접근하면 동시성 문제가 생길 수 있고, 이때 `synchronized`, `Lock`, 동시성 컬렉션 같은 도구가 필요하다.
- `volatile`은 메모리 가시성 문제를 다룰 때 사용할 수 있지만, 복합 연산의 원자성을 보장하지는 않는다고 이해했다.
- 생산자-소비자 문제는 공유 버퍼를 사이에 두고 데이터를 넣는 쪽과 꺼내는 쪽의 동기화가 필요한 대표적인 예시다.

## 어려웠던 점

- SOLID 원칙은 문장으로 이해하는 것보다 실제 코드에서 어떤 문제가 생기는지 연결하는 것이 더 어렵게 느껴졌다.
- 예외를 직접 처리해야 할지, 호출한 쪽으로 넘겨야 할지 판단하는 기준이 아직 명확하지 않았다.
- 스레드 관련 개념은 생명 주기, 메모리 가시성, 동기화, 원자성처럼 서로 연결된 내용이 많아서 한 번에 정리하기 쉽지 않았다.

## 해결 방법

- 각 개념을 문법 자체보다 “왜 필요한가”를 기준으로 다시 정리했다.
- 예외 처리는 복구 가능한 위치에서 처리하고, 의미 있는 처리가 어렵다면 호출한 쪽으로 책임을 넘기는 방향으로 이해했다.
- 스레드는 먼저 생성과 실행 흐름을 잡고, 이후 공유 자원 문제가 생기는 지점에서 동기화 개념을 연결해 보기로 했다.

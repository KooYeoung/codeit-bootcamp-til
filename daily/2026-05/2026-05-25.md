## 오늘 배운 것

### HandlerMapping과 @RequestMapping

- HandlerMapping은 요청 조건에 맞는 Handler를 찾는 역할을 한다.
- 애노테이션 기반 컨트롤러에서는 보통 `RequestMappingHandlerMapping`이 `@RequestMapping` 정보를 분석해 HandlerMethod를 찾는다.
- `@RequestMapping`은 URL뿐 아니라 HTTP method, params, headers, consumes, produces 같은 조건으로 요청을 더 세밀하게 매핑할 수 있다.
- `@GetMapping`, `@PostMapping`은 `@RequestMapping(method = ...)`을 더 편하게 사용하기 위한 축약 애노테이션이다.
- Content-Type은 요청 본문의 형식을 나타내고, Accept는 클라이언트가 원하는 응답 형식을 나타낸다.

### HandlerAdapter와 메서드 파라미터 처리

- HandlerAdapter는 DispatcherServlet이 찾은 Handler를 실제로 호출하는 역할을 한다.
- 애노테이션 기반 컨트롤러에서는 `RequestMappingHandlerAdapter`가 핵심이다.
- 컨트롤러 메서드의 다양한 파라미터는 HandlerMethodArgumentResolver가 만들어준다.
- `@RequestParam`은 쿼리 파라미터나 Form 데이터를 받을 때 사용한다.
- `@PathVariable`은 URL 경로 일부를 변수로 받을 때 사용한다.
- `@ModelAttribute`는 요청 파라미터를 객체에 바인딩할 때 사용한다.
- `@RequestBody`는 HTTP 요청 본문을 객체로 변환할 때 사용한다.
- JSON 요청은 HttpMessageConverter가 Jackson 등을 사용해 객체로 변환한다.

### 타입 변환과 바인딩

- HTTP 요청 값은 대부분 문자열로 들어오기 때문에 필요한 타입으로 변환하는 과정이 필요하다.
- Converter는 한 타입을 다른 타입으로 변환하는 단일 변환 로직이다.
- ConversionService는 여러 Converter를 등록하고 애플리케이션 전반에서 타입 변환을 제공하는 서비스이다.
- Formatter는 문자 기반의 입력/출력 포맷 변환에 특화되어 있으며 Locale 정보를 활용할 수 있다.
- DataBinder는 요청 파라미터를 객체 필드에 바인딩하고, 타입 변환과 검증까지 연결하는 역할을 한다.
- `@InitBinder`를 사용하면 특정 컨트롤러 또는 `@ControllerAdvice` 범위에서 바인딩 설정을 커스터마이징할 수 있다.

### 검증과 반환값 처리

- `BindingResult`는 바인딩 오류와 검증 오류를 담는 객체이다.
- `BindingResult`는 검증 대상 객체 바로 뒤에 위치해야 한다.
- 필드 오류는 `rejectValue()`, 객체 전체 오류는 `reject()`로 처리할 수 있다.
- `Validator`를 구현하면 검증 로직을 컨트롤러에서 분리할 수 있다.
- Bean Validation은 `@NotBlank`, `@NotNull`, `@Min`, `@Max`, `@Pattern` 같은 애노테이션으로 검증 규칙을 선언한다.
- `@Valid` 또는 `@Validated`를 사용하면 스프링 MVC가 자동으로 검증을 수행한다.
- `@ResponseBody`는 반환값을 View 이름으로 해석하지 않고 HTTP 응답 본문에 직접 쓴다.
- `ResponseEntity`는 응답 본문, 상태 코드, 헤더를 함께 제어할 수 있다.

## 어려웠던 점

- `@RequestParam`, `@ModelAttribute`, `@RequestBody`가 모두 요청 데이터를 받는 기능이라 처음에는 사용 기준이 헷갈렸다.
- HandlerMapping은 Handler를 찾고 HandlerAdapter는 Handler를 호출한다는 역할 구분이 중요했다.
- HttpMessageConverter가 요청과 응답 양방향에서 동작한다는 점을 이해하는 데 시간이 필요했다.
- Converter, ConversionService, Formatter, DataBinder가 모두 타입 변환과 관련되어 있어 역할을 나누어 정리해야 했다.
- 검증에서 바인딩 오류와 비즈니스 검증 오류가 모두 `BindingResult`에 담길 수 있다는 점이 중요했다.

## 해결 방법

- 요청 데이터가 URL 쿼리/Form이면 `@RequestParam`, 객체 바인딩이면 `@ModelAttribute`, JSON 본문이면 `@RequestBody`로 구분했다.
- HandlerMapping은 “누구를 실행할지 찾는 역할”, HandlerAdapter는 “찾은 대상을 실행하는 역할”로 정리했다.
- MessageConverter는 HTTP Body와 Java 객체 사이를 변환하는 구성요소로 이해했다.
- Converter는 단일 타입 변환, ConversionService는 변환기 관리, Formatter는 문자 포맷 변환, DataBinder는 바인딩 전체 과정으로 구분했다.
- 검증은 `BindingResult → Validator 분리 → Bean Validation` 흐름으로 점진적으로 발전한다고 정리했다.

## 오늘 배운 것

### 프로젝트 환경설정과 도메인 모델

- Spring Boot 3.x에서는 Java 17 이상이 필요하고, 기존 `javax` 패키지 대신 `jakarta` 패키지를 사용해야 한다.
- H2 데이터베이스는 Spring Boot 3.x 기준으로 2.1.214 이상 버전을 사용하는 것이 좋다.
- `Member`와 `Team`은 다대일/일대다 양방향 관계로 구성되며, 외래 키가 있는 `Member.team`이 연관관계의 주인이다.
- `Team.members`는 `mappedBy = "team"`으로 매핑하고, 외래 키를 직접 변경하지 않는다.
- 양방향 연관관계에서는 `changeTeam()` 같은 연관관계 편의 메서드로 양쪽 참조를 함께 맞춰주는 것이 좋다.
- `@ToString`은 연관관계 필드를 포함하면 무한 순환이나 지연 로딩 문제가 생길 수 있으므로 내부 필드 위주로 제한하는 것이 안전하다.

### 공통 인터페이스 기능

- 순수 JPA 리포지토리에서는 `EntityManager`를 직접 사용해서 CRUD 메서드를 구현해야 한다.
- Spring Data JPA는 `JpaRepository<Entity, ID>` 인터페이스만 상속하면 공통 CRUD 구현체를 자동으로 제공한다.
- Spring Data JPA가 생성한 리포지토리는 프록시 객체로 동작하며, 구현 클래스를 직접 작성하지 않아도 된다.
- `save()`는 새로운 엔티티는 저장하고, 이미 존재하는 엔티티는 병합한다.
- `findById()`는 `Optional`을 반환해 조회 결과가 없을 수 있음을 표현한다.
- 최신 버전에서는 프록시 조회 용도로 `getOne()` 대신 `getReferenceById()`를 사용하는 흐름으로 이해하는 것이 좋다.

### 쿼리 메소드 기능

- Spring Data JPA는 메서드 이름을 분석해서 JPQL을 생성할 수 있다.
- `findByUsernameAndAgeGreaterThan`처럼 조건이 적을 때는 메서드 이름 기반 쿼리가 간단하고 편리하다.
- 조건이 많아지면 메서드 이름이 길어지고 가독성이 떨어지므로 `@Query`를 사용하는 것이 더 적절할 수 있다.
- `@Query`는 리포지토리 메서드에 JPQL을 직접 작성하는 방식이며, 애플리케이션 로딩 시점에 문법 오류를 확인할 수 있다.
- DTO 조회가 필요하면 JPQL의 `new` 명령어를 사용해 생성자 기반으로 바로 조회할 수 있다.
- 파라미터 바인딩은 위치 기반보다 이름 기반 `@Param`을 사용하는 것이 유지보수에 좋다.
- 컬렉션 파라미터는 `in :names`처럼 사용할 수 있다.

### 페이징, 벌크 수정, EntityGraph

- Spring Data JPA는 `Pageable`과 `Sort`를 통해 페이징과 정렬을 지원한다.
- `Page`는 전체 count 쿼리를 함께 실행해 전체 페이지 수와 전체 데이터 수를 제공한다.
- `Slice`는 count 쿼리 없이 다음 페이지 존재 여부만 확인하므로 모바일 무한 스크롤 같은 기능에 적합할 수 있다.
- `List`는 count 쿼리 없이 현재 조회 결과만 반환한다.
- 전체 count 쿼리가 무거울 수 있으므로 복잡한 쿼리에서는 count 쿼리를 분리하는 것이 중요하다.
- 벌크성 수정 쿼리는 영속성 컨텍스트를 거치지 않고 DB에 직접 반영되므로 실행 후 영속성 컨텍스트를 비우거나 동기화해야 한다.
- `@EntityGraph`는 연관 엔티티를 함께 조회해 N+1 문제를 줄이는 데 사용할 수 있다.
- JPA Hint는 조회 전용 최적화 등에 사용할 수 있고, Lock은 동시성 제어가 필요한 상황에서 사용할 수 있다.

### 확장 기능과 내부 분석

- 사용자 정의 리포지토리는 Spring Data JPA가 제공하지 않는 복잡한 조회나 Querydsl, JdbcTemplate 연동이 필요할 때 사용한다.
- 최신 방식으로는 사용자 정의 인터페이스 이름에 `Impl`을 붙이는 구현체 이름이 더 직관적이다.
- Auditing을 사용하면 등록일, 수정일, 등록자, 수정자를 자동으로 관리할 수 있다.
- 등록일과 수정일은 대부분의 엔티티에 필요하므로 `BaseTimeEntity`로 분리하고, 등록자/수정자가 필요한 경우 `BaseEntity`로 확장하는 방식이 실무적으로 좋다.
- 도메인 클래스 컨버터는 URL의 ID 값을 엔티티로 변환해주지만, 단순 조회 용도로만 사용하는 것이 좋다.
- Web 페이징 확장은 컨트롤러에서 `Pageable`을 바로 받을 수 있게 해준다.
- `SimpleJpaRepository`는 기본적으로 조회 메서드에 `@Transactional(readOnly = true)`를 적용하고, 변경 메서드에는 별도 트랜잭션을 적용한다.
- `save()`는 `isNew()` 판단 결과에 따라 `persist` 또는 `merge`를 호출한다.
- 직접 ID를 할당하는 엔티티는 `save()` 시점에 이미 ID가 있으므로 기존 엔티티로 오해되어 `merge()`가 호출될 수 있다.
- 이 경우 `Persistable`을 구현해 `isNew()` 판단 기준을 직접 제공하는 것이 좋다.

## 어려웠던 점

- Spring Data JPA가 인터페이스만으로 리포지토리 구현체를 만들어주는 구조가 처음에는 마법처럼 느껴졌다.
- `save()`가 항상 단순 저장이 아니라, 새로운 엔티티면 `persist`, 기존 엔티티면 `merge`로 동작한다는 점이 중요했다.
- 메서드 이름 기반 쿼리는 편리하지만 조건이 많아지면 이름이 너무 길어져 어느 시점에 `@Query`나 Querydsl로 넘어가야 하는지 기준이 필요했다.
- `Page`, `Slice`, `List`의 차이는 단순 반환 타입 차이가 아니라 count 쿼리 실행 여부와 화면 요구사항의 차이와 연결되어 있었다.
- 벌크 수정 쿼리가 영속성 컨텍스트를 무시하고 DB에 직접 실행된다는 점 때문에, 이후 조회 결과가 예상과 달라질 수 있다는 점이 어려웠다.
- Auditing은 설정이 간단해 보이지만, `@EnableJpaAuditing`, `@EntityListeners`, `AuditorAware`의 역할을 구분해야 했다.
- 직접 할당 ID를 사용하는 경우 `Persistable`이 필요한 이유를 `persist`와 `merge` 차이와 연결해서 이해해야 했다.

## 해결 방법

- Spring Data JPA 리포지토리는 “인터페이스를 기반으로 프록시 구현체를 생성한다”고 정리했다.
- 공통 CRUD는 `JpaRepository`가 제공하고, 복잡한 조회나 동적 쿼리는 사용자 정의 리포지토리나 Querydsl로 확장한다고 이해했다.
- 메서드 이름 기반 쿼리는 조건이 단순할 때 사용하고, 조건이 복잡하거나 DTO 조회가 필요하면 `@Query`를 사용한다고 구분했다.
- 페이징은 화면에서 전체 페이지 수가 필요한지 여부를 기준으로 `Page`와 `Slice`를 선택한다고 정리했다.
- 벌크 수정 후에는 `clearAutomatically = true` 또는 `em.clear()` 같은 방식으로 영속성 컨텍스트 불일치를 해결해야 한다고 이해했다.
- Auditing은 공통 필드 관리 기능으로 보고, 시간 정보와 사용자 정보를 분리해서 설계하는 방식으로 정리했다.
- 직접 할당 ID는 `save()`에서 `merge()`가 호출될 수 있으므로 `Persistable`로 새로운 엔티티 판단 기준을 보완한다고 정리했다.

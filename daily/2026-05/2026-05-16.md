## 오늘 배운 것

### 타임리프와 폼 처리

- 타임리프는 서버에서 HTML을 동적으로 렌더링할 수 있게 도와주는 템플릿 엔진이다.
- `th:text`는 HTML 이스케이프를 적용해 안전하게 텍스트를 출력하고, `th:utext`는 이스케이프하지 않은 내용을 출력한다.
- SpringEL을 사용하면 객체의 프로퍼티, 리스트, 맵 등에 접근할 수 있다.
- `@{...}` 문법을 사용하면 URL 링크를 편리하게 생성할 수 있다.
- `th:object`는 폼에서 사용할 커맨드 객체를 지정하고, `*{...}`는 선택 변수 식으로 해당 객체의 필드에 접근한다.
- `th:field`는 `id`, `name`, `value`를 자동으로 처리해주기 때문에 폼 입력 코드를 줄일 수 있다.
- 체크박스, 라디오 버튼, 셀렉트 박스는 타임리프의 폼 지원 기능을 사용하면 선택 상태를 편리하게 유지할 수 있다.

### 메시지와 국제화

- 메시지 기능은 화면에 표시되는 문구를 HTML에 직접 작성하지 않고, 메시지 파일에서 key-value 형태로 관리하는 방식이다.
- 메시지 파일을 사용하면 여러 화면에서 같은 문구를 한 번에 변경할 수 있다.
- 국제화는 언어별 메시지 파일을 분리해서 사용자의 언어 환경에 맞는 문구를 보여주는 방식이다.
- 타임리프에서는 `#{...}` 문법으로 메시지를 조회할 수 있다.
- 스프링은 `MessageSource`를 통해 메시지와 국제화 기능을 제공한다.

### 검증과 Bean Validation

- 서버 검증은 필수이며, 클라이언트 검증은 사용자 편의성을 높이는 보조 수단으로 이해해야 한다.
- 직접 검증 로직을 작성하면 오류를 `Map` 등에 담아 화면으로 전달할 수 있지만, 코드가 복잡해질 수 있다.
- `BindingResult`를 사용하면 스프링이 검증 오류를 컨트롤러에서 처리할 수 있게 도와준다.
- 필드 단위 오류는 `FieldError`, 객체 전체와 관련된 오류는 `ObjectError`로 표현할 수 있다.
- `Validator`를 분리하면 검증 로직을 컨트롤러에서 분리할 수 있다.
- Bean Validation은 `@NotBlank`, `@NotNull`, `@Range`, `@Max` 같은 애노테이션으로 검증 규칙을 선언할 수 있게 해준다.
- 등록과 수정의 검증 요구사항이 다르면 groups를 사용할 수도 있지만, 실무에서는 폼 전송 객체를 분리하는 방식이 더 명확할 수 있다.

### 로그인, 쿠키, 세션

- 쿠키는 클라이언트가 보관하고 요청마다 서버에 함께 전달하는 값이다.
- 쿠키에 회원 ID 같은 중요한 값을 그대로 저장하면 조작 가능성 때문에 보안 문제가 생길 수 있다.
- 세션은 서버가 중요한 값을 보관하고, 클라이언트에는 세션 ID만 전달하는 방식이다.
- `HttpSession`을 사용하면 세션을 직접 구현하지 않아도 로그인 상태를 관리할 수 있다.
- 로그아웃 시에는 세션을 무효화해서 로그인 상태를 제거해야 한다.

### 필터와 인터셉터

- 필터와 인터셉터는 로그인 체크, 요청 로그 같은 웹 공통 관심사를 분리할 때 사용한다.
- 필터는 서블릿 기술이고, 요청이 서블릿에 도달하기 전에 동작한다.
- 인터셉터는 스프링 MVC 기술이고, DispatcherServlet 이후 컨트롤러 호출 전후에 동작한다.
- 로그인 인증처럼 HTTP 요청 정보가 필요한 공통 로직은 AOP보다 필터나 인터셉터가 더 적합할 수 있다.
- ArgumentResolver를 사용하면 로그인 회원 정보를 컨트롤러 파라미터에서 더 편리하게 받을 수 있다.

### 예외 처리, 타입 컨버터, 파일 업로드

- 웹 애플리케이션에서 예외가 처리되지 않으면 WAS까지 전달되고, 오류 페이지가 호출될 수 있다.
- HTML 오류 페이지와 API 오류 응답은 요구사항이 다르므로 API는 JSON 오류 응답 구조를 별도로 설계해야 한다.
- `@ExceptionHandler`와 `@ControllerAdvice`를 사용하면 API 예외 처리를 컨트롤러와 분리해서 관리할 수 있다.
- 스프링 타입 컨버터는 요청 파라미터처럼 문자로 들어온 값을 필요한 타입으로 변환해준다.
- `Converter`, `ConversionService`, `Formatter`를 통해 타입 변환과 출력 포맷을 다룰 수 있다.
- 파일 업로드는 일반 폼 전송 방식이 아니라 `multipart/form-data` 방식이 필요하다.
- 스프링에서는 `MultipartFile`을 사용해 업로드된 파일을 편리하게 다룰 수 있다.

## 어려웠던 점

- 타임리프 기본 문법과 스프링 통합 폼 문법이 한 번에 많아서 각각의 역할을 구분하는 데 시간이 필요했다.
- `BindingResult`가 왜 `@ModelAttribute` 바로 뒤에 와야 하는지, 그리고 타입 오류까지 어떻게 담는지 이해하는 데 시간이 필요했다.
- 직접 검증, Validator 분리, Bean Validation, 폼 객체 분리 방식의 차이를 구분해야 했다.
- 쿠키와 세션은 단순 로그인 기능처럼 보이지만 보안 관점까지 함께 고려해야 해서 중요했다.
- 필터와 인터셉터는 둘 다 공통 관심사를 처리하지만, 동작 위치와 사용 목적이 다르다는 점을 구분해야 했다.
- 오류 페이지 처리와 API 예외 처리는 같은 예외 처리라도 응답 형태가 다르다는 점이 헷갈렸다.
- 타입 컨버터와 포맷터는 자동으로 동작하는 부분이 많아 내부 흐름을 따로 정리할 필요가 있었다.

## 해결 방법

- 타임리프는 “HTML을 동적으로 렌더링하기 위한 문법”과 “스프링 폼을 편리하게 다루는 문법”으로 나누어 정리했다.
- 검증은 직접 검증에서 시작해 `BindingResult`, `Validator`, Bean Validation으로 점진적으로 발전한다고 이해했다.
- 쿠키는 클라이언트 저장소, 세션은 서버 저장소라는 기준으로 구분했다.
- 인증 공통 로직은 컨트롤러마다 작성하지 않고 필터나 인터셉터로 분리하는 것이 유지보수에 좋다고 정리했다.
- HTML 오류 응답은 사용자 화면 중심, API 오류 응답은 클라이언트가 해석할 JSON 스펙 중심으로 구분했다.
- 타입 컨버터는 요청 값이 대부분 문자로 들어오기 때문에 필요한 타입으로 바꾸는 중간 장치라고 이해했다.
- 파일 업로드는 문자와 바이너리 데이터를 함께 전송해야 하므로 `multipart/form-data`가 필요하다고 정리했다.

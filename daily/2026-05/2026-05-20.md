## 오늘 배운 것

### JPA가 필요한 이유

- 객체지향 언어에서는 객체와 참조를 중심으로 개발하지만, 관계형 데이터베이스는 테이블과 외래 키를 중심으로 데이터를 관리한다.
- JDBC나 MyBatis 중심 개발에서는 객체를 SQL로 변환하고, SQL 결과를 다시 객체로 변환하는 반복 코드가 많아진다.
- 필드가 추가되면 INSERT, UPDATE, SELECT SQL과 결과 매핑 코드도 함께 수정해야 해서 유지보수 비용이 커진다.
- JPA는 객체와 관계형 데이터베이스 사이의 차이를 줄이기 위한 ORM 기술이다.
- JPA를 사용하면 개발자는 엔티티 객체 중심으로 개발하고, JPA가 필요한 SQL을 생성해 데이터베이스와 동기화한다.

### JPA 시작과 기본 사용 흐름

- JPA를 사용하려면 엔티티 클래스에 `@Entity`를 붙이고 기본 키 필드에 `@Id`를 매핑한다.
- `EntityManagerFactory`는 애플리케이션 전체에서 하나만 생성해 공유한다.
- `EntityManager`는 요청이나 트랜잭션 단위로 생성해서 사용하고, 쓰레드 간 공유하면 안 된다.
- JPA의 데이터 변경 작업은 반드시 트랜잭션 안에서 실행해야 한다.
- 기본 흐름은 `EntityManagerFactory` 생성, `EntityManager` 생성, 트랜잭션 시작, 엔티티 저장/조회/수정/삭제, 커밋 또는 롤백, 자원 정리 순서로 진행된다.

### 영속성 컨텍스트

- 영속성 컨텍스트는 엔티티를 영구 저장하는 환경이다.
- 엔티티 매니저를 통해 영속성 컨텍스트에 접근한다.
- 엔티티 생명주기는 비영속, 영속, 준영속, 삭제 상태로 구분할 수 있다.
- `persist()`를 호출하면 엔티티가 영속 상태가 되고, 영속성 컨텍스트가 해당 엔티티를 관리한다.
- 영속성 컨텍스트는 1차 캐시, 동일성 보장, 트랜잭션을 지원하는 쓰기 지연, 변경 감지, 지연 로딩 같은 기능을 제공한다.
- 플러시는 영속성 컨텍스트의 변경 내용을 데이터베이스에 반영하는 과정이다.
- 플러시가 발생해도 영속성 컨텍스트가 비워지는 것은 아니며, 데이터베이스와 변경 내용을 동기화하는 것이다.

### 엔티티 매핑

- `@Entity`가 붙은 클래스는 JPA가 관리하는 엔티티가 된다.
- 엔티티는 기본 생성자가 필요하고, `final` 클래스나 `final` 필드는 피해야 한다.
- `@Table`은 엔티티와 매핑할 테이블을 지정한다.
- `@Column`은 필드와 컬럼을 매핑하고, nullable, length, unique 같은 제약 조건을 설정할 수 있다.
- `@Id`는 기본 키를 매핑한다.
- `@GeneratedValue`는 기본 키 값을 자동 생성할 때 사용한다.
- 운영 DB에서는 `hibernate.hbm2ddl.auto`의 `create`, `create-drop`, `update`를 사용하면 위험하다.
- 운영 환경에서는 보통 `validate` 또는 `none`을 사용하고, DDL은 별도로 관리하는 것이 안전하다.

### 연관관계 매핑

- 테이블은 외래 키로 관계를 맺고, 객체는 참조로 관계를 맺는다.
- JPA에서는 `@ManyToOne`, `@OneToMany`, `@OneToOne`, `@ManyToMany`로 객체 참조와 테이블 외래 키를 매핑한다.
- 가장 기본이 되는 관계는 다대일 단방향이다.
- 양방향 연관관계에서는 두 객체가 서로 참조하지만, 테이블의 외래 키는 하나이므로 외래 키를 관리할 주인을 정해야 한다.
- 연관관계의 주인은 외래 키가 있는 쪽으로 정하는 것이 기본이다.
- 주인이 아닌 쪽은 `mappedBy`를 사용하고, 외래 키 변경에는 영향을 주지 않는다.
- 실무에서는 다대다 직접 매핑보다 연결 엔티티를 두고 일대다, 다대일로 풀어내는 것이 좋다.

### 프록시, 값 타입, JPQL

- `em.find()`는 실제 엔티티를 조회하고, `em.getReference()`는 데이터베이스 조회를 미루는 프록시 객체를 반환한다.
- 실무에서는 예측하기 어려운 즉시 로딩보다 지연 로딩을 기본으로 두는 것이 좋다.
- 값 타입은 식별자가 없고 값 자체가 중요한 타입이므로 불변 객체로 설계하는 것이 안전하다.
- JPQL은 테이블이 아닌 엔티티 객체를 대상으로 하는 객체지향 쿼리 언어이다.
- 페치 조인은 연관된 엔티티를 한 번에 조회해 지연 로딩으로 인한 추가 쿼리 문제를 줄일 수 있다.
- 벌크 연산은 영속성 컨텍스트를 무시하고 DB에 직접 실행되므로, 실행 후 영속성 컨텍스트 초기화가 필요할 수 있다.

## 어려웠던 점

- JPA가 SQL을 대신 만들어준다는 점은 편리하지만, SQL을 몰라도 된다는 의미는 아니라는 점을 이해해야 했다.
- 영속성 컨텍스트의 1차 캐시, 쓰기 지연, 변경 감지, 플러시가 서로 연결되어 있어 한 번에 이해하기 어려웠다.
- 연관관계 매핑에서 객체의 참조와 테이블의 외래 키를 동시에 생각해야 했다.
- 양방향 연관관계에서 연관관계의 주인과 `mappedBy` 개념이 헷갈렸다.
- 프록시와 지연 로딩은 편리하지만, 준영속 상태에서 초기화하면 문제가 생길 수 있다는 점을 주의해야 했다.
- JPQL은 SQL과 비슷하지만 엔티티 객체 기준으로 작성한다는 점을 계속 의식해야 했다.

## 해결 방법

- JPA는 SQL을 없애는 기술이 아니라 객체와 테이블 사이의 반복 매핑을 줄여주는 ORM 기술로 정리했다.
- 영속성 컨텍스트는 “엔티티를 관리하는 1차 작업 공간”으로 보고, 커밋 시점에 DB와 동기화된다고 이해했다.
- 연관관계는 먼저 다대일 단방향을 기준으로 잡고, 필요할 때 양방향을 추가한다고 정리했다.
- 연관관계의 주인은 “외래 키를 실제로 관리하는 쪽”으로 기억했다.
- 기본적으로 모든 연관관계는 지연 로딩으로 설계하고, 필요한 조회에서 페치 조인을 사용한다고 정리했다.
- 값 타입은 setter를 열어두기보다 생성자로 값을 받고 변경이 필요하면 새 객체로 교체하는 방식으로 이해했다.
- JPQL은 SQL처럼 보이지만 테이블명이 아니라 엔티티명, 컬럼명이 아니라 필드명을 사용한다고 구분했다.

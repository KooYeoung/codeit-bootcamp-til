## 오늘 배운 것

### 멀티스레드 기본 개념

- 프로세스는 실행 중인 프로그램이며, 하나의 프로세스는 최소 하나 이상의 스레드를 가진다.
- 스레드는 프로세스 안에서 실행되는 개별적인 연산 흐름이다.
- 하나의 프로세스 안에 여러 스레드가 존재하면 같은 메모리 공간을 공유할 수 있다.
- 멀티스레딩은 CPU 사용 효율을 높이고, 느린 입출력 작업을 분리하는 데 유용하다.
- 하지만 여러 스레드가 같은 데이터에 접근하면 동기화 문제가 발생할 수 있다.

### 멀티스레딩과 멀티프로세싱

- 멀티스레딩은 하나의 프로세스 안에서 여러 스레드가 동작하는 방식이다.
- 스레드들은 같은 프로세스의 메모리 공간과 접근 권한을 공유한다.
- 멀티프로세싱은 여러 프로세스가 각각 독립된 메모리 공간에서 동작하는 방식이다.
- 멀티프로세싱은 프로세스 간 통신이 필요하지만, 한 프로세스의 문제가 다른 프로세스에 영향을 덜 줄 수 있다.

### Java에서 스레드 생성

- Java에서는 `Thread` 클래스를 상속하거나 `Runnable` 인터페이스를 구현해 스레드를 만들 수 있다.
- `run()` 메서드에는 스레드가 실행할 작업을 작성한다.
- `start()`를 호출해야 새로운 스레드가 생성되어 `run()`이 실행된다.
- `run()`을 직접 호출하면 일반 메서드 호출일 뿐 새로운 스레드가 만들어지는 것이 아니다.

### 스레드 속성과 상태

- 스레드는 ID, 이름, 우선순위, 상태 등의 속성을 가진다.
- Java 스레드의 대표적인 상태에는 `NEW`, `RUNNABLE`, `BLOCKED`, `WAITING`, `TIMED_WAITING`, `TERMINATED`가 있다.
- `NEW`는 스레드 객체가 생성되었지만 아직 시작되지 않은 상태이다.
- `RUNNABLE`은 실행 가능 상태이다.
- `BLOCKED`는 Lock을 얻기 위해 대기하는 상태이다.
- `WAITING`은 다른 스레드의 알림이나 종료를 기다리는 상태이다.
- `TIMED_WAITING`은 정해진 시간 동안 대기하는 상태이다.
- `TERMINATED`는 스레드 실행이 끝난 상태이다.

### Daemon Thread

- Daemon Thread는 일반 스레드를 보조하는 스레드이다.
- 메인 스레드와 일반 사용자 스레드가 모두 종료되면 Daemon Thread는 함께 종료될 수 있다.
- `setDaemon(true)`는 `start()` 호출 전에 설정해야 한다.

### sleep, interrupt, join

- `sleep()`은 현재 스레드를 일정 시간 동안 일시 정지한다.
- `sleep()`은 지정한 시간만큼 정확히 멈추는 것이 아니라, 스케줄링 상황에 따라 더 오래 대기할 수 있다.
- `interrupt()`는 대기 중인 스레드에 인터럽트 신호를 보내 종료 흐름을 만들 때 사용할 수 있다.
- `join()`은 특정 스레드가 종료될 때까지 현재 스레드가 기다리도록 한다.
- 스레드 종료 순서를 맞춰야 할 때 `join()`을 사용할 수 있다.

### JVM과 스레드

- JVM은 사용자 모드에서 실행되는 하나의 프로세스이다.
- Java 스레드는 JVM 안에서 실행되며, OS 스케줄링의 영향을 받는다.
- 스레드 전환은 Context Switching 비용이 발생할 수 있다.
- 스레드마다 독립적인 Stack, PC Register, Native Method Stack을 가진다.
- Heap과 Method Area는 여러 스레드가 공유하는 영역이다.

### JVM 메인 메모리와 작업 메모리

- Java 메모리 모델에서는 메인 메모리와 스레드별 작업 메모리 개념을 구분한다.
- 공유 변수는 메인 메모리에 존재한다.
- 각 스레드는 변수 값을 작업 메모리로 가져와 사용한다.
- 작업 메모리의 변경이 메인 메모리에 즉시 반영되지 않을 수 있다.
- 이로 인해 여러 스레드가 같은 변수를 볼 때 값이 다르게 보일 수 있다.

### volatile

- `volatile`은 멀티스레드 환경에서 변수의 가시성을 보장하기 위해 사용한다.
- `volatile` 변수는 값을 읽을 때 메인 메모리와 동기화된다.
- 한 스레드에서 변경한 값을 다른 스레드가 확인해야 할 때 사용할 수 있다.
- 다만 `volatile`은 가시성을 보장하지만 복합 연산의 원자성을 보장하지는 않는다.

### 경쟁 조건

- 경쟁 조건은 여러 스레드가 같은 데이터에 동시에 접근하면서 발생하는 문제이다.
- 예를 들어 여러 스레드가 동시에 `counter++`를 수행하면 기대한 결과와 다른 값이 나올 수 있다.
- `counter++`는 한 줄처럼 보이지만 실제로는 읽기, 증가, 저장 과정으로 나뉘기 때문이다.
- 경쟁 조건은 컴파일 오류가 아니라 논리 오류로 나타날 수 있어 주의해야 한다.

### synchronized

- `synchronized`는 임계 영역을 만들어 여러 스레드가 동시에 접근하지 못하도록 제한하는 키워드이다.
- `synchronized` 메서드는 한 스레드가 실행 중일 때 다른 스레드가 동시에 실행할 수 없다.
- `synchronized` 블록을 사용하면 특정 코드 구간만 임계 영역으로 지정할 수 있다.
- 임계 영역은 가능한 작게 잡는 것이 좋다.

### Monitor Lock

- Java의 `synchronized`는 객체의 Monitor Lock을 사용한다.
- 한 객체의 Monitor Lock을 얻을 수 있는 스레드는 한 번에 하나뿐이다.
- 다른 스레드가 이미 Lock을 가지고 있다면, Lock을 얻으려는 스레드는 `BLOCKED` 상태가 될 수 있다.
- 정적 메서드 동기화와 인스턴스 메서드 동기화는 Lock 기준이 다르므로 주의해야 한다.

### AtomicInteger

- `AtomicInteger`는 원자적 연산을 제공하는 클래스이다.
- `incrementAndGet()` 같은 메서드를 사용하면 여러 스레드가 동시에 접근해도 안전하게 값을 증가시킬 수 있다.
- `synchronized`처럼 스레드를 막는 방식이 아니라 Non-blocking 방식으로 동작할 수 있다.
- 단순 카운터처럼 원자적 연산이 필요한 경우 유용하다.

### ReentrantLock

- `ReentrantLock`은 명시적으로 Lock을 걸고 해제할 수 있는 동기화 도구이다.
- `lock()`으로 잠그고, `unlock()`으로 해제한다.
- `unlock()`은 예외가 발생해도 반드시 실행되어야 하므로 보통 `finally` 블록에서 호출한다.
- `tryLock()`을 사용하면 일정 시간 동안 Lock 획득을 시도하고 실패했을 때 다른 처리를 할 수 있다.
- 동기화가 필요한 코드 범위를 세밀하게 제어할 수 있다.

### wait, notify

- `wait()`와 `notify()`는 스레드 간 대기와 알림을 처리하기 위한 메서드이다.
- `wait()`는 현재 스레드를 대기 상태로 만든다.
- `notify()`는 대기 중인 스레드 중 하나를 깨운다.
- `wait()`와 `notify()`는 `synchronized` 블록 안에서 사용해야 한다.
- 잘못 사용하면 Deadlock이 발생할 수 있다.

### LockSupport

- `LockSupport`는 스레드 제어를 위한 유틸리티 클래스이다.
- `park()`는 현재 스레드를 대기 상태로 만든다.
- `unpark()`는 특정 스레드를 다시 실행 가능 상태로 만든다.
- `wait()`와 `notify()`와 달리 `synchronized` 기반 Lock이 필요하지 않다.
- 스레드의 대기와 재개를 더 유연하게 제어할 수 있다.

### 연결 리스트와 동기화

- 연결 리스트는 Node들이 참조를 통해 연결된 선형 자료구조이다.
- 배열과 달리 노드의 `next`, `prev` 참조를 수정해 삽입과 삭제를 처리할 수 있다.
- 여러 스레드가 동시에 노드를 추가하거나 삭제하면 리스트 구조가 깨질 수 있다.
- 연결 리스트 같은 공유 자료구조를 여러 스레드에서 다룰 때는 동기화가 필요하다.
- `synchronized`나 `ReentrantLock`을 사용해 추가, 삭제 작업을 보호할 수 있다.

### 제네릭

- 제네릭은 클래스나 메서드에서 사용할 타입을 외부에서 지정할 수 있게 해주는 문법이다.
- `Object`를 사용하면 다양한 타입을 담을 수 있지만, 꺼낼 때 형 변환이 필요하고 런타임 오류가 발생할 수 있다.
- 제네릭을 사용하면 컴파일 시점에 타입을 검사할 수 있어 더 안전하다.
- 타입 매개변수 이름으로는 보통 `T`, `E`, `K`, `V` 등을 사용한다.
- 제네릭 메서드는 클래스 전체가 아니라 특정 메서드에만 타입 매개변수를 적용할 때 사용한다.

### 컬렉션 프레임워크

- 컬렉션 프레임워크는 여러 데이터를 효율적으로 저장하고 다루기 위한 Java 표준 자료구조 모음이다.
- 대표적으로 `List`, `Set`, `Map` 계열이 있다.
- `List`는 순서가 있고 중복을 허용한다.
- `Set`은 중복을 허용하지 않는다.
- `Map`은 Key와 Value 쌍으로 데이터를 저장한다.
- 자료구조의 특징을 이해하고 상황에 맞는 컬렉션을 선택하는 것이 중요하다.

## 어려웠던 점

- 스레드가 프로세스 안에서 동작하는 실행 흐름이라는 점은 이해했지만, OS 스케줄링과 연결해서 생각하니 어렵게 느껴졌다.
- `start()`와 `run()`의 차이가 중요하다는 점을 알았지만, 처음에는 둘 다 실행 메서드처럼 보여 헷갈렸다.
- 스레드 상태인 `RUNNABLE`, `BLOCKED`, `WAITING`, `TIMED_WAITING`의 차이를 구분하는 데 시간이 필요했다.
- `sleep()`으로 스레드 실행 순서를 맞추는 방식이 왜 위험한지 이해가 필요했다.
- JVM 메인 메모리와 작업 메모리 개념이 추상적으로 느껴졌다.
- `volatile`이 가시성은 보장하지만 원자성은 보장하지 않는다는 점이 어려웠다.
- `counter++`가 한 번의 연산처럼 보이지만 실제로는 여러 단계로 나뉘기 때문에 경쟁 조건이 발생한다는 점이 헷갈렸다.
- `synchronized`, `AtomicInteger`, `ReentrantLock`의 차이와 사용 시점을 구분하는 것이 어려웠다.
- `wait()`, `notify()`, `LockSupport`는 모두 스레드를 제어하는 기능처럼 보여 차이를 정리할 필요가 있었다.
- 제네릭은 문법 자체보다 왜 `Object` 대신 사용하는지 이해하는 과정이 필요했다.

## 해결 방법

- 스레드는 프로세스 안에서 실행되는 별도의 흐름이고, 여러 스레드는 같은 프로세스의 메모리 공간을 공유한다고 정리했다.
- `start()`는 새 스레드를 시작하는 메서드이고, `run()`은 스레드가 실행할 작업을 담는 메서드로 구분했다.
- 스레드 상태는 Lock 대기, 시간 대기, 종료 대기처럼 왜 대기하는지를 기준으로 구분해서 이해했다.
- `sleep()`은 정확한 순서 보장용이 아니라 일시 정지 기능이므로, 종료 시점 동기화가 필요하면 `join()`을 사용해야 한다고 정리했다.
- JVM 메인 메모리는 공유 변수의 원본 저장소이고, 작업 메모리는 각 스레드가 사용하는 변수 사본으로 이해했다.
- `volatile`은 여러 스레드가 같은 값을 볼 수 있게 하는 가시성 보장 도구로 정리했다.
- `counter++`는 읽기, 증가, 저장 과정으로 나뉘므로 여러 스레드가 동시에 실행하면 값이 누락될 수 있다고 이해했다.
- `synchronized`는 Lock을 사용해 임계 영역을 보호하는 방식으로 정리했다.
- `AtomicInteger`는 단순 원자 연산이 필요할 때 사용할 수 있는 Non-blocking 방식으로 이해했다.
- `ReentrantLock`은 Lock 범위와 획득 방식을 더 세밀하게 제어할 수 있는 도구로 정리했다.
- `wait()`와 `notify()`는 `synchronized` 기반의 대기와 알림이고, `LockSupport`는 Lock 없이 스레드를 park/unpark할 수 있는 도구로 구분했다.
- 제네릭은 타입을 미리 지정해 형 변환 오류를 줄이고 타입 안정성을 높이는 문법으로 이해했다.

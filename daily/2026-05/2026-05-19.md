## 오늘 배운 것

### 로그 추적기와 ThreadLocal

- 로그 추적기는 요청 흐름을 따라 Controller, Service, Repository 호출 단계를 로그로 남기기 위한 기능이다.
- 처음에는 `TraceId`를 메서드 파라미터로 전달해서 로그 깊이와 트랜잭션 ID를 맞출 수 있었다.
- 하지만 모든 메서드에 `TraceId` 파라미터가 추가되면 핵심 기능 코드가 지저분해진다.
- 필드에 `TraceId`를 저장하면 파라미터 전달 문제는 줄일 수 있지만, 스프링 빈은 싱글톤이므로 여러 쓰레드가 같은 필드에 접근해 동시성 문제가 발생할 수 있다.
- `ThreadLocal`은 쓰레드마다 별도의 저장 공간을 제공하므로, 같은 싱글톤 객체 안에서도 쓰레드별로 다른 값을 보관할 수 있다.
- 웹 애플리케이션처럼 쓰레드 풀을 사용하는 환경에서는 `ThreadLocal` 사용 후 반드시 `remove()`로 값을 제거해야 한다.

### 템플릿 메서드, 전략, 콜백 패턴

- 로그 추적기를 적용하면 핵심 기능보다 로그 처리 같은 부가 기능 코드가 더 많아질 수 있다.
- 이 문제를 해결하려면 변하는 부분과 변하지 않는 부분을 분리해야 한다.
- 템플릿 메서드 패턴은 부모 클래스에 변하지 않는 흐름을 두고, 자식 클래스에서 변하는 부분을 구현하게 한다.
- 전략 패턴은 변하는 로직을 별도의 전략 객체로 분리해서 실행 시점에 주입할 수 있게 한다.
- 템플릿 콜백 패턴은 전략 패턴의 변형으로, 실행할 로직을 콜백으로 전달하는 방식이다.
- 스프링에서 `JdbcTemplate`, `RestTemplate` 같은 이름이 붙은 기술들도 템플릿 콜백 패턴과 관련된 구조로 이해할 수 있다.

### 프록시와 동적 프록시

- 프록시는 실제 대상 객체 대신 클라이언트 요청을 먼저 받는 대리 객체이다.
- 프록시를 사용하면 기존 코드를 수정하지 않고 접근 제어, 캐싱, 로그 추적 같은 부가 기능을 추가할 수 있다.
- 프록시 패턴은 접근 제어가 목적이고, 데코레이터 패턴은 기능 확장이 목적이다.
- 직접 프록시 클래스를 만들면 대상 클래스 수만큼 프록시 클래스가 필요해진다.
- JDK 동적 프록시는 인터페이스 기반으로 프록시를 생성한다.
- CGLIB는 구체 클래스를 상속해서 프록시를 생성한다.
- 동적 프록시를 사용하면 프록시 클래스를 직접 계속 만들지 않고, 공통 부가 기능 로직을 재사용할 수 있다.

### 스프링 프록시와 빈 후처리기

- 스프링은 `ProxyFactory`를 통해 JDK 동적 프록시와 CGLIB를 통합해서 사용할 수 있게 해준다.
- 개발자는 `InvocationHandler`나 CGLIB의 `MethodInterceptor`를 각각 직접 만들기보다, 스프링의 `Advice`를 중심으로 부가 기능을 작성할 수 있다.
- `Pointcut`은 부가 기능을 적용할 대상 조건을 판단하고, `Advice`는 실제 부가 기능 로직을 담당한다.
- `Advisor`는 `Pointcut`과 `Advice`를 하나로 묶은 것이다.
- 빈 후처리기는 스프링이 빈 객체를 생성한 뒤 빈 저장소에 등록하기 직전에 객체를 조작하거나 다른 객체로 바꿔치기할 수 있다.
- 자동 프록시 생성기는 빈 후처리기를 사용해서 조건에 맞는 스프링 빈을 프록시로 교체한다.

### Spring AOP

- AOP는 핵심 기능과 부가 기능을 분리해서 관리하기 위한 프로그래밍 방식이다.
- 로그 추적, 트랜잭션, 권한 체크처럼 여러 곳에 반복되는 부가 기능을 하나의 관점으로 모아 적용할 수 있다.
- 스프링 AOP는 기본적으로 프록시 방식으로 동작한다.
- `@Aspect`는 포인트컷과 어드바이스를 애노테이션 기반으로 편리하게 정의할 수 있게 해준다.
- `@Around`는 메서드 실행 전후를 모두 감싸서 처리할 수 있는 가장 강력한 어드바이스이다.
- 포인트컷 표현식은 어디에 AOP를 적용할지 지정하는 조건이다.
- `execution`은 메서드 실행 조인 포인트를 매칭하며, 스프링 AOP에서 가장 많이 사용된다.
- `@annotation`을 사용하면 특정 애노테이션이 붙은 메서드에만 AOP를 적용할 수 있다.

### AOP 실무 주의사항

- 스프링 AOP는 프록시를 통해서 호출될 때만 적용된다.
- 같은 객체 내부에서 `this.internal()`처럼 내부 메서드를 호출하면 프록시를 거치지 않기 때문에 AOP가 적용되지 않는다.
- 내부 호출 문제는 자기 자신 주입, 지연 조회, 구조 변경 등으로 해결할 수 있다.
- 가장 권장되는 방법은 내부 호출이 필요하지 않도록 클래스를 분리해 구조를 변경하는 것이다.
- JDK 동적 프록시는 인터페이스 기반이므로 구체 클래스 타입으로 캐스팅할 수 없다.
- CGLIB는 구체 클래스를 상속하므로 `final` 클래스나 `final` 메서드에는 적용하기 어렵다.

## 어려웠던 점

- 처음에는 로그 추적기 문제를 단순히 로그 코드 중복 문제로만 봤지만, 실제로는 핵심 기능과 부가 기능이 섞이는 구조적 문제라는 점을 이해하는 데 시간이 필요했다.
- ThreadLocal은 편리하지만, 쓰레드 풀 환경에서 값을 제거하지 않으면 이전 요청의 값이 다음 요청에 남을 수 있다는 점이 중요했다.
- 템플릿 메서드 패턴, 전략 패턴, 템플릿 콜백 패턴은 모두 변하는 부분과 변하지 않는 부분을 분리한다는 공통점이 있어 차이를 구분하는 데 시간이 필요했다.
- 프록시 패턴과 데코레이터 패턴은 구조가 비슷하지만, 의도가 다르다는 점을 구분해야 했다.
- JDK 동적 프록시, CGLIB, ProxyFactory, Advisor, Advice, Pointcut 용어가 한 번에 많이 나와서 흐름을 연결하기 어려웠다.
- AOP 내부 호출 문제는 코드상으로는 단순한 메서드 호출처럼 보이지만, 프록시를 거치지 않는다는 점을 이해해야 했다.

## 해결 방법

- 로그 추적기 예제를 기준으로 “핵심 기능”과 “부가 기능”을 먼저 나누어 생각했다.
- ThreadLocal은 “쓰레드마다 별도의 보관함을 제공한다”는 방식으로 이해하고, 사용 후 `remove()`가 필수라고 정리했다.
- 템플릿 메서드 패턴은 상속, 전략 패턴은 위임, 템플릿 콜백 패턴은 콜백 전달 중심으로 구분했다.
- 프록시 패턴은 접근 제어, 데코레이터 패턴은 기능 추가라는 의도 중심으로 정리했다.
- 스프링 프록시 구조는 `Pointcut + Advice = Advisor`라는 식으로 먼저 단순화해서 이해했다.
- AOP는 프록시 방식으로 동작하므로 “외부에서 프록시를 통해 호출되는가?”를 기준으로 적용 여부를 판단했다.

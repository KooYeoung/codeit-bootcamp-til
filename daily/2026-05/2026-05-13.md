## 오늘 배운 것

### 람다와 함수형 인터페이스

- 람다는 이름 없는 함수처럼 표현할 수 있는 문법이다.
- Java에서는 독립적인 함수를 직접 다루지 않고, 함수형 인터페이스를 통해 람다를 사용할 수 있다.
- 함수형 인터페이스는 추상 메서드가 하나만 있는 인터페이스이다.
- 람다식은 함수형 인터페이스의 추상 메서드 시그니처와 맞아야 한다.
- 람다는 변수에 담거나, 메서드의 인자로 전달하거나, 메서드의 반환값으로 사용할 수 있다.
- 람다를 값처럼 주고받는 구조를 통해 고차 함수 개념을 이해할 수 있었다.
- 익명 클래스보다 람다가 더 간결하지만, 여러 메서드를 구현하거나 상태를 가져야 하는 경우에는 익명 클래스가 더 적합할 수 있다.

### 메서드 참조

- 메서드 참조는 이미 존재하는 메서드를 람다식 대신 더 간단하게 표현하는 문법이다.
- `(x, y) -> add(x, y)`처럼 단순히 기존 메서드를 호출하는 람다는 `클래스명::메서드명` 형태로 바꿀 수 있다.
- 메서드 참조는 새로운 로직을 만드는 것이 아니라, 기존 메서드를 함수형 인터페이스의 시그니처에 맞게 연결하는 방식이다.
- 람다식이 너무 단순한 메서드 호출만 하고 있다면 메서드 참조를 고려할 수 있다.

### Stream API

- Stream API는 컬렉션이나 배열의 데이터를 파이프라인 형태로 처리할 수 있게 해주는 기능이다.
- Stream은 I/O Stream과 다른 개념이다.
- Stream을 사용하면 반복문으로 “어떻게 처리할지”를 직접 작성하기보다, “무엇을 할지”를 선언적으로 표현할 수 있다.
- Stream 연산은 중간 연산과 최종 연산으로 나뉜다.
- `filter`, `map`, `sorted`, `distinct`, `limit` 같은 중간 연산은 최종 연산이 실행되기 전까지 바로 동작하지 않는다.
- `toList`, `forEach`, `collect`, `reduce`, `count` 같은 최종 연산이 호출될 때 실제 처리가 시작된다.
- Stream은 한 번 사용하면 다시 사용할 수 없으므로, 다시 처리하려면 새 Stream을 생성해야 한다.

### Collectors와 Optional

- `Collectors`는 Stream 결과를 List, Set, Map, 통계, 문자열 등으로 수집할 때 사용한다.
- `groupingBy`는 특정 기준으로 데이터를 그룹화할 때 사용한다.
- `partitioningBy`는 조건 결과에 따라 `true`, `false` 두 그룹으로 나눌 때 사용한다.
- 다운스트림 컬렉터를 사용하면 그룹화 이후 각 그룹 내부에서 개수, 합계, 평균, 매핑 같은 추가 처리를 할 수 있다.
- `Optional`은 값이 있을 수도 있고 없을 수도 있음을 명시적으로 표현하는 도구이다.
- `Optional`을 사용하면 메서드 시그니처만 보고도 값이 없을 수 있음을 알 수 있다.
- `orElse()`는 대체값을 즉시 준비하고, `orElseGet()`은 값이 없을 때만 대체값을 생성한다는 차이가 있다.

### 디폴트 메서드와 병렬 스트림

- 디폴트 메서드는 인터페이스에 새로운 기능을 추가할 때 기존 구현체를 깨뜨리지 않기 위해 도입되었다.
- 인터페이스에 메서드 본문을 작성할 수 있지만, 남용하면 인터페이스의 역할이 흐려질 수 있다.
- 병렬 스트림은 내부적으로 Fork/Join 공용 풀을 사용해 작업을 병렬로 처리할 수 있다.
- CPU 연산 중심 작업에서는 병렬 스트림이 도움이 될 수 있지만, I/O 대기 작업이나 공유 자원이 많은 작업에서는 오히려 성능이 떨어질 수 있다.
- 병렬 스트림은 편리하지만 항상 빠른 것은 아니므로 작업 성격과 부작용 여부를 확인해야 한다.

### 함수형 프로그래밍

- 함수형 프로그래밍은 “어떻게 처리할지”보다 “무엇을 얻고 싶은지”에 집중하는 선언형 프로그래밍 스타일과 관련이 있다.
- 순수 함수, 불변성, 부수 효과 최소화가 중요한 개념이다.
- Java는 기본적으로 객체지향 언어이지만, Java 8 이후 람다, 함수형 인터페이스, Stream API를 통해 함수형 스타일도 함께 사용할 수 있다.
- 한 가지 패러다임만 고집하기보다, 상황에 맞게 객체지향과 함수형 스타일을 조합하는 것이 중요하다고 느꼈다.

## 어려웠던 점

- 람다가 실제로는 함수형 인터페이스 타입으로 다뤄진다는 점이 처음에는 추상적으로 느껴졌다.
- 람다와 익명 클래스가 비슷해 보이지만, 사용할 수 있는 상황과 `this`의 의미가 다르다는 점을 구분해야 했다.
- Stream의 중간 연산이 바로 실행되지 않고 최종 연산 시점에 지연 실행된다는 개념이 낯설었다.
- `map`, `flatMap`, `collect`, `reduce`의 역할을 각각 구분하는 데 시간이 필요했다.
- `Optional`을 언제 반환 타입으로 쓰고, 언제 쓰지 말아야 하는지 기준을 잡는 것이 어려웠다.
- 병렬 스트림은 편리해 보이지만 항상 성능이 좋아지는 것은 아니라는 점이 중요했다.

## 해결 방법

- 람다는 “함수형 인터페이스의 구현을 간결하게 표현한 것”으로 정리했다.
- 함수형 인터페이스는 추상 메서드가 하나만 있어야 하며, 람다식은 그 메서드의 매개변수와 반환 타입에 맞아야 한다고 이해했다.
- 익명 클래스는 상태나 여러 메서드 구현이 필요할 때, 람다는 단순한 함수형 인터페이스 구현이 필요할 때 사용한다고 구분했다.
- Stream은 `생성 → 중간 연산 → 최종 연산` 흐름으로 나누어 이해했다.
- 중간 연산은 파이프라인을 구성하는 단계이고, 최종 연산이 호출될 때 실제 처리가 시작된다고 정리했다.
- Optional은 null을 완전히 없애는 도구가 아니라, “값이 없을 수 있음”을 명확히 표현하는 도구로 이해했다.
- 병렬 스트림은 CPU 중심 작업인지, 공유 상태나 I/O 대기가 있는지 먼저 확인하고 사용해야 한다고 정리했다.

## 오늘 배운 것

### Git 협업의 중요성

- 협업이 원활하지 않으면 일정 지연, 중복 작업, 소통 부족, 신뢰 저하 같은 문제가 발생할 수 있다.
- Git과 GitHub를 제대로 이해하지 못하면 코드 충돌, 브랜치 관리 실패, 버그 발생 등의 문제가 생길 수 있다.
- 협업에서는 Git 사용 능력뿐만 아니라 커뮤니케이션 능력도 중요하다.
- Pull Request, 코드 리뷰, 브랜치 관리 전략 등을 활용하면 협업 과정의 실수를 줄일 수 있다.

### Git 협업 기본 흐름

- 일반적인 Git 협업 흐름은 다음과 같이 진행된다.

```text
git pull
→ 브랜치 생성
→ 작업
→ git add
→ git commit
→ git push
→ Pull Request 생성
```

- 작업을 시작하기 전에 원격 저장소의 최신 내용을 가져오는 것이 중요하다.
- 새로운 기능이나 수정 작업은 별도 브랜치를 만들어 진행하는 것이 좋다.
- 작업이 끝나면 Pull Request를 통해 코드 리뷰와 병합 요청을 진행한다.

### Pull Request

- Pull Request는 내가 작업한 변경사항을 다른 사람에게 검토받고 병합해 달라고 요청하는 GitHub 기능이다.
- Pull Request는 줄여서 PR이라고도 부른다.
- PR을 사용하면 코드 변경 내용을 리뷰받을 수 있고, 작업 의도와 변경 내용을 문서처럼 남길 수 있다.
- 개인 프로젝트에서도 PR을 사용하면 작업 기록을 명확히 남기고, 변경사항을 다시 검토하는 데 도움이 된다.
- 팀 프로젝트에서는 PR을 통해 코드 리뷰, 지식 공유, 품질 개선을 할 수 있다.

### Pull Request 상태

- PR은 크게 `Open`, `Merged`, `Closed` 상태를 가진다.
- `Open`은 아직 검토 중이거나 추가 작업이 필요한 상태이다.
- `Merged`는 PR의 변경사항이 대상 브랜치에 병합된 상태이다.
- `Closed`는 병합하지 않고 닫힌 상태이다.
- Merged 상태도 넓게 보면 Closed 상태에 포함된다.

### Pull Request 머지 방식

GitHub에서는 PR을 병합할 때 주로 세 가지 방식을 사용할 수 있다.

#### Merge Commit

- 브랜치의 커밋 이력을 유지하면서 병합 커밋을 생성하는 방식이다.
- 장점은 작업 흐름과 브랜치 이력을 명확히 남길 수 있다는 것이다.
- 단점은 커밋 히스토리가 복잡해질 수 있다는 것이다.

#### Squash and Merge

- PR의 여러 커밋을 하나의 커밋으로 합쳐 병합하는 방식이다.
- 장점은 커밋 히스토리를 깔끔하게 유지할 수 있다는 것이다.
- 단점은 개별 커밋의 세부 이력이 사라질 수 있다는 것이다.
- GitHub에는 PR 기록이 남기 때문에 필요한 경우 PR에서 기존 커밋 이력을 확인할 수 있다.

#### Rebase and Merge

- 작업 브랜치의 커밋들을 대상 브랜치 위로 재배치한 뒤 병합하는 방식이다.
- 장점은 커밋 히스토리를 선형적으로 유지할 수 있다는 것이다.
- 단점은 커밋 해시가 바뀌고, 협업 상황에서는 충돌이 복잡해질 수 있다는 것이다.

### Fork

- Fork는 GitHub에서 제공하는 기능으로, 다른 사람의 원격 저장소를 내 계정으로 복사하는 기능이다.
- Fork한 저장소는 원본 저장소와 분리되어 있으므로 내 저장소에서 자유롭게 작업할 수 있다.
- 작업이 끝나면 원본 저장소에 Pull Request를 보내 기여할 수 있다.
- 일반적인 팀 협업에서는 브랜치 기반 PR을 많이 사용하고, 오픈소스 기여에서는 Fork 기반 PR을 많이 사용한다.

### Pull Request 충돌 해결

- 충돌은 두 브랜치에서 같은 파일의 같은 부분을 서로 다르게 수정했을 때 발생한다.
- GitHub에서 충돌이 발생한 PR은 바로 병합할 수 없고, 충돌을 먼저 해결해야 한다.
- 충돌 해결 방법은 크게 두 가지이다.

```text
1. GitHub의 Resolve conflicts 버튼으로 해결
2. 로컬에서 직접 충돌 해결
```

- GitHub UI로 간단한 충돌을 해결할 수 있지만, 복잡한 충돌은 로컬에서 IDE나 에디터를 사용해 해결하는 것이 더 안전하다.
- 충돌을 해결한 후에는 수정된 파일을 저장하고, `git add`, `git commit`, `git push` 흐름으로 반영한다.

### 충돌을 줄이는 방법

- 작업 전에 최신 코드를 자주 가져온다.
- PR을 작게 만든다.
- 파일을 너무 크게 만들지 않고 역할별로 나눈다.
- 팀원들과 작업 범위와 우선순위를 자주 공유한다.
- 오래된 브랜치에서 장기간 작업하지 않도록 주의한다.

### 좋은 Pull Request

- 좋은 PR은 목적과 변경사항이 명확해야 한다.
- 리뷰어가 왜 이 변경이 필요한지 이해할 수 있어야 한다.
- 하나의 PR은 하나의 문제나 기능을 다루는 것이 좋다.
- PR이 너무 크면 리뷰하기 어렵기 때문에 가능한 작게 나누는 것이 좋다.
- 필요한 경우 스크린샷, 테스트 결과, 관련 문서 링크를 함께 작성하면 좋다.
- PR을 올리기 전에는 직접 테스트해서 정상 동작 여부를 확인해야 한다.

### 코드 리뷰

- 코드 리뷰는 Pull Request를 통해 다른 사람이 작성한 코드를 검토하고 피드백을 주고받는 과정이다.
- 코드 리뷰는 코드 품질을 높이고, 버그를 줄이며, 팀원 간 지식을 공유하는 데 도움이 된다.
- GitHub에서는 라인별 코멘트, 멀티라인 코멘트, Suggested Change, Viewed 체크, 커밋별 리뷰 기능을 제공한다.
- 리뷰 제출 방식에는 `Comment`, `Approve`, `Request changes`가 있다.

### 리뷰 제출 방식

- `Comment`는 의견이나 질문을 남길 때 사용한다.
- `Approve`는 변경사항이 문제없다고 승인할 때 사용한다.
- `Request changes`는 수정이 필요하다고 요청할 때 사용한다.
- Branch Protection 설정에 따라 Approve를 받아야만 병합할 수 있도록 제한할 수 있다.

### 좋은 커밋

- 좋은 커밋은 의미 있는 작업 단위로 나누어져 있어야 한다.
- 하나의 커밋은 하나의 기능이나 수정사항을 담는 것이 좋다.
- 커밋은 정상적으로 동작 가능한 상태여야 한다.
- 커밋 메시지는 나중에 봐도 어떤 작업인지 이해할 수 있게 작성해야 한다.
- PR은 큰 작업 단위이고, 커밋은 그 PR을 구성하는 작은 논리 단위라고 볼 수 있다.

### GitHub PR 관리 기능

- GitHub Repository 설정에서 기본 브랜치를 변경할 수 있다.
- PR 머지 방식을 제한할 수 있다.
- PR이 머지된 후 브랜치를 자동 삭제하도록 설정할 수 있다.
- Branch Protection을 통해 main 브랜치에 직접 push하지 못하도록 제한할 수 있다.
- PR 승인 수, Code Owners 리뷰, 선형 히스토리 유지 같은 규칙을 설정할 수 있다.

### 브랜치 관리 전략

- 브랜치 관리 전략은 팀이 어떤 방식으로 브랜치를 만들고, 작업하고, 병합하고, 배포할지 정하는 규칙이다.
- 프로젝트 규모, 배포 주기, 팀의 협업 방식에 따라 적절한 브랜치 전략을 선택해야 한다.

#### GitHub Flow

- GitHub Flow는 `main` 브랜치를 중심으로 하는 단순한 브랜치 전략이다.
- `main` 브랜치는 항상 배포 가능한 상태를 유지해야 한다.
- 새로운 기능이나 수정 작업은 `main`에서 별도 브랜치를 만들어 진행한다.
- 작업이 끝나면 Pull Request를 생성하고, 코드 리뷰 후 `main`에 병합한다.
- 구조가 단순해서 빠르게 개발하고 자주 배포하는 프로젝트에 적합하다.

기본 흐름은 다음과 같다.

```text
main
→ feature branch 생성
→ 작업 및 commit
→ push
→ Pull Request 생성
→ review
→ main에 merge
```

#### Git Flow

- Git Flow는 `main`, `develop`, `feature`, `release`, `hotfix` 브랜치를 사용하는 전략이다.
- `main`은 실제 배포 버전을 관리하고, `develop`은 다음 배포를 위한 개발 내용을 관리한다.
- 새로운 기능은 `feature` 브랜치에서 개발한 뒤 `develop`으로 병합한다.
- 배포 준비는 `release` 브랜치에서 진행하고, 긴급 수정은 `hotfix` 브랜치에서 처리한다.
- 브랜치 구조가 명확하지만, 관리해야 할 브랜치가 많아져 복잡해질 수 있다.
- 배포 주기가 명확하고 버전 관리가 중요한 프로젝트에 적합하다.

#### GitLab Flow

- GitLab Flow는 GitHub Flow와 Git Flow의 장점을 섞고, CI/CD 환경에 맞게 확장한 전략이다.
- `main`, `feature`, `environment` 브랜치를 중심으로 사용한다.
- `main` 브랜치는 항상 배포 가능한 상태를 유지한다.
- `feature` 브랜치에서 기능을 개발하고, 필요에 따라 환경별 브랜치로 배포 흐름을 관리한다.
- `environment` 브랜치는 production, develop, QA, staging 같은 배포 환경을 구분하기 위해 사용한다.
- 여러 배포 환경을 운영하는 프로젝트에서 유용하다.

#### Trunk Based Development

- Trunk Based Development는 `main` 또는 `master` 같은 중심 브랜치에 작은 단위로 자주 병합하는 전략이다.
- 긴 수명의 브랜치를 오래 유지하지 않고, 가능한 빠르게 중심 브랜치에 병합한다.
- Pull Request의 주기가 매우 짧고, 작은 변경사항을 자주 통합한다.
- 아직 완성되지 않은 기능은 `feature toggle`을 사용해 비활성화한 상태로 병합할 수 있다.
- 브랜치 관리가 단순하고 빠른 피드백을 받을 수 있지만, 테스트와 CI/CD 환경이 잘 갖춰져 있어야 한다.

### GitHub를 사용한 협업 자동화

- GitHub를 사용하면 Pull Request, 코드 리뷰, 브랜치 보호뿐만 아니라 협업 과정의 일부를 자동화할 수 있다.
- 대표적인 자동화 기능으로 GitHub Actions가 있다.
- GitHub Actions는 특정 이벤트가 발생했을 때 미리 정의한 작업을 자동으로 실행하는 기능이다.
- 예를 들어 push, pull request 같은 이벤트가 발생했을 때 테스트, 빌드, Linting 등을 자동으로 실행할 수 있다.
- 이를 통해 사람이 매번 직접 확인하지 않아도 코드 스타일, 테스트 통과 여부, 빌드 성공 여부를 확인할 수 있다.

### Workflow와 YAML

- GitHub Actions의 자동화 작업은 Workflow라고 부른다.
- Workflow는 프로젝트의 `.github/workflows` 폴더 안에 YAML 파일로 작성한다.
- YAML 파일에는 워크플로우 이름, 실행 조건, 실행할 작업을 정의한다.

예시:

```yaml
name: Basic Workflow Example

on: [push]

jobs:
  example_job:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Print Hello
        run: echo "Hello, GitHub!"
```

### Linting과 테스트 자동화

- Linting은 코드 스타일이나 규칙 위반을 자동으로 검사하는 과정이다.
- 테스트 자동화는 PR이나 push 시점에 테스트를 자동으로 실행해 코드가 정상 동작하는지 확인하는 과정이다.
- Linting과 테스트를 자동화하면 리뷰어가 코드 스타일보다 핵심 로직 검토에 더 집중할 수 있다.
- 실패한 테스트나 Linting 오류가 있으면 PR을 병합하기 전에 미리 발견할 수 있다.
- Java에서는 IntelliJ Formatter, Checkstyle, Spotless, google-java-format, SpotBugs 같은 도구를 활용할 수 있다.

### Branch Protection과 자동화 연결

- Branch Protection을 사용하면 main 브랜치에 직접 push하지 못하도록 제한할 수 있다.
- 또한 PR이 머지되기 전에 테스트나 Linting 같은 자동화 작업이 통과해야만 병합되도록 설정할 수 있다.
- 이렇게 하면 실수로 문제가 있는 코드가 main 브랜치에 반영되는 것을 줄일 수 있다.

## 어려웠던 점

- Pull Request가 단순히 병합 요청이 아니라 코드 리뷰와 작업 기록의 역할도 한다는 점을 이해하는 데 시간이 필요했다.
- Merge Commit, Squash and Merge, Rebase and Merge의 차이가 헷갈렸다.
- Fork 기반 PR과 브랜치 기반 PR을 어떤 상황에서 사용하는지 구분이 필요했다.
- GitHub에서 충돌을 해결하는 방식과 로컬에서 충돌을 해결하는 방식의 차이가 헷갈렸다.
- 좋은 PR과 좋은 커밋의 기준이 비슷해 보였지만, PR은 더 큰 작업 단위이고 커밋은 그 안의 작은 논리 단위라는 점을 구분할 필요가 있었다.
- GitHub Flow, Git Flow, GitLab Flow, Trunk Based Development가 모두 브랜치 전략이지만, 어떤 상황에서 어떤 전략을 사용하는지 구분이 필요했다.
- GitHub Actions, Workflow, YAML 같은 자동화 관련 용어가 처음에는 낯설었다.
- Linting과 테스트 자동화가 Pull Request 흐름에서 어떤 역할을 하는지 구분이 필요했다.

## 해결 방법

- PR은 작업 내용을 리뷰받고 병합을 요청하는 단위로 정리했다.
- Merge Commit은 이력을 보존하는 방식, Squash는 커밋을 하나로 압축하는 방식, Rebase는 커밋을 재배치해 선형 이력을 만드는 방식으로 정리했다.
- 일반적인 팀 협업에서는 브랜치 기반 PR을 사용하고, 오픈소스 기여에서는 Fork 기반 PR을 많이 사용한다고 정리했다.
- GitHub Actions는 특정 이벤트가 발생했을 때 자동으로 작업을 실행하는 기능으로 정리했다.
- 충돌이 단순하면 GitHub UI에서 해결할 수 있지만, 복잡한 충돌은 로컬에서 해결하는 것이 더 안전하다고 이해했다.
- 커밋은 PR 안에서 변경사항을 의미 있게 나눈 작은 단위로 이해했다.
- GitHub Flow는 단순한 PR 중심 전략, Git Flow는 배포 버전 관리가 명확한 전략, GitLab Flow는 환경별 배포 흐름을 고려한 전략, Trunk Based Development는 중심 브랜치에 자주 통합하는 전략으로 정리했다.
- Workflow는 자동화 작업의 정의 파일이고, YAML은 그 설정을 작성하는 형식으로 이해했다.
- Linting과 테스트 자동화는 PR 병합 전에 코드 품질을 확인하는 안전장치로 정리했다.

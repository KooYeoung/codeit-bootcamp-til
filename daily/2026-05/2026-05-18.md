## 오늘 배운 것

### 데이터 접근 기술 큰 그림

- 데이터 접근 기술은 크게 SQL Mapper 계열과 ORM 계열로 나눌 수 있다.
- SQL Mapper에는 `JdbcTemplate`, MyBatis가 있다.
- SQL Mapper는 SQL을 직접 작성하고, 결과를 객체에 매핑하는 방식이다.
- ORM 계열에는 JPA, Hibernate, 스프링 데이터 JPA, Querydsl이 있다.
- ORM은 객체를 저장하고 조회하면 내부에서 SQL을 만들어 데이터베이스와 연동해준다.
- JPA는 자바 진영의 ORM 표준이고, Hibernate는 대표적인 JPA 구현체이다.
- 스프링 데이터 JPA와 Querydsl은 JPA를 더 편리하게 사용하도록 돕는 도구이다.

### JdbcTemplate과 MyBatis

- `JdbcTemplate`은 순수 JDBC의 반복 코드를 줄여주는 스프링의 데이터 접근 도구이다.
- 개발자는 SQL, 파라미터, 결과 매핑에 집중하고, 커넥션 획득과 리소스 정리는 `JdbcTemplate`이 처리한다.
- `JdbcTemplate`은 설정이 간단하고 단순한 SQL 처리에 적합하다.
- 단점은 동적 SQL을 직접 자바 코드로 조립해야 해서 복잡한 조건이 많아지면 코드가 지저분해질 수 있다는 점이다.
- MyBatis는 SQL을 XML에 작성할 수 있고, 동적 쿼리를 편리하게 작성할 수 있다.
- 복잡한 SQL이나 동적 조건이 많은 프로젝트에서는 MyBatis가 더 편할 수 있다.

### JPA와 스프링 데이터 JPA

- JPA는 SQL 중심 개발에서 객체 중심 개발로 이동할 수 있게 도와준다.
- SQL을 직접 작성하는 대신 엔티티 객체를 저장, 조회, 수정, 삭제하는 방식으로 개발할 수 있다.
- JPA는 1차 캐시, 동일성 보장, 쓰기 지연, 지연 로딩 같은 성능 최적화 기능도 제공한다.
- 스프링 데이터 JPA는 `JpaRepository`를 상속하는 인터페이스만 만들어도 기본 CRUD 기능을 제공한다.
- 스프링 데이터 JPA는 메서드 이름을 분석해서 쿼리를 생성하는 쿼리 메서드 기능을 제공한다.
- 단순 CRUD와 단순 조회는 스프링 데이터 JPA가 매우 편리하다.

### Querydsl

- Querydsl은 자바 코드로 타입 안전한 쿼리를 작성할 수 있게 도와준다.
- 문자열 기반 JPQL이나 SQL은 오타가 있어도 컴파일 시점에 잡기 어렵다.
- Querydsl은 Q 타입을 기반으로 쿼리를 작성하므로 컴파일 시점에 오류를 잡을 수 있다.
- 동적 쿼리는 `BooleanBuilder` 또는 조건 메서드 분리 방식으로 깔끔하게 처리할 수 있다.
- `where()`에 여러 조건을 전달하면 AND 조건으로 처리되고, `null` 조건은 무시된다.
- 단순 조회는 스프링 데이터 JPA로 처리하고, 복잡한 동적 조회는 Querydsl로 처리하는 방식이 실용적이다.

### 스프링 트랜잭션

- 스프링은 `PlatformTransactionManager`를 통해 트랜잭션 기술을 추상화한다.
- `@Transactional`은 선언적 트랜잭션 방식이며, 실무에서 가장 많이 사용한다.
- `@Transactional`은 기본적으로 프록시 기반 AOP를 통해 트랜잭션 시작, 커밋, 롤백을 처리한다.
- 트랜잭션은 보통 서비스 계층의 비즈니스 로직 경계에 적용한다.
- 기본적으로 런타임 예외가 발생하면 롤백하고, 체크 예외가 발생하면 커밋한다.
- 체크 예외도 롤백하고 싶다면 `rollbackFor` 옵션을 사용할 수 있다.

### 트랜잭션 전파

- 트랜잭션 전파는 이미 트랜잭션이 있을 때 새로 호출된 메서드가 기존 트랜잭션에 참여할지, 새 트랜잭션을 만들지 결정하는 규칙이다.
- 기본 전파 옵션은 `REQUIRED`이다.
- `REQUIRED`는 기존 트랜잭션이 있으면 참여하고, 없으면 새로 만든다.
- `REQUIRES_NEW`는 기존 트랜잭션이 있어도 항상 새로운 트랜잭션을 만든다.
- `REQUIRES_NEW`를 사용하면 물리 트랜잭션과 DB 커넥션이 분리될 수 있다.
- 로그 저장 실패가 전체 비즈니스 실패로 이어지면 안 되는 경우처럼, 일부 작업만 독립적으로 롤백하고 싶을 때 `REQUIRES_NEW`를 고려할 수 있다.

## 어려웠던 점

- 데이터 접근 기술이 많아서 각각의 위치와 사용 목적을 구분하는 데 시간이 필요했다.
- JdbcTemplate, MyBatis, JPA, 스프링 데이터 JPA, Querydsl이 서로 대체 관계인지 보완 관계인지 헷갈렸다.
- JPA는 SQL을 직접 작성하지 않아도 된다는 장점이 있지만, 영속성 컨텍스트와 트랜잭션 흐름을 함께 이해해야 해서 어렵게 느껴졌다.
- 스프링 데이터 JPA는 편리하지만, 복잡한 동적 쿼리를 어디까지 처리해야 하는지 기준이 필요했다.
- Querydsl은 장점은 명확하지만 Q 타입 생성과 설정이 필요하다는 점이 낯설었다.
- 트랜잭션 전파에서 논리 트랜잭션과 물리 트랜잭션의 차이가 처음에는 어렵게 느껴졌다.
- `REQUIRED`에서 내부 트랜잭션이 실패했을 때 rollback-only가 설정되는 흐름을 더 복습해야 한다고 느꼈다.

## 해결 방법

- 데이터 접근 기술은 먼저 SQL Mapper와 ORM으로 나누어 큰 그림을 잡았다.
- 단순 SQL 중심이면 JdbcTemplate 또는 MyBatis, 객체 중심 도메인 개발이면 JPA 계열을 우선 고려한다고 정리했다.
- JPA는 핵심이고, 스프링 데이터 JPA와 Querydsl은 JPA를 편하게 쓰기 위한 도구라고 이해했다.
- 단순 CRUD는 스프링 데이터 JPA, 복잡한 동적 조회는 Querydsl, 정말 복잡한 SQL은 JdbcTemplate이나 MyBatis로 보완하는 방향으로 정리했다.
- 트랜잭션은 서비스 계층의 비즈니스 작업 단위에 적용한다고 이해했다.
- 트랜잭션 전파는 “기존 트랜잭션에 참여할 것인가, 새 트랜잭션을 만들 것인가”라는 기준으로 정리했다.
- `REQUIRES_NEW`는 독립적인 물리 트랜잭션을 만들기 때문에 커넥션 사용량과 설계 의도를 함께 고려해야 한다고 이해했다.

## 오늘 배운 것

### 실행 구조와 Reflection

- Spring MVC는 컨트롤러 메서드를 직접 정적으로 호출하지 않고, 요청 정보에 맞는 HandlerMethod를 찾아 동적으로 호출한다.
- Java Reflection을 사용하면 런타임에 클래스와 메서드 정보를 조회하고 실행할 수 있다.
- MethodParameter는 메서드 파라미터의 타입, 애노테이션, 이름 같은 정보를 표현한다.
- `InvocableHandlerMethod`는 ArgumentResolver를 통해 파라미터 값을 준비하고 실제 메서드를 호출한다.
- `ServletInvocableHandlerMethod`는 Servlet 환경에서 메서드 실행 결과를 ReturnValueHandler로 처리하는 역할까지 포함한다.

### Interceptor와 ControllerAdvice

- 인터셉터는 DispatcherServlet 이후, 컨트롤러 호출 전후에 공통 로직을 실행할 수 있는 Spring MVC 기능이다.
- `preHandle()`은 컨트롤러 호출 전에 실행되며, false를 반환하면 요청 처리를 중단할 수 있다.
- `postHandle()`은 컨트롤러 호출 후 View 렌더링 전에 실행된다.
- `afterCompletion()`은 View 렌더링 후 실행되며, 예외 발생 여부와 관계없이 정리 작업을 수행하기 좋다.
- `@ControllerAdvice`는 여러 Controller에 공통 기능을 적용할 수 있게 해준다.
- `@InitBinder`, `@ModelAttribute`, `@ExceptionHandler`와 함께 사용하면 전역 바인딩, 전역 모델, 전역 예외 처리를 구성할 수 있다.

### 예외 처리

- Servlet에서 예외가 처리되지 않으면 WAS까지 전달되고, WAS는 오류 페이지를 다시 요청할 수 있다.
- `response.sendError()`는 예외를 던지지 않아도 오류 상태를 설정하고 오류 처리 흐름을 시작할 수 있다.
- Spring Boot는 기본적으로 `/error` 경로와 BasicErrorController를 통해 오류 응답을 처리한다.
- HTML 요청에는 오류 페이지를 반환하고, JSON API 요청에는 오류 정보를 JSON 형태로 반환할 수 있다.
- HandlerExceptionResolver는 컨트롤러에서 발생한 예외를 처리할 기회를 제공한다.
- `@ExceptionHandler`는 특정 컨트롤러 또는 `@ControllerAdvice` 범위에서 예외 처리 메서드를 정의한다.
- `@RestControllerAdvice`를 사용하면 API 예외 응답을 전역으로 일관되게 관리하기 좋다.

### Multipart와 RestClient

- 파일 업로드는 일반 Form 전송이 아니라 `multipart/form-data` 형식으로 처리한다.
- Multipart 요청은 여러 part로 나뉘며, 각 part는 일반 텍스트 필드나 파일 데이터를 담을 수 있다.
- Servlet에서는 `Part` 또는 `HttpServletRequest` 기반으로 파일을 처리할 수 있지만, Spring MVC에서는 `MultipartFile`을 사용하면 더 편리하다.
- `@RequestParam MultipartFile`은 단순 파일 업로드에 적합하다.
- `@ModelAttribute`는 파일과 일반 Form 필드를 함께 객체로 받을 때 유용하다.
- `@RequestPart`는 JSON part와 파일 part를 함께 받는 API에서 유용하다.
- RestClient는 Spring 6.1부터 사용할 수 있는 동기 HTTP 클라이언트로, 외부 API 호출을 체이닝 방식으로 작성할 수 있다.
- HTTP Interface는 인터페이스에 HTTP 호출 정보를 선언하고 프록시 방식으로 외부 API를 호출할 수 있게 해준다.

## 어려웠던 점

- Reflection은 일반 메서드 호출과 다르게 런타임에 메서드 정보를 찾아 호출하므로 추상적으로 느껴졌다.
- Spring MVC 내부에서 ArgumentResolver와 ReturnValueHandler가 언제 각각 사용되는지 흐름을 연결해야 했다.
- Interceptor, Filter, AOP가 모두 공통 기능 처리에 사용될 수 있어 적용 위치와 목적을 구분해야 했다.
- 예외 처리 방식이 Servlet 오류 페이지, Spring Boot BasicErrorController, HandlerExceptionResolver, `@ExceptionHandler` 등 여러 단계로 나뉘어 복잡했다.
- Multipart 요청은 요청 본문이 여러 part로 나뉘기 때문에 일반 JSON 요청과 다르게 처리해야 한다는 점이 중요했다.
- RestClient와 HTTP Interface는 둘 다 외부 API 호출에 사용되지만, 직접 호출 방식과 선언형 인터페이스 방식이라는 차이를 구분해야 했다.

## 해결 방법

- Reflection은 “메서드를 이름으로 찾고 런타임에 호출할 수 있게 해주는 기능”으로 먼저 단순화해서 이해했다.
- ArgumentResolver는 “컨트롤러 메서드 인자 생성”, ReturnValueHandler는 “컨트롤러 반환값 처리”로 구분했다.
- Interceptor는 Spring MVC 요청 처리 흐름 안에서 Controller 전후를 다루는 기능으로 정리했다.
- 예외 처리는 HTML 화면 오류와 API JSON 오류를 분리해서 생각했다.
- Multipart는 요청 본문이 여러 조각으로 나뉜다는 구조를 먼저 이해하고, Spring에서는 `MultipartFile`과 `@RequestPart`로 처리한다고 정리했다.
- RestClient는 간단한 직접 HTTP 호출, HTTP Interface는 반복되는 외부 API 호출을 인터페이스로 추상화할 때 유용하다고 구분했다.

## 오늘 배운 것

### API 개발 기본

- API 요청과 응답에는 엔티티를 직접 사용하지 않는 것이 좋다.
- 엔티티를 요청 DTO로 직접 받으면 엔티티에 API 검증, 프레젠테이션 계층 요구사항이 섞인다.
- 엔티티를 응답으로 직접 반환하면 엔티티의 모든 값이 노출될 수 있고, 엔티티 변경이 API 스펙 변경으로 이어질 수 있다.
- API 스펙에 맞는 Request DTO와 Response DTO를 따로 만들면 엔티티와 API 계층을 분리할 수 있다.
- 회원 수정은 영속 상태의 엔티티를 조회한 뒤 값을 변경하면 변경 감지로 UPDATE SQL이 실행된다.
- 부분 수정 API는 `PUT`보다 `PATCH` 또는 `POST`가 더 자연스러울 수 있다.
- 컬렉션 응답은 `List`를 바로 반환하기보다 `Result<T>` 같은 래퍼 객체로 감싸면 이후 count 같은 부가 필드를 추가하기 쉽다.

### 지연 로딩과 조회 성능 최적화

- JPA 연관관계는 기본적으로 지연 로딩을 사용하고, 필요한 조회에서 페치 조인으로 최적화하는 것이 좋다.
- 엔티티를 API 응답으로 직접 노출하면 프록시 객체 직렬화 문제와 양방향 연관관계 무한 루프 문제가 발생할 수 있다.
- Hibernate 모듈로 프록시 직렬화 문제를 일부 해결할 수 있지만, 실무에서는 DTO로 변환해서 반환하는 것이 더 안전하다.
- 단순 주문 조회에서 엔티티를 DTO로 변환하면 코드 구조는 좋아지지만, 지연 로딩 때문에 `1 + N + N` 쿼리가 발생할 수 있다.
- xToOne 관계는 페치 조인해도 row 수가 증가하지 않으므로 페이징에 비교적 안전하고, 쿼리 수를 줄이는 데 효과적이다.
- JPA에서 DTO를 직접 조회하면 필요한 컬럼만 선택할 수 있지만, 리포지토리 재사용성이 떨어지고 API 스펙에 맞춘 조회 코드가 생긴다.

### 컬렉션 조회 최적화

- 컬렉션 관계를 페치 조인하면 SQL 한 번으로 조회할 수 있지만, 일대다 조인으로 row 수가 증가한다.
- 컬렉션 페치 조인에는 `distinct`가 필요할 수 있으며, JPA는 같은 엔티티 중복을 애플리케이션 레벨에서도 제거해준다.
- 컬렉션 페치 조인은 페이징이 불가능하다. 하이버네이트는 모든 데이터를 메모리에 올린 뒤 페이징할 수 있으므로 매우 위험하다.
- 페이징이 필요한 컬렉션 조회는 ToOne 관계만 페치 조인하고, 컬렉션은 지연 로딩과 `hibernate.default_batch_fetch_size` 또는 `@BatchSize`로 최적화하는 방식이 좋다.
- DTO 직접 조회 방식에서는 루트 데이터를 먼저 조회하고, 컬렉션 데이터를 IN 절로 한 번에 조회한 뒤 Map으로 매칭하면 `1 + 1` 쿼리로 최적화할 수 있다.
- 플랫 데이터 조회는 쿼리 한 번으로 가져올 수 있지만, 중복 데이터가 많고 애플리케이션에서 다시 그룹핑해야 하며 페이징이 어렵다.

### OSIV와 실무 최적화

- OSIV가 켜져 있으면 API 응답이 끝날 때까지 영속성 컨텍스트가 유지되어 컨트롤러나 뷰에서도 지연 로딩이 가능하다.
- OSIV ON은 편리하지만 DB 커넥션을 오래 잡고 있을 수 있어 실시간 트래픽이 많은 서비스에서는 커넥션 부족 문제를 만들 수 있다.
- OSIV를 끄면 트랜잭션 종료 시 영속성 컨텍스트와 DB 커넥션을 반환하므로 커넥션 리소스를 아낄 수 있다.
- OSIV OFF 상태에서는 모든 지연 로딩을 트랜잭션 안에서 처리해야 한다.
- 복잡한 조회는 `OrderQueryService`처럼 조회 전용 서비스를 분리하고, 주로 읽기 전용 트랜잭션 안에서 필요한 데이터를 모두 조회해 DTO로 변환하는 방식이 좋다.
- Command와 Query를 분리하면 핵심 비즈니스 로직과 화면/API 조회 최적화 로직을 분리할 수 있다.

## 어려웠던 점

- 엔티티를 DTO로 변환하는 방식과 JPA에서 DTO를 직접 조회하는 방식의 장단점을 구분하는 데 시간이 필요했다.
- 지연 로딩은 편리하지만 DTO 생성 시점에 연관 객체를 접근하면 N+1 문제가 발생한다는 점을 코드 흐름과 연결해야 했다.
- xToOne 관계와 컬렉션 관계의 최적화 방식이 다르다는 점이 중요했다.
- 컬렉션 페치 조인이 왜 페이징에 위험한지 처음에는 직관적으로 와닿지 않았다.
- V4, V5, V6는 모두 DTO 직접 조회 방식이지만 쿼리 수, 중복 데이터, 페이징 가능 여부가 달라서 비교가 필요했다.
- OSIV를 끄면 성능에는 유리하지만, 지연 로딩 코드를 트랜잭션 안으로 옮겨야 하는 구조 변경이 필요하다는 점이 어려웠다.

## 해결 방법

- API 설계는 “엔티티는 내부 모델, DTO는 외부 API 스펙”이라는 기준으로 구분했다.
- 조회 최적화는 먼저 엔티티 조회 후 DTO 변환을 기본으로 두고, 필요하면 페치 조인을 적용하는 순서로 정리했다.
- xToOne은 페치 조인, ToMany 컬렉션은 batch fetch size나 IN 절 조회로 최적화한다고 구분했다.
- 컬렉션 페치 조인은 row 수가 N 기준으로 증가하기 때문에 페이징이 깨질 수 있다고 이해했다.
- DTO 직접 조회는 V4는 단건 또는 소량 조회, V5는 여러 건 조회 최적화, V6는 플랫 데이터 조회지만 페이징 불가라는 기준으로 정리했다.
- OSIV는 “편의성은 ON, 커넥션 관리와 실시간 트래픽은 OFF” 관점으로 정리했다.

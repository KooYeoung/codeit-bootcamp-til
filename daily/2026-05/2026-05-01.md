## 오늘 배운 것

### Git 생성 배경

- Git은 리누스 토발즈가 만든 분산 버전 관리 시스템이다.
- 빠른 속도, 단순한 구조, 비선형적 개발 지원, 완전 분산형 시스템을 목표로 만들어졌다.
- Git은 버전 관리뿐만 아니라 협업을 위해서도 중요한 도구이다.

### Git의 3가지 작업 영역

- Git은 `Working Directory`, `Staging Area`, `Repository`라는 작업 영역을 기준으로 동작한다.
- Working Directory는 실제 파일을 수정하는 공간이다.
- Staging Area는 커밋에 포함할 파일을 임시로 올려두는 공간이다.
- Repository는 커밋 이력이 저장되는 공간이다.
- `git add`를 해야 Staging Area에 올라가고, `git commit`을 해야 Repository에 기록된다.

### Git 파일 상태

- Git에서 파일은 크게 `Untracked`와 `Tracked` 상태로 나뉜다.
- `Untracked`는 Git이 아직 추적하지 않는 파일이다.
- `Tracked`는 Git이 변경사항을 추적하는 파일이다.
- Tracked 상태는 다시 `Staged`, `Unmodified`, `Modified` 상태로 나뉜다.
- 파일을 수정하고, add하고, commit하는 흐름에 따라 상태가 바뀐다.

### GitHub와 원격 저장소

- GitHub는 Git 저장소를 온라인에서 관리하고 공유할 수 있는 서비스이다.
- 로컬 저장소의 내용을 원격 저장소에 반영하려면 `git push`를 사용한다.
- 원격 저장소의 내용을 로컬 저장소로 가져오려면 `git pull`을 사용한다.
- GitHub에 있는 프로젝트를 내 컴퓨터로 가져오려면 `git clone`을 사용한다.
- 원격 저장소에 push하려면 권한이 필요하며, 다른 사람이 push할 수 있게 하려면 collaborator로 초대해야 한다.

### README와 Markdown

- GitHub에서는 `README.md` 파일이 저장소 메인 화면에 표시된다.
- README에는 프로젝트 소개, 사용 방법, 실행 방법 등을 작성한다.
- `.md`는 Markdown 파일을 의미한다.
- Markdown을 사용하면 간단한 문법으로 제목, 목록, 코드 블록 등을 표현할 수 있다.

### 커밋 메시지

- 커밋 메시지는 프로젝트 변경 이력을 이해하는 데 중요한 역할을 한다.
- 하나의 커밋에는 하나의 작업 단위만 포함하는 것이 좋다.
- 커밋 메시지는 나중에 다시 봤을 때 어떤 작업을 했는지 이해할 수 있게 작성해야 한다.
- 필요하다면 제목과 본문을 나누어 작성할 수 있다.

### Git Alias

- Git alias는 긴 Git 명령어에 별명을 붙이는 기능이다.
- 자주 사용하는 긴 명령어를 짧게 줄여 사용할 수 있다.

예시:

```bash
git config alias.history "log --pretty=oneline"
```

위 설정을 하면 다음 명령어로 커밋 이력을 간단히 볼 수 있다.

```bash
git history
```

### Git Reset

- `git reset`은 HEAD가 특정 커밋을 가리키도록 이동시키는 명령어이다.
- `--soft`, `--mixed`, `--hard` 옵션에 따라 Staging Area와 Working Directory에 미치는 영향이 다르다.
- 옵션을 생략하면 기본적으로 `--mixed`가 적용된다.
- 커밋 해시 대신 `HEAD^`, `HEAD~2` 같은 상대 표현을 사용할 수 있다.

### Git Tag

- tag는 특정 커밋에 이름표를 붙이는 기능이다.
- 주로 버전 출시 지점처럼 중요한 커밋에 사용한다.

```bash
git tag 태그명 커밋해시
git tag
git show 태그명
```

### Merge Conflict와 Abort

- merge conflict는 Git이 자동으로 병합할 수 없는 충돌 상황이다.
- 충돌이 발생하면 직접 파일을 수정한 뒤 `git add`, `git commit`으로 병합을 완료한다.
- 당장 병합을 진행하지 않고 이전 상태로 되돌리고 싶다면 `git merge --abort`를 사용할 수 있다.

### Remote, Origin, Upstream

- `remote`는 원격 저장소와 관련된 작업을 할 때 사용하는 개념이다.
- `origin`은 원격 저장소에 붙이는 기본 이름으로 많이 사용된다.
- `git push -u origin main`에서 `-u`는 로컬 브랜치와 원격 브랜치를 연결하는 옵션이다.
- 한 번 upstream 연결을 해두면 이후에는 `git push`, `git pull`만으로도 연결된 원격 브랜치와 동기화할 수 있다.

## 어려웠던 점

- 로컬 브랜치와 원격 브랜치가 어떻게 연결되는지 처음에는 헷갈렸다.
- `git reset`의 `--soft`, `--mixed`, `--hard` 옵션 차이가 헷갈렸다.

## 해결 방법

- `git push -u origin main`을 실행하면 로컬 브랜치와 원격 브랜치가 연결되고, 이후에는 `git push`, `git pull`만으로 동기화할 수 있다고 정리했다.
- `reset` 옵션은 HEAD, Staging Area, Working Directory 중 어디까지 되돌리는지 기준으로 정리했다.

## 오늘 배운 것

### 웹 애플리케이션과 서블릿

- 웹 애플리케이션은 클라이언트의 HTTP 요청을 받고, 서버에서 처리한 뒤 HTTP 응답을 반환하는 구조이다.
- 서블릿은 HTTP 요청과 응답을 객체로 다룰 수 있게 해주는 Java 웹 기술이다.
- `HttpServletRequest`는 HTTP 요청 메시지를 편리하게 조회할 수 있게 해준다.
- 요청 메시지의 start line, header, query parameter, form parameter, message body 등을 직접 파싱하지 않고 request 객체를 통해 사용할 수 있다.
- `HttpServletResponse`는 HTTP 응답 코드, 헤더, 바디를 생성할 때 사용한다.
- HTML을 직접 응답할 때는 `content-type`을 `text/html`로 지정하고, JSON 응답을 반환할 때는 `application/json`을 사용한다.
- 서블릿만으로 화면을 만들면 Java 코드 안에서 HTML 문자열을 직접 작성해야 하므로 유지보수가 어렵다.

### JSP와 MVC 패턴

- JSP를 사용하면 HTML 안에서 동적인 값을 표현할 수 있어 서블릿보다 화면 작성이 편해진다.
- 하지만 JSP 안에 회원 저장, 조회 같은 비즈니스 로직이 들어가면 화면과 로직이 섞이는 문제가 생긴다.
- MVC 패턴은 Controller, Model, View로 역할을 나누어 유지보수성을 높이는 구조이다.
- Controller는 요청을 받고 비즈니스 로직을 호출한 뒤 View에 전달할 데이터를 준비한다.
- Model은 View에 전달할 데이터를 담는다.
- View는 화면을 렌더링하는 역할에 집중한다.
- MVC 패턴을 적용하면 JSP는 화면 출력에 집중하고, 서블릿은 Controller 역할을 담당하게 된다.
- `forward`는 서버 내부에서 View로 제어를 넘기는 방식이고, `redirect`는 클라이언트에게 다시 요청하라고 응답하는 방식이다.

### Front Controller와 Spring MVC 구조

- Front Controller 패턴은 모든 요청을 하나의 입구에서 받은 뒤, 요청에 맞는 컨트롤러를 찾아 호출하는 구조이다.
- 공통 처리 로직을 Front Controller에 모을 수 있어 중복을 줄일 수 있다.
- 직접 만든 MVC 프레임워크에서는 FrontController, HandlerMappingMap, HandlerAdapter, ModelView, ViewResolver, MyView 같은 구조를 단계적으로 만들었다.
- Spring MVC에서는 이 구조가 `DispatcherServlet`, `HandlerMapping`, `HandlerAdapter`, `ModelAndView`, `ViewResolver`, `View`로 대응된다.
- `DispatcherServlet`은 Spring MVC의 핵심이며, Front Controller 패턴으로 구현되어 있다.
- 요청이 들어오면 `DispatcherServlet`이 핸들러를 찾고, 핸들러 어댑터를 통해 컨트롤러를 실행한 뒤, ViewResolver로 View를 찾고 렌더링한다.

### Spring MVC 기본 기능

- `@RequestMapping`, `@GetMapping`, `@PostMapping`을 사용해 요청 URL과 HTTP 메서드를 매핑할 수 있다.
- `@PathVariable`은 URL 경로에 포함된 값을 꺼낼 때 사용한다.
- `@RequestParam`은 쿼리 파라미터나 HTML Form 데이터를 받을 때 사용한다.
- `@ModelAttribute`는 요청 파라미터를 객체에 바인딩할 때 사용한다.
- `@RequestBody`는 HTTP 메시지 바디를 직접 읽어 객체나 문자열로 변환할 때 사용한다.
- `@ResponseBody`는 View를 찾지 않고 HTTP 메시지 바디에 직접 데이터를 반환할 때 사용한다.
- HTTP 메시지 컨버터는 요청 바디를 객체로 변환하거나, 객체를 응답 바디로 변환하는 역할을 한다.
- JSON 요청을 객체로 받을 때는 `@RequestBody`를 생략하면 `@ModelAttribute`로 처리될 수 있으므로 주의해야 한다.

### 웹 페이지 만들기

- 상품 관리 예제에서는 상품 ID, 상품명, 가격, 수량을 가진 `Item` 도메인을 사용했다.
- 상품 목록, 상품 상세, 상품 등록, 상품 수정 기능을 Spring MVC와 Thymeleaf로 구현하는 흐름을 학습했다.
- Controller는 요청을 처리하고 Model에 데이터를 담아 View에 전달한다.
- Thymeleaf는 서버에서 동적으로 HTML을 렌더링하는 템플릿 엔진이다.
- 상품 등록 후 새로고침 시 중복 등록이 발생할 수 있으므로 PRG(Post/Redirect/Get) 패턴을 적용해야 한다.
- `RedirectAttributes`를 사용하면 redirect 경로에 필요한 값을 안전하게 전달할 수 있다.

## 어려웠던 점

- 서블릿, JSP, MVC 패턴의 차이는 이해했지만, 왜 단계적으로 발전했는지를 흐름으로 연결하는 데 시간이 필요했다.
- Front Controller를 직접 만드는 과정에서 v1부터 v5까지 구조가 조금씩 바뀌어 각 버전의 개선점을 구분하는 것이 어려웠다.
- `DispatcherServlet`, `HandlerMapping`, `HandlerAdapter`, `ViewResolver`가 각각 어떤 책임을 가지는지 처음에는 헷갈렸다.
- `@RequestParam`, `@ModelAttribute`, `@RequestBody`가 모두 요청 데이터를 받는 기능처럼 보여 구분이 필요했다.
- `@ResponseBody`와 View 반환 방식의 차이를 명확히 이해해야 했다.
- PRG 패턴은 개념은 간단하지만, 실제로 왜 중복 등록을 막는지 요청 흐름을 기준으로 다시 정리해야 했다.

## 해결 방법

- 서블릿은 HTTP 요청과 응답을 직접 다루는 기술, JSP는 화면 작성 기술, MVC는 역할 분리 구조라고 나누어 이해했다.
- MVC 패턴은 Controller가 요청을 처리하고, Model에 데이터를 담고, View가 화면을 출력하는 구조로 정리했다.
- Front Controller는 “요청의 입구를 하나로 만드는 패턴”이라고 이해했다.
- Spring MVC 구조는 직접 만든 MVC 프레임워크와 비교하며 대응 관계를 정리했다.
- 요청 데이터 처리 방식은 데이터가 어디에 담겨 오는지를 기준으로 구분했다.
  - query parameter / HTML Form: `@RequestParam`, `@ModelAttribute`
  - HTTP message body / JSON: `@RequestBody`
- 응답 방식은 View를 렌더링하는 경우와 HTTP Body에 직접 쓰는 경우로 나누어 이해했다.
- PRG는 POST 요청 처리 후 바로 View를 반환하지 않고 redirect를 통해 GET 요청으로 전환하는 패턴이라고 정리했다.

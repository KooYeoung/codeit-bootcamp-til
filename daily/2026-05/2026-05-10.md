## 오늘 배운 것

### 디자인 패턴 기본 관점

- 디자인 패턴은 무조건 외워서 적용하는 정답이 아니라, 반복해서 등장하는 설계 문제를 해결하기 위한 경험적 방법이라고 이해했다.
- 패턴을 사용할 때는 먼저 코드에서 어떤 문제가 반복되는지 확인하는 것이 중요하다고 느꼈다.
- 단순한 코드에 패턴을 억지로 적용하면 오히려 클래스가 늘어나고 흐름을 이해하기 어려워질 수 있다.
- 객체지향 설계에서는 역할과 책임을 분리하고, 구체 클래스보다 인터페이스에 의존하는 방향이 유연한 구조를 만드는 데 도움이 된다고 정리했다.

### 주요 패턴 정리

- 파사드 패턴은 복잡한 내부 기능을 하나의 쉬운 창구로 감싸서 클라이언트 코드의 복잡도를 낮추는 방식이다.
- 전략 패턴은 같은 목적의 여러 처리 방식을 인터페이스와 구현체로 분리해 상황에 따라 갈아 끼울 수 있게 한다.
- 템플릿 메서드 패턴은 전체 실행 순서는 부모 클래스에서 정하고, 세부 단계는 자식 클래스에서 다르게 구현하는 방식이다.
- 싱글턴 패턴은 객체가 하나만 필요할 때 사용할 수 있지만, 전역 상태처럼 남용하면 테스트와 의존성 관리가 어려워질 수 있다.
- 상태 패턴은 상태별 행동을 클래스로 분리해 상태에 따른 조건문이 계속 늘어나는 문제를 줄일 수 있다.
- 어댑터 패턴은 기존 코드나 외부 라이브러리의 사용 방식이 현재 인터페이스와 맞지 않을 때 중간에서 연결해 주는 역할을 한다.
- 팩토리 메서드 패턴은 객체 생성 책임을 사용하는 쪽에서 분리해 생성 로직을 한곳에서 관리할 수 있게 한다.

### 적용 관점

- 정렬, 검색, 검증처럼 처리 방식이 여러 개로 바뀔 수 있는 기능은 전략 패턴으로 분리할 수 있다고 생각했다.
- 객체 생성 로직이 여러 곳에 흩어져 있다면 팩토리 메서드 패턴으로 생성 책임을 모아볼 수 있다.
- 여러 단계로 요청을 처리해야 하는 흐름이 있다면 책임 연쇄 패턴을 검토할 수 있다.
- 공통 설정이나 공유 객체처럼 하나만 존재해야 자연스러운 객체가 있다면 싱글턴을 고려할 수 있지만, 숨은 의존성이 생기지 않도록 조심해야 한다.

## 어려웠던 점

- 패턴의 이름은 다르지만 공통적으로 인터페이스, 위임, 조합을 사용하는 경우가 많아서 처음에는 차이를 구분하기 어려웠다.
- 어떤 상황에서 패턴을 적용하는 것이 적절하고, 어떤 상황에서는 과한 설계가 되는지 판단하는 기준이 아직 명확하지 않았다.
- 패턴별 코드 예시는 이해할 수 있었지만 실제 프로젝트 코드에 자연스럽게 연결하는 것은 더 많은 연습이 필요하다고 느꼈다.

## 해결 방법

- 패턴을 이름 중심으로 외우기보다 “어떤 문제를 해결하려고 나온 패턴인가”를 기준으로 다시 정리했다.
- 각 패턴의 장점뿐 아니라 주의할 점도 함께 보면서 남용하지 않아야 한다는 기준을 세웠다.

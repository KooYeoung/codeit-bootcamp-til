## 오늘 배운 것


### 스프링 핵심 원리

- 객체지향 설계에서 중요한 것은 역할과 구현을 분리하는 것이다.
- 다형성만으로는 OCP와 DIP를 완전히 지키기 어렵고, 클라이언트 코드가 구현체를 직접 선택하면 변경에 취약해진다.
- AppConfig처럼 객체 생성과 의존관계 연결을 담당하는 설정 클래스를 두면 관심사를 분리할 수 있다.
- IoC는 객체 생성과 제어 흐름을 개발자가 직접 관리하지 않고 외부에 맡기는 것이다.
- DI는 필요한 의존 객체를 외부에서 주입받는 방식이다.
- 스프링 컨테이너는 객체를 생성하고, 의존관계를 주입하고, 스프링 빈으로 관리한다.
- `ApplicationContext`는 일반적으로 스프링 컨테이너라고 부르며, 등록된 빈을 조회하고 관리한다.

### 스프링 빈과 DI

- 스프링 빈은 스프링 컨테이너가 관리하는 객체이다.
- `@Bean`을 사용하면 자바 설정 클래스에서 직접 스프링 빈을 등록할 수 있다.
- `@ComponentScan`을 사용하면 `@Component`가 붙은 클래스를 자동으로 스캔해서 빈으로 등록한다.
- `@Controller`, `@Service`, `@Repository`, `@Configuration`은 내부적으로 컴포넌트 스캔 대상이 될 수 있다.
- 의존관계 주입 방법에는 생성자 주입, 수정자 주입, 필드 주입, 일반 메서드 주입이 있다.
- 실무에서는 불변성과 테스트 편의성 때문에 생성자 주입을 권장한다.
- 생성자가 하나만 있으면 `@Autowired`를 생략해도 자동 주입된다.

### 싱글톤, 생명주기, 스코프

- 스프링 빈은 기본적으로 싱글톤 스코프로 등록된다.
- 웹 애플리케이션은 여러 요청이 동시에 들어오기 때문에 객체를 매번 생성하면 비효율적일 수 있다.
- 싱글톤 방식에서는 하나의 객체 인스턴스를 공유하므로 상태를 무상태로 설계해야 한다.
- 빈 생명주기는 스프링 컨테이너 생성, 빈 생성, 의존관계 주입, 초기화 콜백, 사용, 소멸 전 콜백, 스프링 종료 흐름으로 이해할 수 있다.
- 초기화 작업과 생성자 작업은 분리하는 것이 좋다.
- `@PostConstruct`, `@PreDestroy`를 사용하면 초기화와 종료 시점의 콜백 메서드를 지정할 수 있다.
- 빈 스코프에는 싱글톤, 프로토타입, request, session, application 등이 있다.
- 프로토타입 스코프는 요청할 때마다 새 객체가 생성되지만, 스프링 컨테이너는 생성과 의존관계 주입까지만 관리한다.

## 어려웠던 점

- `@Bean` 수동 등록과 `@ComponentScan` 자동 등록의 차이를 실제 코드 구조 기준으로 구분하는 것이 중요했다.
- 싱글톤 컨테이너에서 객체를 하나만 공유한다는 점은 이해했지만, 왜 무상태 설계가 필요한지 다시 정리할 필요가 있었다.
- 생성자 주입, 수정자 주입, 필드 주입의 차이를 단순 문법이 아니라 유지보수성과 테스트 관점에서 이해해야 했다.
- 빈 생명주기와 빈 스코프는 비슷하게 느껴졌지만, 하나는 객체가 생성되고 종료되는 흐름이고 다른 하나는 빈이 존재하는 범위라는 점을 구분해야 했다.

## 해결 방법

- `AppConfig → ApplicationContext → Bean → DI` 흐름으로 스프링 컨테이너 개념을 정리했다.
- 의존관계 주입은 가능하면 생성자 주입을 기본으로 생각하고, 선택적 의존관계가 필요한 경우에만 수정자 주입을 고려한다고 정리했다.
- 빈 생명주기는 “생성 후 초기화, 종료 전 정리”로 이해하고, 빈 스코프는 “빈이 얼마나 오래 유지되는가”로 구분했다.
